# 004.5 - SSR Module Loader Cache Staleness

## Severity: HIGH - Stale SSR Modules in Production

## The Problem

The SSR module loader cache key includes `filePath` but uses content hash only for cache busting within the temp file path, not for distributed cache isolation. When the same file path has different content across pods (due to preview vs release, or timing), stale modules can be served.

### Location: `/Users/mattboon/Sites/veryfront-renderer/src/modules/react-loader/ssr-module-loader/loader.ts:259-272`

```typescript
private getCacheKey(filePath: string): string {
  if (!this.options.contentSourceId) {
    throw new Error(
      `Missing contentSourceId for SSR module cache (project: ${this.options.projectId}, file: ${filePath})`,
    );
  }
  const reactVersion = this.options.reactVersion ?? "default";
  return buildSSRModuleCacheKey(
    TRANSFORM_CACHE_VERSION,
    this.options.projectId,
    `${this.options.contentSourceId}:${reactVersion}:${filePath}`,
  );
  // NOTE: filePath is included, but NOT the content hash!
}
```

### Location: `/Users/mattboon/Sites/veryfront-renderer/src/cache/keys.ts:240-246`

```typescript
export function buildSSRModuleCacheKey(
  version: string | number,
  projectId: string,
  filePath: string,
): string {
  return `${CacheKeyPrefix.SSR_VERSION}${version}:${projectId}:${filePath}`;
  // Key: v18:projectId:contentSourceId:reactVersion:filePath
  // NO content hash in the distributed cache key!
}
```

## The Content Hash Disconnect

The SSR loader **does compute** a content hash, but uses it differently:

### Location: `/Users/mattboon/Sites/veryfront-renderer/src/modules/react-loader/ssr-module-loader/loader.ts:443-448`

```typescript
const code = source ?? (await this.options.adapter.fs.readFile(filePath));
const contentHash = await this.hashContentAsync(code);
const contentCacheKey = this.getCacheKey(`${filePath}:${contentHash}`);
const filePathCacheKey = this.getCacheKey(filePath);
// Uses content hash in inProgressKey to prevent race condition where
// different content versions wait for each other's transforms
```

Two cache keys are created:
- `contentCacheKey` - includes content hash (used for in-memory cache)
- `filePathCacheKey` - does NOT include content hash (also used)

### Location: `/Users/mattboon/Sites/veryfront-renderer/src/modules/react-loader/ssr-module-loader/loader.ts:726-727`

```typescript
// Use transformedHash for cache busting in dynamic imports
const entry: ModuleCacheEntry = { tempPath, contentHash: transformedHash };
globalModuleCache.set(contentCacheKey, entry);
globalModuleCache.set(filePathCacheKey, entry);  // <-- Stored by path too!
```

The module is stored by **both** keys. Later lookups by `filePathCacheKey` may return stale content.

## Redis Cache Key Issue

### Location: `/Users/mattboon/Sites/veryfront-renderer/src/modules/react-loader/ssr-module-loader/loader.ts:493-540`

```typescript
if (isSSRDistributedCacheEnabled()) {
  const redisCode = await getFromRedis(contentCacheKey);  // Uses contentCacheKey
  if (redisCode) {
    // ...
    const entry: ModuleCacheEntry = { tempPath, contentHash: transformedHash };
    globalModuleCache.set(contentCacheKey, entry);
    globalModuleCache.set(filePathCacheKey, entry);  // But also sets filePathCacheKey
    // ...
  }
}
```

The problem: While `contentCacheKey` includes the hash, the code ALSO stores by `filePathCacheKey` which doesn't. Future lookups by `filePathCacheKey` can return wrong content.

## Failure Scenario

### Sequence of Events

| Time | Pod | File Content | contentCacheKey | filePathCacheKey | Result |
|------|-----|--------------|-----------------|------------------|--------|
| T1 | A | v1 (abc123) | `...:file.tsx:abc123` | `...:file.tsx` | Transforms v1 |
| T2 | A | - | - | - | Sets both keys to v1 |
| T3 | B | v2 (def456) | `...:file.tsx:def456` | `...:file.tsx` | Memory cache miss |
| T4 | B | - | - | - | Checks Redis by contentCacheKey |
| T5 | B | - | - | - | Redis miss (v2 not there) |
| T6 | B | - | - | - | Transforms v2, sets both keys |
| T7 | A | v2 | - | `...:file.tsx` | **Memory hit returns v1!** |

Pod A's memory cache has `filePathCacheKey` pointing to v1. When v2 content comes in, the memory lookup by `filePathCacheKey` returns the stale v1 module.

## Impact Assessment

| Scenario | Expected | Actual |
|----------|----------|--------|
| Deploy new release | All pods serve new code | Some pods serve old code |
| Preview branch update | Immediate update | Stale preview |
| Component hotfix | Fix everywhere | Fix only on new pods |
| Long-running pod | Serves current code | Serves stale code |

## The Fix is Simple

The dual-key storage pattern is unnecessary and dangerous:

```typescript
// BEFORE: Storing by two keys
globalModuleCache.set(contentCacheKey, entry);
globalModuleCache.set(filePathCacheKey, entry);  // REMOVE THIS

// AFTER: Store only by content-aware key
globalModuleCache.set(contentCacheKey, entry);
```

## Related: Memory Cache Lookup

### Location: `/Users/mattboon/Sites/veryfront-renderer/src/modules/react-loader/ssr-module-loader/loader.ts:450-491`

```typescript
const cachedEntry = globalModuleCache.get(contentCacheKey);
if (cachedEntry) {
  // Verify HTTP bundles exist for in-memory cached transforms

    globalModuleCache.set(filePathCacheKey, cachedEntry);  // <-- Propagates stale entry!

```

Even the lookup path propagates entries to `filePathCacheKey`.

## Solution

See **[004.0 - Dependency Tracking RFC](./004.0-dependency-tracking-rfc.md)** for the complete solution.

Immediate fix:
1. Remove all `globalModuleCache.set(filePathCacheKey, ...)` calls
2. Use only `contentCacheKey` which includes the content hash
3. Ensure Redis cache key also includes content hash

Comprehensive fix (per RFC):
1. Include dependency hash in cache key
2. Single cache key format across all module loaders
3. Consistent content + deps hash for cache isolation
