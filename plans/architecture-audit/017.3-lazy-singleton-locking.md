# 017.3: Lazy Singleton Without Lock

## Severity: MEDIUM

## Location

**Files:** Module system and service initialization
- `src/module-system/`
- `src/core/services/`

```typescript
// VULNERABLE - Classic double-check locking without actual lock
let instance: Service | null = null;

export async function getService(): Promise<Service> {
  if (instance === null) {
    instance = await createService();  // RACE: Multiple creates
  }
  return instance;
}
```

## The Problem

JavaScript's single-threaded model doesn't prevent async race conditions. The `await` between null check and assignment allows multiple calls to see `null` and create duplicate instances.

### Race Window

```
t0: Request A: if (instance === null) → true
t1: Request A: createService() starts (await)
t2: Request B: if (instance === null) → true (A hasn't assigned yet)
t3: Request B: createService() starts (await)
t4: Request A: instance = serviceA
t5: Request B: instance = serviceB (overwrites A)

Result:
- Two Service instances created
- A's connections/state lost
- B's instance used going forward
- Resource leak from A
```

## Proposed Fix

### Option A: Promise-Based Lock

```typescript
let instance: Service | null = null;
let initPromise: Promise<Service> | null = null;

export async function getService(): Promise<Service> {
  if (instance !== null) {
    return instance;
  }

  if (initPromise !== null) {
    return initPromise;
  }

  initPromise = createService().then((service) => {
    instance = service;
    initPromise = null;
    return service;
  });

  return initPromise;
}
```

### Option B: Mutex Pattern

```typescript
import { Mutex } from "async-mutex";

const mutex = new Mutex();
let instance: Service | null = null;

export async function getService(): Promise<Service> {
  // Fast path - no lock needed
  if (instance !== null) {
    return instance;
  }

  // Slow path - acquire lock
  const release = await mutex.acquire();
  try {
    // Double-check after acquiring lock
    if (instance === null) {
      instance = await createService();
    }
    return instance;
  } finally {
    release();
  }
}
```

### Option C: Module-Level Initialization

```typescript
// Initialize at module load (synchronous context)
const instance = createServiceSync();

export function getService(): Service {
  return instance;
}
```

## Affected Patterns

Look for this pattern:
```typescript
let something: T | null = null;
async function getSomething() {
  if (something === null) {
    something = await create();
  }
  return something;
}
```

## Blast Radius

| Affected | Impact |
|----------|--------|
| Database connections | Multiple connection pools |
| Cache clients | Duplicate Redis connections |
| Logger instances | Split logs |
| Config loaders | Inconsistent config |

## Migration Checklist

- [ ] Audit all lazy initialization patterns
- [ ] Add `async-mutex` or custom lock utility
- [ ] Update singletons to use lock pattern
- [ ] Add concurrent initialization test
- [ ] Consider eager initialization where possible
