## The Problem

The `failedComponents` map is a **global circuit breaker** that tracks component load failures. When a component fails to transform/load N times within a time window, it's blocked to prevent repeated failures. However:

1. The map is **global across all projects**
2. The cache key comes from `getCacheKey()` which **does include projectId**
3. But `clearSSRModuleCache()` clears **all projects**
4. More critically, the **FailureRecord type** doesn't store projectId for debugging

## Why It's Bad for Multi-Tenancy

### Scenario 1: Cache Clear Affects All Projects

```
Timeline:
=========
t0: Project A has component /components/Button.tsx fail 3 times
    -> Circuit breaker activated for Project A's Button

t1: Project B calls clearSSRModuleCache() (e.g., after deployment)
    -> failedComponents.clear() removes ALL entries including Project A's

t2: Project A's circuit breaker is now cleared unexpectedly
    -> Button starts failing again, causing repeated error spikes
    -> User sees 3 more failures before circuit breaker re-engages

Impact: Cache clear in one project destabilizes another project's circuit breaker
```

### Scenario 2: Debugging Confusion

```
Timeline:
=========
t0: Admin investigating why Project A's Button is circuit-broken
    -> Looks at failedComponents map
    -> Sees entry with key "ssr:v5:proj-uuid-a:preview-draft:...:components/Button.tsx"
    -> But FailureRecord doesn't include projectId, only count and timestamp
    -> Admin can't easily see WHICH project's error message applies

Problem: When multiple projects have same component path, error messages blend together
```

### Scenario 3: Key Format Dependency

The safety of this map depends entirely on `getCacheKey()` always including projectId:

```typescript
// loader.ts:259-272
private getCacheKey(filePath: string): string {
  if (!this.options.contentSourceId) {
    throw new Error(
      `Missing contentSourceId for SSR module cache (project: ${this.options.projectId}, file: ${filePath})`,
    );
  }
  const reactVersion = this.options.reactVersion ?? "default";
  return buildSSRModuleCacheKey(
    TRANSFORM_CACHE_VERSION,
    this.options.projectId,  // <-- projectId IS included
    `${this.options.contentSourceId}:${reactVersion}:${filePath}`,
  );
}
```

This is **currently correct**, but:
- No type enforcement that keys must include projectId
- No test validating isolation
- Easy to break in refactoring

## Failure Scenario

```
User Request Flow:
==================
1. User A visits projectA.veryfront.com/dashboard
2. Dashboard uses <Chart /> component
3. Chart component has syntax error
4. SSRModuleLoader.loadModule() called
5. Transform fails -> failedComponents.set(keyA, { count: 1, lastFailure: now })
6. Retry fails -> failedComponents.set(keyA, { count: 2, lastFailure: now })
7. Third fail -> count reaches threshold -> circuit breaker activates
8. User sees error: "Component is temporarily blocked"

Meanwhile:
9. Admin deploys Project B
10. Post-deployment webhook calls clearSSRModuleCache() for fresh start
11. failedComponents.clear() removes Project A's entry too
12. User A refreshes -> Chart component loads again -> fails again
13. Error spike continues until circuit breaker re-engages

Result: Project A's users experience 3x more errors than necessary
        because Project B's deployment affected their circuit breaker
```

## Blast Radius

| Affected | Impact |
|----------|--------|
| Circuit breaker state | Cleared by any project's cache clear |
| Error debugging | Can't identify which project's error |
| Component retry behavior | Unpredictable retries across projects |
| Error metrics | False positives in error rate |

## Solution

See **[010.0 - Error Handling RFC](./010.0-error-handling-rfc.md)** for the complete solution.

**Summary of fixes:**

1. Add `projectId` field to `FailureRecord` for debugging
2. Create `clearSSRModuleCacheForProject()` that ONLY clears that project
3. Remove or deprecate `clearSSRModuleCache()` global clear
4. Add type enforcement that cache keys must include projectId
5. Add tests for cross-project isolation
