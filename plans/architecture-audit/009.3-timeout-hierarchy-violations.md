# 009.3 - Timeout Hierarchy Violations

## Severity: HIGH

## Location

**Files:**
- `/Users/mattboon/Sites/veryfront-renderer/proxy/main.ts` (line 53)
- `/Users/mattboon/Sites/veryfront-renderer/src/middleware/builtin/timeout.ts` (line 6)
- `/Users/mattboon/Sites/veryfront-renderer/src/rendering/renderer.ts` (line 91)

## The Problem

The timeout hierarchy has **parent-child violations** where child timeouts can exceed or equal parent timeouts:

### Current Hierarchy (Broken)

```
PROXY_REQUEST_TIMEOUT (90s)
    |
    +-- REQUEST_TIMEOUT (60s)  <- 66% of parent, should work
            |
            +-- RENDER_PIPELINE_TIMEOUT (60s)  <- EQUAL TO PARENT! BUG!
```

### Code Evidence

```typescript
// proxy/main.ts:53
const RENDERER_REQUEST_TIMEOUT_MS = parseInt(getEnv("RENDERER_REQUEST_TIMEOUT_MS") || "90000");

// middleware/builtin/timeout.ts:6
const DEFAULT_TIMEOUT_MS = 60000;

// rendering/renderer.ts:91
const RENDER_PIPELINE_TIMEOUT_MS = parseInt(getEnv("RENDER_TIMEOUT_MS") ?? "60000", 10);
```

**Violation**: `RENDER_PIPELINE_TIMEOUT_MS` (60s) === `DEFAULT_TIMEOUT_MS` (60s)

## Why It's Bad for Multi-Tenancy

### Race Condition Between Timeout Layers

When render takes exactly 60 seconds:

1. `RENDER_PIPELINE_TIMEOUT_MS` fires at T+60s
2. `DEFAULT_TIMEOUT_MS` fires at T+60s

Both fire **simultaneously**, creating a race:
- Which error reaches the client first?
- Which cleanup code runs?
- Are resources properly released?

### Unpredictable Error Messages

| Render Duration | Winner | Client Sees |
|-----------------|--------|-------------|
| 59.9s | Neither | Success |
| 60.0s | Race | Random error |
| 60.1s | Both | Double error logs |

### Real Example: Error Log Confusion

```json
// Log at T+60.000s
{"level":"error","msg":"TIMEOUT_HARD operation timed out (throwing)","label":"Render pipeline for proj-123:/about","timeoutMs":60000}

// Log at T+60.001s (from middleware)
{"level":"warn","msg":"[timeout] Request timed out","path":"/about","method":"GET","timeoutMs":60000}
```

Both timeouts fire, creating duplicate error handling.

## Failure Scenario

### Timeline: 60-Second Render

```
T+0s:     Request arrives
          - Middleware starts 60s timeout
          - Renderer starts 60s timeout

T+59.9s:  SSR rendering completes
          - Render timeout cleared
          - Response starts streaming

T+60.0s:  Middleware timeout fires!
          - Client sees 504 even though render completed
          - Response stream is corrupted

--- OR ---

T+59.9s:  SSR still rendering (slow)

T+60.0s:  RACE CONDITION
          - Render timeout: throws TimeoutError
          - Middleware timeout: returns 504 Response
          - Both handlers try to respond
          - "Headers already sent" error in logs
```

## Additional Hierarchy Issues

### Stage Timeouts Not Enforced

The render pipeline has internal stage timeouts that are defined but not consistently enforced:

```typescript
// These values exist in documentation but are not all implemented
MODULE_LOAD_TIMEOUT_MS = 10000   // Sometimes enforced
DATA_FETCH_TIMEOUT_MS = 10000   // Enforced in static-data-fetcher
SSR_RENDER_TIMEOUT_MS = 20000   // Not enforced
CSS_GENERATE_TIMEOUT_MS = 5000  // Not enforced
```

### Duplicate Timeout Constants

```typescript
// config/defaults.ts:7
export const DATA_FETCH_TIMEOUT_MS = 10000;

// rendering/orchestrator/pipeline.ts (hypothetical, from documentation)
const DATA_FETCH_TIMEOUT_MS = 15_000;  // DIFFERENT VALUE!
```

Two files define the same constant with different values.

## Solution

See **[009.0 - Timeout Handling RFC](./009.0-timeout-handling-rfc.md)** Sections 1-2 for the complete hierarchy solution.

### Quick Fix: Add Safety Margins

```typescript
// src/utils/constants/timeouts.ts

const SAFETY_MARGIN = 0.8;  // Child should be 80% of parent max

export const PROXY_TIMEOUT_MS = ms(90_000);
export const REQUEST_TIMEOUT_MS = ms(Math.floor(PROXY_TIMEOUT_MS * SAFETY_MARGIN));  // 72s
export const RENDER_PIPELINE_TIMEOUT_MS = ms(Math.floor(REQUEST_TIMEOUT_MS * SAFETY_MARGIN));  // 57s
```

### Resulting Hierarchy

```
PROXY_TIMEOUT (90s)
    |
    +-- REQUEST_TIMEOUT (72s = 80% of 90s)
            |
            +-- RENDER_PIPELINE (57s = 80% of 72s)
                    |
                    +-- Stage timeouts (sum < 57s)
```

Now render has 15s margin before request timeout, and request has 18s margin before proxy timeout.
