# 011.2: SSR/Browser Resolution Divergence

## Severity: HIGH

## Location

**Files:**
- `/Users/mattboon/Sites/veryfront-renderer/src/transforms/esm/import-rewriter.ts` (browser)
- `/Users/mattboon/Sites/veryfront-renderer/src/modules/server/ssr-import-rewriter.ts` (SSR)
- `/Users/mattboon/Sites/veryfront-renderer/src/transforms/mdx/compiler/import-rewriter.ts` (MDX)

## The Problem

The same import specifier resolves to different URLs depending on whether code runs in SSR or browser context, and which rewriter processes it. This creates hydration mismatches and runtime errors.

### Divergent Resolution Examples

#### Path Alias `@/Button`

**Browser (import-rewriter.ts:53-63):**
```typescript
function shouldSkipRewrite(specifier: string): boolean {
  return (
    // ...
    specifier.startsWith("@/") ||  // SKIPPED - resolved elsewhere
    // ...
  );
}
```
Result: `@/Button` unchanged, handled by path-resolver.ts later.

**SSR (ssr-import-rewriter.ts:92-106):**
```typescript
function rewritePathAliases(code: string, options: SSRRewriteOptions): string {
  const { projectSlug, branch, cacheBuster = Date.now(), crossProjectRef } = options;
  const projectParam = projectSlug ? `&project=${projectSlug}` : "";
  const branchParam = branch ? `&branch=${branch}` : "";

  return code.replace(/from\s+["']@\/([^"']+)["']/g, (_match, path: string) => {
    const jsPath = path.endsWith(".js") ? path : `${path}.js`;

    if (crossProjectRef) {
      return `from "/_vf_modules/_cross/${crossProjectRef}/@/${jsPath}?ssr=true&v=${cacheBuster}"`;
    }

    return `from "/_vf_modules/${jsPath}?ssr=true${projectParam}${branchParam}&v=${cacheBuster}"`;
  });
}
```
Result: `/_vf_modules/Button.js?ssr=true&project=foo&branch=main&v=1705123456789`

**MDX (mdx/compiler/import-rewriter.ts:41-47):**
```typescript
if (spec.startsWith("@/")) {
```

Result (browser): `/_vf_modules/Button.js`
Result (SSR): `@/Button` unchanged

### Concrete Failure: Hydration Mismatch

```tsx
// pages/dashboard.tsx
import { Button } from "@/components/Button";

export default function Dashboard() {
  return <Button>Click me</Button>;
}
```

**SSR Output (with ssr-import-rewriter):**
```html
<script type="module">
  import { Button } from "/_vf_modules/components/Button.js?ssr=true&project=myapp&v=1705123456789";
  // React renders on server
</script>
```

**Client Hydration (with different rewriter):**
```javascript
// Client bundle
import { Button } from "/_vf_modules/components/Button.js";
// Different URL! Browser may cache different version
```

The version parameter (`v=`) changes between requests. If SSR and hydration use different cache busters, the browser may fetch a different module version than what SSR used, causing React hydration errors.

## Why It's Bad for Multi-Tenancy

### 1. Project Parameter Divergence

SSR adds `&project=projectSlug` to URLs, but browser transform doesn't:

```
SSR URL:    /_vf_modules/Button.js?ssr=true&project=alpha
Browser URL: /_vf_modules/Button.js

Different cache entries! Different module served!
```

### 2. Branch Parameter Missing

Only SSR includes branch information:

```typescript
// SSR includes branch
const branchParam = branch ? `&branch=${branch}` : "";
// Result: /_vf_modules/Button.js?ssr=true&branch=feature-x

// Browser has no branch awareness
// Result: /_vf_modules/Button.js (from main branch?)
```

Preview deployments on feature branches may SSR with feature branch code but hydrate with main branch code.

### 3. Cache Buster Race Condition

```typescript
// SSR: cacheBuster = Date.now()
return `from "/_vf_modules/${jsPath}?ssr=true&v=${cacheBuster}"`;
```

Each SSR request generates a new timestamp. If the browser caches the hydration script with a different `v=` parameter, stale code may execute.

## Failure Scenario

```
Timeline: Branch Preview Rendering
==================================

t0: User visits preview.myapp.com (branch: feature-dark-mode)

t1: SSR runs with ssr-import-rewriter
    -> Imports: /_vf_modules/Button.js?ssr=true&project=myapp&branch=feature-dark-mode&v=123
    -> Server renders <Button class="dark-theme">

t2: Browser hydration bundle (different transform path)
    -> Imports: /_vf_modules/Button.js
    -> No branch parameter!
    -> Server returns MAIN BRANCH code
    -> <Button class="light-theme">

t3: React hydration fails
    -> "Text content does not match server-rendered HTML"
    -> User sees flash of wrong styling
```

## Blast Radius

| Affected | Impact |
|----------|--------|
| All SSR-hydrated pages | Potential hydration mismatch |
| Branch previews | Wrong branch code may hydrate |
| Module caching | Cache pollution between SSR/client |
| React Fast Refresh | HMR may load wrong version |
| User experience | Content flash, broken interactivity |

## Solution

See **[011.0 - Import Rewriting RFC](./011.0-import-rewriting-rfc.md)** for the complete unified solution.

### Key Principle: Consistent URLs

Both SSR and browser MUST resolve to the same URL for the same import:

```typescript
// Unified strategy for @/ aliases
export const pathAliasStrategy: ImportRewriteStrategy = {
  name: "path-alias",
  priority: 20,
  matches: (s) => s.startsWith("@/"),
  rewrite: (specifier, ctx) => {
    const path = specifier.substring(2);
    const jsPath = ensureJsExtension(path);

    // SAME URL for both SSR and browser
    const baseUrl = ctx.moduleServerUrl ?? "";

    // Project and branch as immutable path segments, not query params
    const projectSegment = ctx.projectId;
    const branchSegment = ctx.branch ?? "main";

    return `${baseUrl}/_vf/${projectSegment}/${branchSegment}/${jsPath}`;
  },
};
```

### URL Structure Recommendation

Move from query parameters to path segments for cache-critical routing:

```
Before (divergent):
  SSR:     /_vf_modules/Button.js?ssr=true&project=foo&branch=bar&v=123
  Browser: /_vf_modules/Button.js

After (unified):
  Both:    /_vf/foo/bar/components/Button.js
```

Path-based routing ensures:
1. CDN caching works correctly
2. Browser cache keys are consistent
3. SSR and hydration fetch identical modules

## Testing

```typescript
// tests/resolution-consistency.test.ts
import { assertEquals } from "@std/assert";
import { getImportRewriter } from "../src/transforms/imports/unified-rewriter.ts";

Deno.test("@/ alias resolves identically for SSR and browser", async () => {
  const rewriter = getImportRewriter();

  const ssrContext = {
    target: "ssr" as const,
    runtime: "deno" as const,
    filePath: "/project/pages/index.tsx",
    projectDir: "/project",
    projectId: "test-project",
    reactVersion: "19.0.0",
    moduleServerUrl: "https://cdn.example.com",
  };

  const browserContext = {
    ...ssrContext,
    target: "browser" as const,
  };

  const code = `import { Button } from "@/components/Button";`;

  const ssrResult = await rewriter.rewrite(code, ssrContext);
  const browserResult = await rewriter.rewrite(code, browserContext);

  // Extract the import URL from both
  const ssrUrl = ssrResult.match(/from ["']([^"']+)["']/)?.[1];
  const browserUrl = browserResult.match(/from ["']([^"']+)["']/)?.[1];

  // MUST BE IDENTICAL
  assertEquals(ssrUrl, browserUrl, "SSR and browser should resolve to same URL");
});

Deno.test("branch parameter is included consistently", async () => {
  const rewriter = getImportRewriter();

  const context = {
    target: "browser" as const,
    runtime: "deno" as const,
    filePath: "/project/pages/index.tsx",
    projectDir: "/project",
    projectId: "test-project",
    branch: "feature-x",
    reactVersion: "19.0.0",
  };

  const code = `import { util } from "@/lib/util";`;
  const result = await rewriter.rewrite(code, context);

  // Branch should be in the URL
  assertEquals(result.includes("feature-x"), true);
});
```
