## The Problem

### DATA_FETCH_TIMEOUT_MS - Two Different Values

The documentation in `009-timeout-handling.md` mentions two different values for the same constant:

```typescript
// config/defaults.ts:7
export const DATA_FETCH_TIMEOUT_MS = 10000;  // 10 seconds

// rendering/orchestrator/pipeline.ts (documented)
const DATA_FETCH_TIMEOUT_MS = 15_000;  // 15 seconds - 50% HIGHER!
```

**Impact**: Depending on which file imports the constant, data fetching times out at different thresholds.

| Feature | `utils/semaphore.ts` | `ssr-module-loader/.../semaphore.ts` |

| Auto-release | Yes (`acquire()` wraps operation) | No (manual `release()`) |

| Named semaphores | Yes | No |
| Error type | `SemaphoreTimeoutError` | Returns `false` |

## Why It's Bad for Multi-Tenancy

### Inconsistent Behavior

When the same logical operation uses different timeout values:

```
Project A render path → imports from config/defaults.ts → 10s timeout
Project B render path → imports from pipeline.ts → 15s timeout
```

Same operation, different timeouts, based on import path!

### Debugging Nightmare

When investigating timeout issues:
1. Check which constant is being used
2. Trace the import chain
3. Discover different values in different files
4. Unclear which is "correct"

### Semaphore API Confusion

```typescript
// Using utils/semaphore.ts
await semaphore.acquire(async () => {
  // Operation
}); // Auto-releases

// Using ssr-module-loader semaphore
const acquired = await semaphore.tryAcquire(500);
if (acquired) {
  try {
    // Operation
  } finally {
    semaphore.release(); // Must manually release!
  }
}
```

Different calling conventions based on which Semaphore is imported.

## Solution

See **[009.0 - Timeout Handling RFC](./009.0-timeout-handling-rfc.md)** for the unified approach.

### Quick Fix: Consolidate Semaphores

1. Choose one Semaphore implementation (`utils/semaphore.ts` is more feature-complete)
2. Add `tryAcquire` method to match the other API
3. Deprecate the duplicate

```typescript
// utils/semaphore.ts - Add tryAcquire for compatibility

export class Semaphore {
  // ... existing code ...

  /** Try to acquire without auto-release (for manual control) */
  async tryAcquire(timeoutMs?: number): Promise<boolean> {
    const effectiveTimeout = timeoutMs ?? this.acquireTimeoutMs ?? 0;
    try {
      await this.waitForPermit(effectiveTimeout);
      return true;
    } catch (error) {
      if (error instanceof SemaphoreTimeoutError) {
        return false;
      }
      throw error;
    }
  }
}
```
