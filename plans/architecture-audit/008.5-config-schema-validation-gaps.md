# 008.5 - Config Schema Validation Gaps

## Severity: MEDIUM

## Location

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/config/schema.ts`
**Lines:** 23-318 (veryfrontConfigSchema), 322-346 (validateVeryfrontConfig)

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/config/loader.ts`
**Lines:** 101-109 (validateConfigShape), 189-208 (validateAndCacheConfig)

## The Problem

### Gap 1: Partial Schema with `.partial()` Throughout

The entire schema is wrapped in `.partial()`:

```typescript
// src/config/schema.ts:23-318
export const veryfrontConfigSchema = z
  .object({
    projectSlug: z.string().optional(),
    title: z.string().optional(),
    // ... all fields
  })
  .partial();  // Makes EVERYTHING optional
```

Every nested object is also `.partial()`:

```typescript
// src/config/schema.ts:49-62
build: z
  .object({
    outDir: z.string().optional(),
    trailingSlash: z.boolean().optional(),
    esbuild: z
      .object({
        wasmURL: z.string().url().optional(),
        worker: z.boolean().optional(),
      })
      .partial()   // Nested partial
      .optional(), // And optional
  })
  .partial()       // Parent partial
  .optional(),     // And optional
```

This means the schema validates almost any object shape.

### Gap 2: No Type Coercion or Strict Validation

User could provide wrong types that technically pass:

```typescript
// This passes validation:
export default {
  dev: {
    port: "3000",  // String, not number - still passes partial validation
  },
};
```

The schema uses `.optional()` everywhere, so missing/wrong types just become `undefined`.

### Gap 3: Unknown Keys Only Logged, Not Rejected

```typescript
// src/config/loader.ts:101-109
function validateConfigShape(userConfig: unknown): void {
  validateVeryfrontConfig(userConfig);
  if (!userConfig || typeof userConfig !== "object") return;

  const unknown = findUnknownTopLevelKeys(userConfig as Record<string, unknown>);
  if (unknown.length > 0) {
    serverLogger.warn(`Unknown config keys: ${unknown.join(", ")}. These will be ignored.`);
    // Unknown keys are LOGGED but NOT REJECTED
  }
}
```

Typos in config keys silently fail:

```typescript
// User typo:
export default {
  experimantal: { rsc: true },  // Typo - silently ignored
  experimental: undefined,       // Correct key - undefined
};
// Result: RSC stays disabled, user has no idea why
```

### Gap 4: No Nested Unknown Key Detection

`findUnknownTopLevelKeys` only checks top level:

```typescript
// src/config/schema.ts:352-356
const knownConfigKeys = new Set(Object.keys(veryfrontConfigSchema.shape));

export function findUnknownTopLevelKeys(input: Record<string, unknown>): string[] {
  return Object.keys(input).filter((key) => !knownConfigKeys.has(key));
}
```

Nested typos are not detected:

```typescript
export default {
  dev: {
    prot: 3000,   // Typo in nested key - NOT detected
  },
  tailwind: {
    pluginz: ["forms"],  // Typo - NOT detected
  },
};
```

### Gap 5: Security Fields Not Validated for Dangerous Values

```typescript
// src/config/schema.ts:101-118
security: z
  .object({
    auth: z
      .object({
        basic: basicAuthSchema.optional(),
        bearer: bearerAuthSchema.optional(),
      })
      .partial()
      .optional(),
    csp: z.record(z.array(z.string())).optional(),  // Any strings allowed
    remoteHosts: z.array(z.string().url()).optional(),
    cors: corsSchema.optional(),
    // ...
  })
```

User could set dangerous CSP:

```typescript
export default {
  security: {
    csp: {
      "script-src": ["'unsafe-eval'", "'unsafe-inline'", "*"],  // Dangerous but valid
      "default-src": ["*"],  // Allows everything
    },
  },
};
```

No validation for security-sensitive values.

## Why It's Bad

1. **Silent failures**: Config typos are ignored without error
2. **Type coercion issues**: Wrong types may cause runtime errors later
3. **Security risks**: Dangerous security config values not flagged
4. **Debugging difficulty**: User doesn't know their config is invalid

## Multi-Tenancy Impact

In production:
- Projects with invalid configs may render incorrectly
- No feedback to users that their config has issues
- Security misconfigurations go undetected

## Solution

See **[008.0 - Userland Config RFC](./008.0-userland-config-rfc.md)** for the complete solution.

Key changes:
1. Use `.strict()` instead of `.partial()` for object schemas
2. Add recursive unknown key detection for nested objects
3. Add explicit type coercion with error on wrong type
4. Add security validation for dangerous CSP values
5. Make warnings errors in production (or at least more visible)

```typescript
// Stricter schema example
export const veryfrontConfigSchema = z.object({
  dev: z.object({
    port: z.number().int().positive(),  // Required type
    host: z.string().default("localhost"),
    hmr: z.boolean().default(true),
  }).strict().optional(),  // Strict: reject unknown nested keys
}).strict();  // Strict: reject unknown top-level keys

// Security validation
function validateSecurityConfig(config: VeryfrontConfig): void {
  const csp = config.security?.csp;
  if (!csp) return;

  const dangerousDirectives = ["'unsafe-eval'", "'unsafe-inline'", "*"];
  for (const [directive, values] of Object.entries(csp)) {
    for (const value of values) {
      if (dangerousDirectives.includes(value)) {
        throw new ConfigSecurityWarning(
          `Dangerous CSP directive: ${directive} contains ${value}`
        );
      }
    }
  }
}
```
