# 003.3 - Multi-Tenancy Cache Isolation Violations

## Severity: CRITICAL - Cross-Project Data Leakage

## The Problem

Multiple projects share the same renderer pod, but several cache systems don't properly scope their keys by project. This can cause:

1. **Data leakage**: Project A sees cached data from Project B
2. **Cache poisoning**: Project A's cache entry overwrites Project B's
3. **Incorrect behavior**: Project A gets transforms compiled for Project B's configuration

## Critical Violations Found

### 1. HTTP Bundle Cache - NO Project Scope

**Location:** `src/transforms/esm/http-cache.ts:223-228`

```typescript
// Cache key uses ONLY the URL hash - no project identifier!
const cacheKey = `${cacheDir}:${normalizedUrl}`;
const cachePath = join(cacheDir, `http-${simpleHash(normalizedUrl)}.mjs`);
```

**Problem:** All projects share the same HTTP bundles by URL. While this seems efficient, it breaks when:
- Different projects use different React versions
- Different projects have different import maps
- A project's import map resolves `react` to a specific version

**Scenario:**
```
Project A: import map resolves "react" -> "react@18.2.0"
Project B: import map resolves "react" -> "react@18.3.1"

Pod caches: https://esm.sh/react@18.2.0 (from Project A)

Project B request hits cache, gets 18.2.0 instead of 18.3.1!
```

### 2. Transform Cache - Content Hash Only, No Project

**Location:** `src/transforms/esm/transform-cache.ts:63-69`

```typescript
export function generateCacheKey(
  filePath: string,
  contentHash: string,
  ssr: boolean = false,
  studioEmbed: boolean = false,
): string {
  return buildTransformCacheKey(filePath, contentHash, ssr, studioEmbed);
}

// keys.ts:260-269
export function buildTransformCacheKey(
  filePath: string,
  contentHash: string,
  ssr: boolean = false,
  studioEmbed: boolean = false,
): string {
  const ssrKey = ssr ? "ssr" : "browser";
  const studioKey = studioEmbed ? ":studio" : "";
  return `v${TRANSFORM_CACHE_VERSION}:${filePath}:${contentHash}:${ssrKey}${studioKey}`;
}
```

**Problem:** Key includes `filePath` + `contentHash` but NO project identifier. Two projects with the same file path and content will share cached transforms.

**Scenario:**
```
Project A: pages/index.tsx with content hash abc123
           Uses jsxImportSource: "@emotion/react"

Project B: pages/index.tsx with content hash abc123 (same content!)
           Uses jsxImportSource: "react"

Pod caches Project A's transform with Emotion JSX runtime.
Project B gets Project A's transform - wrong JSX runtime!
```

### 3. Cross-Project Import Cache - Specifier Only

**Location:** `src/modules/react-loader/ssr-module-loader/loader.ts:309-311`

```typescript
// Cross-project import cache uses ONLY specifier as key
const cacheKey = specifier;  // e.g., "react", "lodash"
const cachedEntry = globalCrossProjectCache.get(cacheKey);
```

**Problem:** Bare specifiers like "react" are cached globally across all projects.

**Scenario:**
```
Project A imports "react" -> resolves to esm.sh/react@18.2.0
Cached globally as: "react" -> tempPath

Project B imports "react" -> should resolve to esm.sh/react@18.3.1
Gets Project A's cached path instead!
```

### 4. Global Module Cache - Limited Scoping

**Location:** `src/modules/react-loader/ssr-module-loader/cache/memory.ts:27-31`

```typescript
export const globalModuleCache = new LRUCache<string, ModuleCacheEntry>({
  max: 10000,
});

export const globalCrossProjectCache = new LRUCache<string, ModuleCacheEntry>({
  max: 1000,
});
```

**Problem:** These are truly GLOBAL across all projects on the pod. While SSR module cache keys include `projectId`, the cross-project cache does not.

### 5. Temp Directory Cache - Project Scoped BUT...

**Location:** `src/modules/react-loader/ssr-module-loader/loader.ts:1033-1041`

```typescript
const cacheKey = `${baseCacheDir}|${projectKey}|${sourceKey}`;
const existingDir = globalTmpDirs.get(cacheKey);
if (existingDir) return existingDir;
```

**This is CORRECT** - temp dirs are scoped by project. However, the CONTENTS written to these dirs reference global HTTP bundles, creating indirect cross-project contamination.

## Cache Key Audit

| Cache | Project Scoped? | Key Format | Violation |
|-------|-----------------|------------|-----------|
| SSR Module | YES | `v{ver}:{projectId}:{contentSourceId}:{reactVer}:{path}` | None |
| HTTP Bundle | **NO** | `{cacheDir}:{urlHash}` | Cross-project shared |
| Transform | **NO** | `v{ver}:{path}:{contentHash}:{ssr}` | Cross-project shared |
| Cross-Project Import | **NO** | `{specifier}` | Bare specifier only |
| File Cache | YES | `{projectSlug}:{contentSourceId}:{path}` | None |
| Context-Aware | YES | `{projectId}:{environment}:{releaseKey}:{version}` | None |
| Tailwind CSS | YES | `{projectSlug}:{stylesheetHash}` | None |
| MDX-ESM | YES | `{cacheDir}/{projectKey}/{sourceKey}/...` | None |

## Failure Scenarios

### Scenario 1: React Version Mismatch

```
1. Project A (React 18.2.0) renders page, HTTP bundles cached globally
2. Project B (React 18.3.1) renders page
3. Project B's transform references cached 18.2.0 bundles
4. Runtime: Multiple React instances, hooks break
5. Error: "Invalid hook call" or silent behavior bugs
```

### Scenario 2: JSX Transform Collision

```
1. Project A uses Emotion JSX runtime, transform cached
2. Project B uses default React JSX runtime
3. Same file path, same content hash -> cache hit
4. Project B gets Emotion-compiled code
5. Runtime error: "jsx is not a function" or missing styles
```

### Scenario 3: Import Map Conflict

```
1. Project A's import map: "ui" -> "@company/ui-v2"
2. Project B's import map: "ui" -> "@company/ui-v3"
3. Cross-project cache stores "ui" -> v2 temp path
4. Project B gets v2 components instead of v3
5. Result: Wrong component versions, API mismatches
```

## The Fix

### Option 1: Add Project ID to All Cache Keys

```typescript
// HTTP Bundle cache key
const cacheKey = `${projectId}:${cacheDir}:${normalizedUrl}`;

// Transform cache key
export function buildTransformCacheKey(
  projectId: string,  // ADD THIS
  filePath: string,
  contentHash: string,
  ssr: boolean = false,
  studioEmbed: boolean = false,
): string {
  return `v${TRANSFORM_CACHE_VERSION}:${projectId}:${filePath}:${contentHash}:${ssrKey}${studioKey}`;
}

// Cross-project import key
const cacheKey = `${projectId}:${specifier}`;
```

### Option 2: Namespace Caches by Project Context

```typescript
// Use AsyncLocalStorage to inject project context
const cacheContext = new AsyncLocalStorage<{ projectId: string }>();

function getCacheKey(baseKey: string): string {
  const ctx = cacheContext.getStore();
  if (!ctx) throw new Error("Cache operation outside project context");
  return `${ctx.projectId}:${baseKey}`;
}
```

### Option 3: Project-Specific Cache Instances

```typescript
// Instead of global caches, create per-project instances
class ProjectCacheManager {
  private caches = new Map<string, LRUCache>();

  getCache(projectId: string): LRUCache {
    let cache = this.caches.get(projectId);
    if (!cache) {
      cache = new LRUCache({ max: 1000 });
      this.caches.set(projectId, cache);
    }
    return cache;
  }
}
```

## Solution

See **[003.0 - Cache Consistency RFC](./003.0-cache-consistency-rfc.md)** for the complete solution.

The key insight is that ALL caches in a multi-tenant environment MUST include the project identifier in their keys. Shared caches are only safe for truly immutable, project-independent data (e.g., npm packages by exact version).
