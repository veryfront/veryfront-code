# 019.3: Cache Key Builder Duplication

## Severity: MEDIUM

## Locations

30+ different cache key patterns found across codebase.

## The Problem

Cache keys built ad-hoc with no standard format. Risk of collisions, inconsistent invalidation, and debugging difficulty.

### Sample of Patterns Found

```typescript
// Pattern 1: Simple concatenation
const key = `${projectId}:${hash}`;

// Pattern 2: With version prefix
const key = `v1:${type}:${id}`;

// Pattern 3: Environment-scoped
const key = `cache:${slug}:${env}:${hash}`;

// Pattern 4: Nested structure
const key = `${namespace}/${project}/${file}:${version}`;

// Pattern 5: With metadata
const key = `transform:${input}:${options.target}:${options.minify}`;

// Pattern 6: Hash-only
const key = hashString(JSON.stringify({ project, path, version }));

// ... 24 more variations
```

### Collision Risk

```
Key pattern A: `${projectId}:${hash}`
Key pattern B: `${cacheType}:${id}`

Project "transform" + hash "abc123"
→ "transform:abc123"

Cache type "transform" + id "abc123"
→ "transform:abc123"

COLLISION! Different data, same key.
```

## Relationship to Task 027

Task 027 (Cache Key Standard) defines the target format. This document identifies the current state and migration scope.

### Target Format (from Task 027)

```
v{version}:{type}:{scope?}:{identifier}:{hash}
```

Examples:
- `v18:transform:proj-123:pages/index.tsx:abc123`
- `v18:module:esm.sh/react@18.3.1:def456`

## Proposed Fix

### Centralized Cache Key Builder

```typescript
// src/core/cache/key-builder.ts

const CACHE_VERSION = 18;

export type CacheType =
  | "transform"
  | "module"
  | "render"
  | "data"
  | "css"
  | "layout"
  | "config";

export interface CacheKeyParams {
  type: CacheType;
  scope?: string;  // projectId, environment, etc.
  identifier: string;  // path, url, etc.
  hash?: string;  // content hash
  variant?: string;  // browser, server, etc.
}

export function buildCacheKey(params: CacheKeyParams): string {
  const parts = [`v${CACHE_VERSION}`, params.type];

  if (params.scope) {
    parts.push(params.scope);
  }

  parts.push(params.identifier);

  if (params.hash) {
    parts.push(params.hash);
  }

  if (params.variant) {
    parts.push(params.variant);
  }

  return parts.join(":");
}

// Type-safe builders for each cache type
export const transformKey = (projectId: string, path: string, hash: string) =>
  buildCacheKey({ type: "transform", scope: projectId, identifier: path, hash });

export const moduleKey = (url: string, hash: string) =>
  buildCacheKey({ type: "module", identifier: url, hash });

export const renderKey = (projectId: string, path: string, hash: string) =>
  buildCacheKey({ type: "render", scope: projectId, identifier: path, hash });
```

### Migration Example

```typescript
// Before
const cacheKey = `${projectSlug}:${stylesheetHash}`;

// After
import { buildCacheKey } from "@/core/cache/key-builder";
const cacheKey = buildCacheKey({
  type: "css",
  scope: projectSlug,
  identifier: "tailwind",
  hash: stylesheetHash,
});
```

The adhoc patterns are context-specific (tailwind hash, http cache dir, module loader) and don't justify forcing into the central builder. The existing centralization is sufficient.
