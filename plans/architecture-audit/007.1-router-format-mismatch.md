# 007.1 - Router Format Mismatch

## Severity: MEDIUM

## Location

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/config/types.ts`
**Lines:** 21, 236-238

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/cli/commands/generate.ts`
**Lines:** 42, 47-48

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/server/dev-server/route-discovery.ts`
**Lines:** 81, 85-86

## The Problem

Two different formats represent the same concept (router type):

```typescript
// src/config/types.ts:21
router?: "app" | "pages";

// src/config/types.ts:236-238
generate?: {
  preferredRouter?: "app-router" | "pages-router";
};
```

The generate command attempts to handle both formats:

```typescript
// src/cli/commands/generate.ts:42, 47-48
async function getPreferredRouter(
  projectDir: string,
): Promise<"pages-router" | "app-router"> {
  // ...
  const pref = cfg?.generate?.preferredRouter ?? cfg?.router;
  if (pref === "app-router" || pref === "pages-router") return pref;
  // ...
}
```

But `cfg.router` is `"app" | "pages"`, which will NEVER match `"app-router" | "pages-router"`:

```typescript
// If user sets router: "app", the fallback returns "pages-router" instead of "app-router"!
const pref = cfg?.generate?.preferredRouter ?? cfg?.router;  // pref = "app"
if (pref === "app-router" || pref === "pages-router") return pref;  // FALSE!
// Falls through to return "pages-router"  <-- WRONG!
```

Meanwhile, route-discovery.ts only handles the short format:

```typescript
// src/server/dev-server/route-discovery.ts:81, 85-86
const preferredRouter = this.config?.router;  // "app" | "pages" | undefined
if (preferredRouter === "app") candidates.push({ type: "app", dir: "app" });
else if (preferredRouter === "pages") candidates.push({ type: "pages", dir: "pages" });
// "app-router" and "pages-router" are NEVER handled here
```

## Why It's Bad for Multi-Tenancy

In a multi-tenant environment, different projects may use different config formats:

```
Project Alpha: router: "app"           → route-discovery works, CLI broken
Project Beta:  generate.preferredRouter: "app-router" → CLI works, route-discovery ignores
```

When both projects are served by the same renderer pod, behavior is inconsistent:
- Developers see different behavior based on which format they used
- Documentation becomes confusing about which format to use
- Silent fallback to wrong default causes hard-to-debug issues

## Failure Scenario

```
Step 1: Developer sets router: "app" in veryfront.config.ts
Step 2: Developer runs "vf generate page my-page"
Step 3: generate.ts reads config, pref = "app"
Step 4: pref === "app-router" is FALSE, pref === "pages-router" is FALSE
Step 5: Falls through to default: return "pages-router"
Step 6: Page is generated in pages/my-page.mdx instead of app/my-page/page.tsx
Step 7: Developer is confused why their app-router project has pages-router structure
```

## Solution

See **[007.0 - Config Normalization RFC](./007.0-config-normalization-rfc.md)** for the complete solution.

Specifically, the `normalizeRouter()` function will convert all formats to canonical `"app" | "pages"`:

```typescript
function normalizeRouter(
  router?: "app" | "pages",
  preferredRouter?: "app-router" | "pages-router"
): "app" | "pages" {
  if (preferredRouter === "app-router") return "app";
  if (preferredRouter === "pages-router") return "pages";
  return router ?? "pages";
}
```

After normalization, all consumers only see `"app" | "pages"` - no format mismatches.
