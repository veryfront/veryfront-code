# 001.2 - Unsafe Type Casting Patterns

## Severity: 🟠 HIGH - Type Safety Violation

## The Pattern

Throughout the codebase, `getUnderlyingAdapter()` is called and immediately cast to an ad-hoc interface:

```typescript
const wrappedAdapter = fs.getUnderlyingAdapter() as {
  getProjectData?: () => { updated_at?: string } | undefined;
};
```

This is **type-unsafe** and creates undocumented contracts.

## Every Instance Found

### 1. pipeline.ts:651-654 - Project Metadata

```typescript
if (fs && isExtendedFSAdapter(fs) && fs.isVeryfrontAdapter()) {
  const wrappedAdapter = fs.getUnderlyingAdapter() as {
    getProjectData?: () => { updated_at?: string } | undefined;  // 🔴 Ad-hoc type
  };
  projectUpdatedAt = wrappedAdapter.getProjectData?.()?.updated_at;
}
```

**Problem**: The `getProjectData` method is NOT in any interface. It's a runtime guess.

### 2. html.ts:320-332 - getAllSourceFiles

```typescript
const wrappedFs = this.config.adapter.fs as unknown as {  // 🔴 Double cast through unknown!
  getUnderlyingAdapter?: () => unknown;
};

if (typeof wrappedFs.getUnderlyingAdapter !== "function") return classes;

const fsAdapter = wrappedFs.getUnderlyingAdapter() as {
  getAllSourceFiles?: () =>
    | Array<{ path: string; content?: string }>
    | Promise<Array<{ path: string; content?: string }>>;  // 🔴 Ad-hoc type
};
```

**Problems**:
1. `as unknown as` - Double cast is a red flag
2. Runtime type checking (`typeof ... !== "function"`)
3. No compile-time safety
4. Method signature guessed, not guaranteed

### 3. styles-css-handler.ts:115-131 - Same Pattern

## Why This Happens

The `FSAdapter` interface doesn't include adapter-specific methods like:
- `getProjectData()`
- `getAllSourceFiles()`
- `getEntityIdForPath()`

These are **Veryfront API adapter specific** but accessed through generic type casting.

## The Real Issue

This pattern reveals a **leaky abstraction**:

1. Code wants API-adapter-specific data (project metadata, source files)
2. But the adapter interface doesn't expose these methods
3. So developers cast through `unknown` to access them
4. Result: No compile-time safety, no IDE autocompletion, no documentation

## Impact on Multi-Tenancy

When these casts fail (method doesn't exist):
- `getProjectData()` → `undefined` → Cache keys differ between adapters
- `getAllSourceFiles()` → Falls back to local file scanning → Different CSS extraction
- `getEntityIdForPath()` → Falls back to file path → Entity IDs inconsistent

## Correct Solution

### Option A: Extend the Interface

```typescript
interface ExtendedFileSystemAdapter {
  // Existing methods...

  // Add adapter-specific methods as optional
  getProjectMetadata?(): ProjectMetadata | undefined;
  getAllSourceFiles?(): Promise<SourceFile[]>;
  getEntityIdForPath?(path: string): string | undefined;
}
```

### Option B: Capability Queries

```typescript
interface ExtendedFileSystemAdapter {
  hasCapability(cap: 'projectMetadata' | 'sourceFiles' | 'entityIds'): boolean;
  getProjectMetadata(): ProjectMetadata;  // Only call if hasCapability('projectMetadata')
}
```

### Option C: Separate Interfaces

```typescript
interface ProjectMetadataProvider {
  getProjectData(): { updated_at?: string };
}

function isProjectMetadataProvider(fs: FSAdapter): fs is ProjectMetadataProvider {
  return 'getProjectData' in fs && typeof fs.getProjectData === 'function';
}
```

## Solution

See **[001.0 - Unified Adapter RFC](./001.0-unified-adapter-rfc.md)** for the complete solution.

The fix is to add these methods to the unified interface as optional methods that return `undefined` when not available (NOT different behavior):
- `getProjectMetadata?()` - replaces `getProjectData` casting
- `getEntityId?(path)` - replaces `getEntityIdForPath` casting
- `getAllSourceFiles?()` - replaces source file access casting
