## Why It's Marked LOW Severity

Unlike most global state issues, this cache is **content-addressed** and therefore safe to share:

### Cache Key Analysis

```typescript
// Line 223: Cache key includes BOTH cacheDir AND URL
const cacheKey = `${cacheDir}:${normalizedUrl}`;
```

The cache key format is:
```
/Users/mattboon/.cache/veryfront-http-bundle:https://esm.sh/react@18.3.1?target=es2022
```

This means:
1. **Same URL = Same Content**: esm.sh returns deterministic content for a given URL
2. **Different Projects = Same Cached File**: If ProjectA and ProjectB both need `react@18.3.1`, they share the same cached file
3. **No Credential Leakage**: HTTP module URLs are public (esm.sh, deno.land, etc.)

### Safe Sharing Pattern

```
ProjectA Request: esm.sh/react@18.3.1 → Cache Miss → Fetch → Store → Return path
ProjectB Request: esm.sh/react@18.3.1 → Cache Hit → Return same path

Result: Both projects get the same cached file
        This is CORRECT behavior - same input, same output
```

## Where Caution Is Needed

### processingStack Concurrent Access

```typescript
// Line 267-273: Processing stack prevents circular deps
if (processingStack.has(normalizedUrl)) {
  logger.debug("[HTTP-CACHE] Circular dependency detected, returning expected path", {
    url: normalizedUrl,
  });
  return cachePath;
}

// Line 345-350: Add/remove from stack during processing
processingStack.add(normalizedUrl);
try {
  code = await rewriteModuleImports(code, normalizedUrl, options);
} finally {
  processingStack.delete(normalizedUrl);
}
```

**Potential Issue:** If two projects simultaneously request the same URL:
1. ProjectA adds URL to `processingStack`
2. ProjectB sees URL in stack, assumes circular dependency
3. ProjectB returns expected path (which might not exist yet!)

**Risk Assessment:** LOW because:
- Both projects want the same content
- Worst case: One project gets a brief "module not found" that resolves on retry
- Recovery mechanism exists in `recoverHttpBundleByHash()`

### Memory Pressure from Large LRU

```typescript
const HTTP_MODULE_CACHE_MAX_ENTRIES = 10000; // from constants
```

With 10,000 entries and average key+value of ~200 bytes:
- Memory: ~2MB per pod (acceptable)
- No per-project isolation needed

## What Would Actually Be Dangerous

If the HTTP cache stored **project-specific content**, that would be dangerous:

```typescript
// HYPOTHETICAL DANGEROUS PATTERN (NOT in actual code)
const cachedPaths = new LRUCache<string, string>({ maxEntries: 10000 });

async function cacheProjectModule(projectId: string, url: string): Promise<string> {
  // WRONG: Using only URL as key, but content varies by project
  const cacheKey = url;

  // This would leak Project A's customized module to Project B!
  return cachedPaths.get(cacheKey) ?? await fetchAndCache(projectId, url);
}
```

The actual implementation uses content-addressed keys (URL hash) which avoids this problem.

## Comparison with Other Caches

| Cache | Content-Addressed? | Safe to Share? |
|-------|-------------------|----------------|
| `http-cache.ts` cachedPaths | YES (URL hash) | YES |
| `module-cache.ts` moduleCache | NO (projectId:path) | YES (project-scoped key) |
| `domain-lookup.ts` domainCache | YES (domain) | YES |
| `head-collector.ts` collected | NO | **NO - Needs AsyncLocalStorage** |

## Recommendation

No immediate changes needed. The HTTP module cache is correctly designed for sharing.

### Optional Improvements

1. **Add Comments Explaining Safety**:
```typescript
/**
 * Pod-level module cache singleton.
 * SAFE to share across projects because:
 * - Keys are content-addressed (URL hash → file path)
 * - Same URL always returns same content (esm.sh is deterministic)
 * - No project-specific data is stored
 */
const cachedPaths = new LRUCache<string, string>({ ... });
```

2. **Add Metrics for Cache Hit Ratio**:
```typescript
// Track how effectively we're sharing across projects
let cacheHits = 0;
let cacheMisses = 0;

// In cacheHttpModule()
const existing = cachedPaths.get(cacheKey);
if (existing) {
  cacheHits++;
  return existing;
}
cacheMisses++;
```

3. **Consider Per-Pod vs Distributed Trade-off**:
The distributed cache (Redis) is already used for cross-pod sharing. The LRU is just a fast local cache. This layering is appropriate.

## Related

See **[012.0 - HTTP Clients RFC](./012.0-http-clients-rfc.md)** for the overall HTTP client architecture.

The module cache is separate from the HTTP client concerns (retry, timeout, circuit breaker) because caching happens at a different layer:

```
Request Flow:
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│ Transform   │ ──▶ │ http-cache.ts│ ──▶ │ HTTP Client │
│ Pipeline    │     │ (LRU+Redis)  │     │ (fetch)     │
└─────────────┘     └──────────────┘     └─────────────┘
                         │
                    Cache Hit?
                    ├── Yes: Return file path
                    └── No: Call HTTP Client, cache result
```

The HTTP client RFC focuses on the `fetch()` layer, while this document analyzes the caching layer above it.
