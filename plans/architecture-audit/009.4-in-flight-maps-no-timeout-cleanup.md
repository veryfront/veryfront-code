# 009.4 - In-Flight Request Maps Without Timeout Cleanup

## Severity: HIGH

## Location

**Files:**
- `/Users/mattboon/Sites/veryfront-renderer/src/server/utils/domain-lookup.ts` (line 26)
- `/Users/mattboon/Sites/veryfront-renderer/src/modules/react-loader/ssr-module-loader/cache/memory.ts` (line 35)

## The Problem

Multiple components use Maps to track in-flight requests for deduplication, but several lack timeout-based cleanup:

### Domain Lookup - No Cleanup

```typescript
// server/utils/domain-lookup.ts:26
const inFlightRequests = new Map<string, Promise<DomainLookupResult | null>>();

// Line 71-78: Only removed on promise resolution
const inFlight = inFlightRequests.get(cacheKey);
if (inFlight) {
  logger.debug("[DomainLookup] Waiting for in-flight request", { domain });
  return inFlight;  // Waits forever if promise never resolves!
}

// Line 78-95
inFlightRequests.set(cacheKey, requestPromise);
try {
  const result = await requestPromise;
  // ...
} finally {
  inFlightRequests.delete(cacheKey);  // Only deleted if promise completes
}
```

**Issue**: If `requestPromise` never resolves (hanging fetch), the entry stays forever.

### Global In-Progress - Timeout Exists but Entry Stays

```typescript
// ssr-module-loader/cache/memory.ts:35
export const globalInProgress = new Map<string, Promise<void>>();

// loader.ts:571-591
const existingTransform = globalInProgress.get(inProgressKey);

        ),
      { "ssr.file": filePath.split("/").pop() || filePath },
    );
    return;
  } catch (error) {
    globalInProgress.delete(inProgressKey);  // Deleted on timeout...
    logger.warn("[SSR-MODULE-LOADER] In-progress transform timed out, retrying", {
```

**Partial Fix**: The waiting side times out after 30s, but the entry is only deleted by the **waiter**, not by the original transform. If no one waits, the entry persists.

### Pending Revalidations - No Explicit Cleanup


## Why It's Bad for Multi-Tenancy

### Memory Leak Pattern

Each in-flight map that doesn't clean up stale entries:

1. **Accumulates entries** for every unique key with issues
2. **Holds references** to Promises, closures, and context objects
3. **Grows unbounded** over pod lifetime
4. **Affects all projects** since maps are global

### Cross-Project Impact

```
Project A: Slow API causes hanging domain lookups
           inFlightRequests grows with A's domains

Project B: Normal operation
           But shares same inFlightRequests Map
           Memory pressure from A affects B's allocations

All Projects: GC pressure increases
              Response times degrade
              Eventually OOM kill
```

## Code Evidence: Proper Cleanup Pattern Exists

The `ReadOperations` class has a proper cleanup pattern:

```typescript
// platform/adapters/fs/veryfront/read-operations.ts:33-43
const IN_FLIGHT_REQUEST_TIMEOUT_MS = 15_000;
const MAX_IN_FLIGHT_REQUESTS = 100;

interface InFlightEntry {
  promise: Promise<string>;
  startedAt: number;  // Timestamp for age-based cleanup!
}

export class ReadOperations {
  private readonly inFlightRequests = new Map<string, InFlightEntry>();
  private lastCleanupTime = 0;
```

```typescript
// read-operations.ts:74-98
private cleanupStaleInFlightRequests(): void {
  const now = Date.now();
  if (now - this.lastCleanupTime < 1000) return;

  this.lastCleanupTime = now;

  let cleanedCount = 0;

  // Age-based cleanup
  for (const [key, entry] of this.inFlightRequests) {
    if (now - entry.startedAt > IN_FLIGHT_REQUEST_TIMEOUT_MS) {
      this.inFlightRequests.delete(key);
      cleanedCount++;
    }
  }

  // Size-based eviction
  if (this.inFlightRequests.size > MAX_IN_FLIGHT_REQUESTS) {
    const entries = [...this.inFlightRequests.entries()].sort((a, b) =>
      a[1].startedAt - b[1].startedAt
    );
    const toRemove = entries.slice(0, this.inFlightRequests.size - MAX_IN_FLIGHT_REQUESTS);
    for (const [key] of toRemove) {
      this.inFlightRequests.delete(key);
      cleanedCount++;
    }
  }
}
```

This pattern should be applied to all in-flight tracking Maps.

## Solution

See **[009.0 - Timeout Handling RFC](./009.0-timeout-handling-rfc.md)** for the complete solution.

### Quick Fix: Add Cleanup to Domain Lookup

```typescript
// server/utils/domain-lookup.ts

interface InFlightEntry {
  promise: Promise<DomainLookupResult | null>;
  startedAt: number;
}

const inFlightRequests = new Map<string, InFlightEntry>();
const IN_FLIGHT_TIMEOUT_MS = 15_000;
const MAX_IN_FLIGHT = 100;

function cleanupStaleRequests(): void {
  const now = Date.now();
  for (const [key, entry] of inFlightRequests) {
    if (now - entry.startedAt > IN_FLIGHT_TIMEOUT_MS) {
      inFlightRequests.delete(key);
    }
  }
  // Also enforce max size
  if (inFlightRequests.size > MAX_IN_FLIGHT) {
    const oldest = [...inFlightRequests.entries()]
      .sort((a, b) => a[1].startedAt - b[1].startedAt)
      .slice(0, inFlightRequests.size - MAX_IN_FLIGHT);
    for (const [key] of oldest) {
      inFlightRequests.delete(key);
    }
  }
}

export function lookupProjectByDomain(...): Promise<...> {
  cleanupStaleRequests();  // Call on every lookup

  const cacheKey = normalizeDomain(domain);
  const now = Date.now();

  const entry = inFlightRequests.get(cacheKey);
  if (entry && now - entry.startedAt < IN_FLIGHT_TIMEOUT_MS) {
    return entry.promise;
  }

  const requestPromise = fetchDomainLookup(domain, config);
  inFlightRequests.set(cacheKey, { promise: requestPromise, startedAt: now });

  // ... rest unchanged
}
```

### Quick Fix: Add Periodic Cleanup to globalInProgress

```typescript
// ssr-module-loader/cache/memory.ts

interface InProgressEntry {
  promise: Promise<void>;
  startedAt: number;
}

export const globalInProgress = new Map<string, InProgressEntry>();

// Call this from loader.ts before setting new entries
export function cleanupStaleInProgress(): void {
  const now = Date.now();
  for (const [key, entry] of globalInProgress) {
    if (now - entry.startedAt > IN_PROGRESS_WAIT_TIMEOUT_MS * 2) {
      globalInProgress.delete(key);
    }
  }
}
```
