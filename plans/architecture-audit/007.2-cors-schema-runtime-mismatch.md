## The Problem

The Zod schema and TypeScript types allow different values than what the runtime actually supports:

```typescript
// src/config/schema.ts:5 - SCHEMA (validation)
const corsSchema = z.union([
  z.boolean(),
  z.object({ origin: z.string().optional() }).strict()  // Only string!
]);

// src/config/types.ts:87 - USER-FACING TYPE
cors?: boolean | { origin?: string };  // Only string!

// src/security/http/cors/types.ts:3-10 - RUNTIME TYPE
export interface CORSConfig {
  origin?: string | string[] | OriginValidator;  // ALSO arrays and functions!
  credentials?: boolean;
  methods?: string[];
  allowedHeaders?: string[];
  exposedHeaders?: string[];
  maxAge?: number;
}

// src/security/http/cors/validators.ts:42-56 - RUNTIME HANDLING
if (Array.isArray(corsConfig.origin)) {         // Handles arrays!
  const allowed = corsConfig.origin.includes(requestOrigin);
  // ...
}

// src/security/http/cors/validators.ts:115
if (typeof corsConfig.origin === "function") {  // Handles functions!
  // ...
}
```

## Why It's Bad for Multi-Tenancy

### Security Gap

Projects may attempt to configure strict CORS with an array of allowed origins:

```typescript
// veryfront.config.ts
export default {
  security: {
    cors: {
      origin: ["https://app.example.com", "https://admin.example.com"],
    }
  }
}
```

This FAILS Zod validation at load time, but developers might expect it to work because:
1. The runtime code clearly supports arrays
2. Other frameworks (like Express CORS) support this pattern
3. The error message says "origin must be a string" but doesn't explain the workaround

### Type Confusion Across Projects

In a multi-tenant environment:

```
Project Alpha: Uses cors: true (works)
Project Beta: Uses cors: { origin: "https://example.com" } (works)
Project Gamma: Uses cors: { origin: ["..."] } (FAILS at config load)
```

Project Gamma's failure affects the entire pod's ability to serve that project, while the runtime clearly supports the feature.

## Solution

See **[007.0 - Config Normalization RFC](./007.0-config-normalization-rfc.md)** for the complete solution.

The schema should accept all formats the runtime supports:

```typescript
// Expanded schema
const corsOriginSchema = z.union([
  z.string(),
  z.array(z.string()),
  z.function().args(z.string()).returns(z.union([z.boolean(), z.string()])),
]);

const corsSchema = z.union([
  z.boolean(),
  z.object({
    origin: corsOriginSchema.optional(),
    credentials: z.boolean().optional(),
    methods: z.array(z.string()).optional(),
    allowedHeaders: z.array(z.string()).optional(),
    exposedHeaders: z.array(z.string()).optional(),
    maxAge: z.number().positive().optional(),
  }),
]);
```

The normalizer then converts all inputs to a consistent internal format:

```typescript
function normalizeCors(cors?: CorsInput): NormalizedCorsConfig {
  // Handle boolean, string, array, function - all converted to consistent format
}
```
