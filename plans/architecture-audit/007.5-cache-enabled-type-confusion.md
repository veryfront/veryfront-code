# 007.5 - Cache Config enabled/type Confusion

## Severity: MEDIUM

## Location

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/config/types.ts`
**Lines:** 37-54

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/utils/bundle-manifest-init.ts`

## The Problem

Cache configuration has inconsistent patterns for enabling/disabling features:

```typescript
// src/config/types.ts:37-54
cache?: {
  dir?: string;
  bundleManifest?: {
    type?: "redis" | "kv" | "memory";  // What does this mean when enabled: false?
    redisUrl?: string;
    keyPrefix?: string;
    ttl?: number;
    enabled?: boolean;  // Explicit enabled flag
  };
  render?: {
    type?: "memory" | "filesystem" | "kv" | "redis";  // No enabled flag - implicit?
    ttl?: number;
    maxEntries?: number;
    kvPath?: string;
    redisUrl?: string;
    redisKeyPrefix?: string;
    // NO enabled field!
  };
};
```

`bundleManifest` has BOTH `type` AND `enabled`, while `render` only has `type`. The relationship is unclear:

```typescript
// src/utils/bundle-manifest-init.ts:16-26
const manifestConfig = config.cache?.bundleManifest;
const enabled = manifestConfig?.enabled ?? mode === "production";  // Default based on mode

if (!enabled) {
  logger.info("[bundle-manifest] Bundle manifest disabled");
  setBundleManifestStore(new InMemoryBundleManifestStore());
  return;
}

// What if: type: "redis" but enabled: false?
// The type is configured but not used - confusing!
const storeType = manifestConfig?.type ?? adapter?.env.get("VERYFRONT_BUNDLE_MANIFEST_TYPE") ??
  "memory";
```

## Why It's Bad for Multi-Tenancy

### Semantic Confusion

Different projects may have different interpretations:

```yaml
# Project Alpha: Explicit disable, but with type configured
cache:
  bundleManifest:
    type: "redis"        # Configured!
    redisUrl: "redis://prod-cache"
    enabled: false       # But disabled?
```

What does this mean?
- "Disable caching entirely" - ignore type
- "Disable persistent caching but use memory" - current behavior
- "Disable this config but inherit from environment" - unclear

### Default Behavior Varies by Mode

```typescript
const enabled = manifestConfig?.enabled ?? mode === "production";
```

- In production: `enabled` defaults to `true`
- In development: `enabled` defaults to `false`

Projects may not realize their cache config behaves differently in prod vs dev.

### Inconsistency with render Cache

```typescript
// render cache has NO enabled field
render?: {
  type?: "memory" | "filesystem" | "kv" | "redis";
  // How do you disable render cache? Set type: undefined?
  // Or does type: "memory" act as "disabled"?
}
```

## Failure Scenario

```
Step 1: DevOps configures production with:
        cache.bundleManifest.type = "redis"
        cache.bundleManifest.redisUrl = "redis://..."
        # Forgets to set enabled: true (relies on production default)

Step 2: Staging uses same config but PRODUCTION_MODE=false
        → enabled defaults to false
        → Redis config ignored, falls back to memory
        → Staging behaves differently than prod

Step 3: Bug found in staging that doesn't repro in prod
        → Hours debugging reveals cache difference
        → Config looked identical!
```

## Multi-Tenancy Scenario

```
Timeline:
=========
t0: Renderer pod starts in production mode

t1: Project Alpha config: { cache: { bundleManifest: { type: "redis" } } }
    → enabled = true (production default)
    → Uses Redis

t2: Project Beta config: { cache: { bundleManifest: { type: "redis", enabled: false } } }
    → enabled = false (explicit)
    → Uses InMemory
    → Project Beta developer confused: "I configured Redis, why isn't it used?"

t3: Project Gamma config: { cache: { render: { type: "redis" } } }
    → No enabled flag exists for render cache
    → How does Gamma disable render caching? No way to do so!
```

## Solution

See **[007.0 - Config Normalization RFC](./007.0-config-normalization-rfc.md)** for the complete solution.

Normalize to consistent structure with explicit enabled flags everywhere:

```typescript
interface NormalizedCacheConfig {
  readonly dir: string;
  readonly bundleManifest: {
    readonly enabled: boolean;  // ALWAYS present
    readonly type: "redis" | "kv" | "memory";  // ALWAYS present
    readonly redisUrl: string | null;
    readonly keyPrefix: string;
    readonly ttl: number;
  };
  readonly render: {
    readonly enabled: boolean;  // NOW has enabled flag
    readonly type: "memory" | "filesystem" | "kv" | "redis";
    readonly ttl: number;
    readonly maxEntries: number;
  };
}

function normalizeCacheConfig(cache?: CacheInput, mode?: string): NormalizedCacheConfig {
  return {
    dir: cache?.dir ?? ".veryfront",
    bundleManifest: {
      enabled: cache?.bundleManifest?.enabled ?? (mode === "production"),
      type: cache?.bundleManifest?.type ?? "memory",
      redisUrl: cache?.bundleManifest?.redisUrl ?? null,
      keyPrefix: cache?.bundleManifest?.keyPrefix ?? "veryfront:bundle:",
      ttl: cache?.bundleManifest?.ttl ?? (mode === "production" ? 86400000 : 0),
    },
    render: {
      enabled: cache?.render?.type !== undefined,  // Infer from type presence
      type: cache?.render?.type ?? "memory",
      ttl: cache?.render?.ttl ?? 0,
      maxEntries: cache?.render?.maxEntries ?? 500,
    },
  };
}
```

Clear rules:
- `enabled: true, type: "redis"` = Use Redis
- `enabled: false, type: "redis"` = Disabled (type ignored)
- `enabled: true, type: undefined` = Use default type (memory)
- No explicit `enabled` = Infer from context/mode with clear defaults
