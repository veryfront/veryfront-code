# 006.3 - Module Loading Conditionals

## Severity: HIGH

## Summary

Runtime-specific module loading strategies are scattered across business logic with 28+ conditional imports. This creates:
1. Different React instance risks (SSR vs client)
2. Inconsistent esm.sh URL patterns
3. Cache key mismatches between environments

## The Problem

### Pattern 1: AWS SDK Loading (s3-storage.ts)

```typescript
// src/workflow/blob/s3-storage.ts:17-30
import { isDeno } from "#veryfront/platform/compat/runtime.ts";

type S3Module = typeof import("@aws-sdk/client-s3");
let s3Module: S3Module | null = null;

async function getS3Module(): Promise<S3Module> {
  if (s3Module) return s3Module;

  try {
    s3Module = isDeno
      ? await import("https://esm.sh/@aws-sdk/client-s3@3.490.0")
      : await import("@aws-sdk/client-s3");
    return s3Module;
  } catch (error) {
    throw new Error(`Failed to load @aws-sdk/client-s3: ${error}`);
  }
}
```

**Problems:**
1. Hardcoded esm.sh version (`@3.490.0`) - gets stale
2. Business logic (S3 storage) knows about module resolution
3. No capability check - what if Deno is behind a firewall blocking esm.sh?

### Pattern 2: Redis Client Loading (redis/modules.ts)

```typescript
// src/platform/adapters/redis/modules.ts:14-27
export async function loadRedisModules(): Promise<RedisModules> {
  if (isDeno) {
    // Split URL to avoid bundler/linter detection
    const denoRedisUrl = ["https://deno.land/x/redis", "@v0.32.1/mod.ts"].join("");
    DenoRedis = await import(denoRedisUrl);
  } else {
    // Split module name to avoid bundler hoisting
    const redisModuleName = ["re", "dis"].join("");
    NodeRedis = await import(redisModuleName);
  }
  return { DenoRedis, NodeRedis };
}
```

**Problems:**
1. URL obfuscation (`join("")`) is a code smell - indicates bundler conflicts
2. Different Redis clients for different runtimes = different APIs
3. No fallback if deno.land is down

### Pattern 3: React Import Resolution (ssr-import-rewriter.ts)

```typescript
// src/modules/server/ssr-import-rewriter.ts:44-75
function resolveReactForRuntime(specifier: string, version?: string): string | null {
  // For Bun: Use local React paths from veryfront's node_modules.
  if (!isDeno && !isNode) {
    const localPaths = getLocalReactPaths();
    const localPath = localPaths[specifier];
    if (localPath) return localPath;
  }

  // For Deno: Use esm.sh URLs (Deno supports HTTP imports natively).
  // For Node.js: Use esm.sh URLs which will be cached to disk.
  const v = version ?? getReactVersion();
  const reactMap = getReactImportMap(v);

  const mapped = reactMap[specifier];
  if (mapped) return mapped;

  // Handle React subpath imports not in the map
  if (specifier.startsWith("react/")) {
    const subpath = specifier.slice("react/".length);
    return `https://esm.sh/react@${v}/${subpath}?external=react&target=es2022`;
  }
  // ...
}
```

**Problems:**
1. Three different React resolution strategies (Bun/Deno/Node)
2. Bun gets local paths, others get esm.sh - version mismatch risk
3. `external=react` query param for deduplication is fragile

### Pattern 4: HTTP Cache Skip Logic (http-cache.ts)

```typescript
// src/transforms/esm/http-cache.ts:211-220
// For Deno: Skip React core modules (prevents multiple instances).
// For Node.js: Must cache React to disk because Node.js can't import HTTP URLs.
if (isDeno && isReactCoreUrl(normalizedUrl)) {

```

**Problem:** The caching strategy differs by runtime, creating different module graphs.

## Why It's Bad

### Multi-React Instance Risk

```
Scenario: Project uses both a UI library and custom code that import React

Deno path:
1. UI library imports https://esm.sh/react@18.2.0?external=react
2. Custom code imports https://esm.sh/react@18.2.0
3. Without ?external=react, esm.sh bundles React into the package
4. Now TWO React instances exist
5. Hooks fail with "Invalid hook call" error

Node.js path:
1. UI library imports cached /tmp/http-cache/react-xxx.mjs
2. Custom code imports cached /tmp/http-cache/react-xxx.mjs
3. Same file = same instance
4. Works correctly

Result: "Works on Node, breaks on Deno" bugs
```

### Cache Key Mismatch

```typescript
// Deno: Uses HTTP URL directly (no local cache for React)
// Transforms cache key: "project:page:/app/page.tsx:ssr"
// Module reference: "https://esm.sh/react@18.2.0"

// Node: Caches to disk
// Transforms cache key: "project:page:/app/page.tsx:ssr"
// Module reference: "file:///tmp/http-cache/react-abc123.mjs"

// Problem: Same transform cache key, different module references
// If distributed cache is shared across Deno/Node pods:
// - Deno pod writes transform with esm.sh URLs
// - Node pod reads it, can't import HTTP URLs
// - 500 error!
```

## Failure Scenario

**Timeline for Multi-React Bug on Deno:**

```
1. User installs @shadcn/ui (UI library)
2. @shadcn/ui package.json has: "react": "^18.0.0" (peer dep)
3. esm.sh builds @shadcn/ui with external=react query param

4. User code:
   import { Button } from "@shadcn/ui"  // Uses external react
   import { useState } from "react"      // Direct import

5. Transform pipeline resolves:
   - "@shadcn/ui" → "https://esm.sh/@shadcn/ui?external=react"
   - "react" → "https://esm.sh/react@18.2.0" (no external!)

6. Two React bundles load:
   - shadcn expects external React
   - Direct import loads bundled React

7. useState() called → "Invalid hook call" error
8. User sees cryptic error, thinks it's their code
```

## Solution

### Centralized Module Loader

```typescript
// src/platform/adapters/modules/loader.ts
export interface ModuleLoaderAdapter {
  /**
   * Load an optional dependency by name.
   * Handles runtime-specific resolution internally.
   */
  loadOptionalDep<T>(
    name: "aws-s3" | "redis" | "esbuild",
  ): Promise<T | null>;

  /**
   * Resolve a package to its importable specifier.
   * Ensures consistent React deduplication.
   */
  resolvePackage(
    name: string,
    options?: { version?: string; external?: string[] },
  ): string;

  /** Runtime supports native HTTP imports */
  readonly supportsHttpImports: boolean;

  /** Runtime supports npm: protocol */
  readonly supportsNpmProtocol: boolean;
}

// Deno implementation
export const denoModuleLoader: ModuleLoaderAdapter = {
  supportsHttpImports: true,
  supportsNpmProtocol: true,

  async loadOptionalDep<T>(name: string): Promise<T | null> {
    const urls: Record<string, string> = {
      "aws-s3": "https://esm.sh/@aws-sdk/client-s3@3",  // Major version only
      "redis": "https://deno.land/x/redis@v0.32.1/mod.ts",
      "esbuild": "https://deno.land/x/esbuild@v0.20.0/mod.js",
    };
    const url = urls[name];
    if (!url) return null;
    try {
      return await import(url) as T;
    } catch {
      return null;
    }
  },

  resolvePackage(name: string, options?: { version?: string; external?: string[] }): string {
    const version = options?.version ?? "latest";
    const external = options?.external?.join(",") ?? "";
    const externalParam = external ? `&external=${external}` : "";
    return `https://esm.sh/${name}@${version}?target=es2022${externalParam}`;
  },
};

// Node implementation
export const nodeModuleLoader: ModuleLoaderAdapter = {
  supportsHttpImports: false,
  supportsNpmProtocol: false,

  async loadOptionalDep<T>(name: string): Promise<T | null> {
    const packages: Record<string, string> = {
      "aws-s3": "@aws-sdk/client-s3",
      "redis": "redis",
      "esbuild": "esbuild",
    };
    const pkg = packages[name];
    if (!pkg) return null;
    try {
      return await import(pkg) as T;
    } catch {
      return null;
    }
  },

  resolvePackage(name: string): string {
    return name;  // Node uses bare specifiers
  },
};
```

## Multi-Tenancy Considerations

The module loading pattern is generally safe for multi-tenancy because:
1. Module imports are stateless (no shared mutable state)
2. Each request gets fresh module resolution

However, the **module cache** (`s3Module`, `DenoRedis`, etc.) is a singleton:

```typescript
// s3-storage.ts - Module-level singleton
let s3Module: S3Module | null = null;
```

**Risk**: If one tenant's request initializes the module with specific config, all subsequent tenants use that config.

**Mitigation**: The adapter pattern should return fresh instances or use project-scoped caching:

```typescript
// Better: Factory function per request
async function getS3Client(config: S3Config): Promise<S3Client> {
  const module = await adapter.modules.loadOptionalDep<S3Module>("aws-s3");
  return new module.S3Client(config);  // Fresh instance per call
}
```
