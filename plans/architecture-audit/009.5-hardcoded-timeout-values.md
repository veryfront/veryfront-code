# 009.5 - Hardcoded Timeout Values

## Severity: MEDIUM

## Location

- `/Users/mattboon/Sites/veryfront-renderer/src/transforms/esm/http-cache.ts` (line 319)
- `/Users/mattboon/Sites/veryfront-renderer/src/modules/react-loader/ssr-module-loader/loader.ts` (line 324)

- `/Users/mattboon/Sites/veryfront-renderer/src/server/handlers/dev/endpoints.ts` (lines 198, 239)

## The Problem

Many timeout values are hardcoded inline as magic numbers, making them:
1. **Impossible to configure** without code changes
2. **Hard to discover** when auditing timeout behavior
3. **Inconsistent** when the same operation uses different values
4. **Undocumented** - no explanation why that specific value

### HTTP Cache - Hardcoded 30s

```typescript
// transforms/esm/http-cache.ts:317-319
const urlObj = new URL(normalizedUrl);
const controller = new AbortController();
const timeout = setTimeout(() => controller.abort(), 30000);  // Why 30s? Not documented
```

### SSR Module Loader - Hardcoded 30s

```typescript
// modules/react-loader/ssr-module-loader/loader.ts:323-324
const controller = new AbortController();
const timeout = setTimeout(() => controller.abort(), 30000);  // Same 30s, but no shared constant
```

### Client Router - Hardcoded 10s

```typescript
// html/hydration-script-builder/templates/router.ts:524-527
// Wait up to 10 seconds for hydration
await Promise.race([
  hydrationPromise,
  new Promise((_, reject) => setTimeout(() => reject(new Error('Hydration timeout')), 10000))
]);
```

This is **client-side code bundled into the HTML**, so it cannot be configured at runtime.

### Dev Server - Hardcoded 2s and 100ms

```typescript
// server/handlers/dev/endpoints.ts:198
reconnectTimeoutId = setTimeout(() => window.location.reload(), 2000);

// server/handlers/dev/endpoints.ts:239
setTimeout(() => window.location.reload(), 100);
```

## Why It's Bad for Multi-Tenancy

### Configuration Inflexibility

Different projects may need different timeout values:
- **Fast project**: 5s module fetch timeout is sufficient
- **Heavy project**: Needs 60s for complex module graphs

With hardcoded values, all projects get the same timeout, which is either:
- Too short for heavy projects (errors)
- Too long for fast projects (slow failure detection)

### Inconsistent Behavior

The same logical operation has different timeouts in different places:

| Operation | Location | Value |
|-----------|----------|-------|
| HTTP module fetch | `http-cache.ts:319` | 30s |
| Cross-project fetch | `loader.ts:324` | 30s |
| Module server fetch | `module-server.ts:572` | None! |

### Maintenance Burden

When adjusting timeouts:
1. Must search entire codebase for magic numbers
2. Easy to miss some occurrences
3. No single place to audit all timeout values

## Code Evidence: Mixed Patterns

The codebase has a mix of patterns:

### Good: Centralized Constants

```typescript
// config/defaults.ts
export const DATA_FETCH_TIMEOUT_MS = 10000;

// Used consistently
import { DATA_FETCH_TIMEOUT_MS } from "#veryfront/config/defaults.ts";
await withTimeoutThrow(promise, DATA_FETCH_TIMEOUT_MS, label);
```

### Bad: Inline Magic Numbers

```typescript
// transforms/esm/http-cache.ts:319
const timeout = setTimeout(() => controller.abort(), 30000);

// html/hydration-script-builder/templates/router.ts:526
setTimeout(() => reject(new Error('Hydration timeout')), 10000)
```

### Worse: Comment Hardcodes Same Value

```typescript
// server/handlers/dev/dashboard/api.ts (hypothetical from documentation)
const timeoutMs = 30000; // 30 seconds timeout for dev testing
// ...
setTimeout(() => reject(new Error("Workflow execution timed out (30s)")), timeoutMs)
```

If `timeoutMs` changes, the error message becomes incorrect.

## Client-Side Special Case

Client-side timeouts (`router.ts`) are bundled into HTML at build time:

```typescript
// html/hydration-script-builder/templates/router.ts
const FETCH_TIMEOUT_MS = 10000;  // Hardcoded in template
const CACHE_TTL_MS = 5 * 60 * 1000;  // 5 minutes
```

These cannot be configured via environment variables because they run in the browser.

### Options for Client Timeouts:

1. **Build-time injection**: Template variables replaced during build
2. **Runtime injection**: `window.__VF_CONFIG__.hydrationTimeoutMs`
3. **Accept as unchangeable**: Document as fixed values

## Solution

See **[009.0 - Timeout Handling RFC](./009.0-timeout-handling-rfc.md)** Section 2 for the centralized constants solution.

### Quick Fix: Extract to Constants

```typescript
// src/utils/constants/timeouts.ts

// HTTP/Module operations
export const HTTP_FETCH_TIMEOUT_MS = ms(30_000);
export const CROSS_PROJECT_FETCH_TIMEOUT_MS = ms(30_000);

// Client-side (documented, not configurable)
export const CLIENT_HYDRATION_TIMEOUT_MS = ms(10_000);
export const CLIENT_FETCH_TIMEOUT_MS = ms(10_000);

// Dev server
export const DEV_RECONNECT_TIMEOUT_MS = ms(2_000);
export const DEV_RELOAD_DELAY_MS = ms(100);
```

### Refactor http-cache.ts

```typescript
// Before
const timeout = setTimeout(() => controller.abort(), 30000);

// After
import { HTTP_FETCH_TIMEOUT_MS } from "#veryfront/utils/constants/timeouts.ts";
const timeout = setTimeout(() => controller.abort(), HTTP_FETCH_TIMEOUT_MS);
```

### Refactor Client Router

For client-side code, inject configuration:

```typescript
// Build template
const FETCH_TIMEOUT_MS = ${config.client?.fetchTimeoutMs ?? 10000};

// Or use runtime config
const FETCH_TIMEOUT_MS = window.__VF_CONFIG__?.fetchTimeoutMs ?? 10000;
```
