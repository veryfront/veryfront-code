# 003.4 - Cache Hit Skips Critical Validations

## Severity: HIGH - Silent Behavior Divergence

## The Problem

When cache hits occur, multiple validation steps that run on cache misses are skipped entirely. This creates a situation where:

1. **Fresh requests** (cache miss) get full validation
2. **Cached requests** (cache hit) skip validation
3. **Same input, different behavior** depending on cache state

This violates the fundamental principle: **cached responses must be indistinguishable from fresh responses**.

## Validations Skipped on Cache Hit

### 1. Content Hash Verification - SKIPPED

**Location:** `src/transforms/mdx/esm-module-loader/cache/index.ts:215-273`

```typescript
export async function lookupMdxEsmCache(
  filePath: string,
  cacheDir: string,
  projectDir?: string,
  _contentHash?: string,  // INTENTIONALLY UNUSED!
): Promise<string | null> {
  // ...
  const cachedPath = cache.get(cacheKey);
  if (!cachedPath) return null;

  // Note: We intentionally skip contentHash validation for MDX-ESM cached files.
  // The MDX-ESM cache uses transformed-code hashes in filenames (vfmod-v{VERSION}-{hash}.mjs),
  // while the SSR loader provides source-code hashes. These will never match.
  // ...

  return cachedPath;  // Returns without verifying content hasn't changed!
}
```

**Problem:** The `_contentHash` parameter is accepted but ignored. If source file content changes but cache key components don't (version + path), stale cached content is returned.

**Why it exists:** The comment explains the hash mismatch between source hash and transform hash. But this is a **design flaw**, not a valid reason to skip validation.

### 2. Import Resolution Validation - SKIPPED

## Cache HIT vs MISS Validation Matrix

| Validation | Cache HIT | Cache MISS |
|------------|-----------|------------|
| Content hash match | SKIPPED | N/A (computed) |
| Import path resolution | SKIPPED | Full resolution |
| Dependency existence | Partial | Full |
| Dependency freshness | SKIPPED | Current |
| Syntax validation | SKIPPED | esbuild validates |
| React version check | Inconsistent | Configured |
| Env config check | SKIPPED | Current |
| Cross-project availability | SKIPPED | Fetched |

## Real-World Failure Scenarios

### Scenario 1: Source File Edit Not Reflected

```
1. User edits lib/utils.ts, changes function signature
2. pages/index.tsx imports lib/utils.ts
3. pages/index.tsx transform is cached (contentHash of pages/index.tsx unchanged)
4. Cache hit returns transform with OLD import path
5. Import path points to OLD utils.ts content (if content hash in path)
   OR points to NEW utils.ts but with incompatible call sites
6. Result: Runtime error "TypeError: utils.foo is not a function"
```

### Scenario 2: Config Change Ignored

```
1. User changes veryfront.config.ts: { jsxImportSource: "@emotion/react" }
2. Transform cache has existing entry for pages/index.tsx
3. Cache key doesn't include JSX config
4. Cache hit returns transform with OLD JSX runtime
5. Result: Styles not applied, or "jsx is not a function"
```

### Scenario 3: Corrupted Cache Entry

```
1. Redis write partially fails, truncates cached transform
2. Cache hit returns truncated JavaScript
3. No syntax validation on hit
4. Runtime import: SyntaxError: Unexpected end of input
5. User sees 500 error instead of helpful transform error
```

## The Fix

### Option 1: Validation Hooks on Cache Hit

```typescript
interface CacheEntry<T> {
  value: T;
  validators: Array<{
    type: string;
    checksum: string;
  }>;
}

async function getCachedWithValidation<T>(
  key: string,
  validators: Array<() => Promise<string>>,
): Promise<T | null> {
  const entry = await cache.get(key);
  if (!entry) return null;

  // Re-run validators, compare checksums
  for (const [i, validator] of validators.entries()) {
    const currentChecksum = await validator();
    if (currentChecksum !== entry.validators[i].checksum) {
      // Validator failed - invalidate
      await cache.delete(key);
      return null;
    }
  }

  return entry.value;
}
```

### Option 2: Content-Addressed Cache

```typescript
// Key includes hash of ALL inputs that affect output
const cacheKey = buildCacheKey({
  sourceHash: hashContent(sourceCode),
  configHash: hashContent(JSON.stringify(config)),
  depsHash: hashContent(JSON.stringify(dependencyVersions)),
  envHash: hashContent(JSON.stringify(relevantEnvVars)),
});

// Any change to any input = different key = cache miss
```

### Option 3: Mandatory Freshness Check

```typescript
// Every cache hit runs lightweight freshness check
async function getWithFreshnessCheck<T>(
  key: string,
  freshnessCheck: () => Promise<boolean>,
): Promise<T | null> {
  const entry = await cache.get(key);
  if (!entry) return null;

  // Always check freshness
  if (!await freshnessCheck()) {
    await cache.delete(key);
    return null;
  }

  return entry;
}
```

## Solution

See **[003.0 - Cache Consistency RFC](./003.0-cache-consistency-rfc.md)** for the complete solution.

The key insight is that cache hits MUST run the same validations as cache misses. The difference should be in WHERE data comes from (cache vs source), not WHETHER it's validated.
