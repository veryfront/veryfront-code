# 001.5 - Config/Middleware Loading Divergence

## Severity: 🟠 HIGH - Inconsistent Loading Paths

## The Problem

Config and middleware loading use **completely different code paths** based on adapter type:

```
Virtual Filesystem:
  Content → esbuild transpile → temp file → dynamic import

Local Filesystem:
  file:// URL → native Deno import (bypass esbuild)
```

## Config Loading (`src/config/loader.ts`)

### Lines 278-285 - The Branch

```typescript
if (isVirtualFilesystem(adapter)) {
  return loadConfigFromVirtualFS(configPath, cacheKey, adapter);
}

// Local filesystem: direct import
const configModule = await import(`file://${configPath}?t=${Date.now()}`);
```

### Virtual FS Path (Lines 214-275)

```typescript
function loadConfigFromVirtualFS(
  configPath: string,
  cacheKey: string,
  adapter: RuntimeAdapter,
): Promise<VeryfrontConfig> {
  // 1. Read file content via adapter
  const content = await adapter.fs.readFile(configPath);

  // 2. Transpile with esbuild
  const { build } = await import("esbuild");
  const transpileResult = await build({
    bundle: false,
    write: false,
    format: "esm",
    platform: "neutral",
    target: "es2022",
    stdin: {
      contents: source,
      loader,  // 'ts' or 'js'
      resolveDir: dirname(configPath),
      sourcefile: configPath,
    },
  });

  // 3. Write to temp file
  const tempDir = await fs.makeTempDir({ prefix: "vf-config-" });
  const tempFile = join(tempDir, "config.mjs");
  await fs.writeTextFile(tempFile, js);

  // 4. Dynamic import temp file
  const { default: userConfig } = await import(`file://${tempFile}?t=${Date.now()}`);

  // 5. Clean up temp file
  await fs.rm(tempDir, { recursive: true });
}
```

### Local FS Path (Lines 283-284)

```typescript
// Direct native import - no esbuild, no temp files

```

### The Same Pattern

Virtual FS goes through esbuild transpilation, local FS uses native import.

## Why This Matters

### Problem 1: Different Transform Behavior

esbuild transpilation and Deno's native TypeScript handling are NOT identical:

```typescript
// This TypeScript syntax...
const x = { a: 1 } satisfies Record<string, number>;

// esbuild (platform: neutral, target: es2022):
const x = { a: 1 };

// Deno native import:
const x = { a: 1 };  // But handles it differently internally
```

**Subtle differences in:**
- Decorator handling
- Import resolution
- Type erasure timing
- Source map generation

### Problem 2: Error Messages Differ

```typescript
// Virtual FS error:
ConfigValidationError: Failed to transpile config: Unexpected token at line 5

// Local FS error:
SyntaxError: Unexpected identifier at file:///path/to/veryfront.config.ts:5:1
```

**Debugging production issues when local dev works differently.**

### Problem 3: Import Resolution Differs

```typescript
// In veryfront.config.ts:
import { something } from "./utils/config-helpers.ts";
```

**Virtual FS (esbuild):**
- `resolveDir: dirname(configPath)` - uses the original file's directory
- But the config runs from a temp file in `/tmp/vf-config-xxx/`
- **Relative imports may break** if not bundled

**Local FS (native):**
- Imports resolve from the actual file location
- Works as expected

### Problem 4: No Import Bundling for Virtual FS

```typescript
const transpileResult = await build({
  bundle: false,  // 🔴 NOT BUNDLED
  // ...
});
```

Virtual FS configs **cannot import local modules**:

```typescript
// veryfront.config.ts
import { sharedSettings } from "./config-shared.ts";  // 🔴 FAILS

export default {
  ...sharedSettings,
};
```

The import isn't bundled, and the temp file can't resolve `./config-shared.ts`.

### Problem 5: Bun-Specific Workaround

Check lines 285-311 in loader.ts:

```typescript
// Note: Bun crashes on certain valid TypeScript constructs
// Fall back to virtual FS loading to use esbuild
if (isBun()) {
  try {
    return await loadConfigFromVirtualFS(configPath, cacheKey, adapter);
  } catch {
    // Continue with native import
  }
}
```

**Bun needs special handling**, adding another code path.

### Problem 6: Cache Key Differences

```typescript
// Virtual FS cache key (line 282):
const cacheKey = buildConfigCacheKey(projectIdOrDir, true);  // 🟢 projectId

// Local FS cache key (line 278 implied):
const cacheKey = buildConfigCacheKey(projectDir, false);     // 🟢 projectDir
```

From `src/cache/keys.ts:154-156`:

```typescript
export function buildConfigCacheKey(projectIdOrDir: string, isVirtualFilesystem: boolean): string {
  const baseKey = isVirtualFilesystem
    ? `config:project:${projectIdOrDir}`   // "config:project:proj-123"
    : `config:local:${projectIdOrDir}`;    // "config:local:/path/to/project"
}
```

**Same config, different cache key format.**

## The Real Issue: Duplicated `isVirtualFilesystem`

As documented in **001.3**, these two files define `isVirtualFilesystem()` **differently**:

| File | Includes GitHubFSAdapter |
|------|--------------------------|
| `config/loader.ts` | ✅ YES (via VIRTUAL_FS_ADAPTERS set) |
| `middleware.ts` | ❌ NO (only checks `isVeryfrontAdapter()`) |

**Consequence**: GitHub adapter projects get:
- Config loaded via esbuild (virtual FS path)
- Middleware loaded via native import (local FS path)

**The same project uses different loading strategies for config vs middleware.**

## Multi-Tenancy Impact

| Scenario | Local Dev | Production |
|----------|-----------|------------|
| Config syntax error | Deno error message | esbuild error message |
| Relative imports | Work | May break |
| Decorators | Deno handling | esbuild handling |
| Debug info | Full stack trace | Temp file paths |

**Users report bugs in production that can't be reproduced locally.**

## The Fix

### Option A: Always Use esbuild

```typescript
async function loadConfig(
  configPath: string,
  cacheKey: string,
  adapter: RuntimeAdapter,
): Promise<VeryfrontConfig> {
  // 🟢 SINGLE PATH - always transpile with esbuild
  return await loadConfigWithEsbuild(configPath, cacheKey, adapter);
}
```

Pros: Consistent behavior everywhere
Cons: Slower local dev, esbuild dependency always needed

### Option B: Always Use Native Import

```typescript
async function loadConfig(
  configPath: string,
  cacheKey: string,
  adapter: RuntimeAdapter,
): Promise<VeryfrontConfig> {
  // Write virtual content to temp file first
  if (isVirtualFilesystem(adapter)) {
    const content = await adapter.fs.readFile(configPath);
    const tempPath = await writeTempFile(content, configPath);
    configPath = tempPath;
  }

  // 🟢 SINGLE PATH - always native import
  return await import(`file://${configPath}?t=${Date.now()}`);
}
```

Pros: Consistent Deno behavior
Cons: Requires temp file writing

### Option C: Bundle for Virtual FS

```typescript
const transpileResult = await build({
  bundle: true,  // 🟢 Bundle imports
  // ...
});
```

At least make virtual FS configs work with imports.
