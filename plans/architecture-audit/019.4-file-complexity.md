# 019.4: Large File Complexity

## Severity: LOW

## Locations

Files exceeding 1000 lines of code:

| File | Lines | Primary Concern |
|------|-------|-----------------|
| `src/ai/tools/advanced-tools.ts` | 1,996 | God object, all AI tools |
| `src/rendering/ssr/renderer.ts` | 1,200+ | SSR pipeline monolith |
| `src/build/transforms/esm/bundler.ts` | 1,100+ | Transform pipeline |
| `src/routing/file-router.ts` | 1,050+ | Routing logic |
| `src/module-system/loader.ts` | 1,000+ | Module loading |

## The Problem

Large files with many responsibilities:
- Hard to navigate and understand
- Difficult to test in isolation
- Merge conflicts more likely
- Cognitive load for reviewers
- Encourages adding "one more thing"

### Complexity Indicators

```
advanced-tools.ts (1,996 lines):
├── 50+ exported functions
├── 12 different tool categories
├── Mixed concerns (file, web, AI, system)
└── Single flat structure

renderer.ts (1,200+ lines):
├── Hydration logic
├── Streaming SSR
├── Error boundaries
├── Head management
├── State serialization
└── All interleaved
```

## Proposed Decomposition

### advanced-tools.ts → tool-categories/

```
src/ai/tools/
├── index.ts              # Re-exports all tools
├── file-tools.ts         # File operations
├── web-tools.ts          # Web/fetch operations
├── ai-tools.ts           # AI/LLM operations
├── system-tools.ts       # System operations
├── data-tools.ts         # Data manipulation
└── types.ts              # Shared types
```

### renderer.ts → phases/

```
src/rendering/ssr/
├── index.ts              # Main entry, orchestration
├── phases/
│   ├── hydration.ts      # Client hydration
│   ├── streaming.ts      # Streaming SSR
│   └── serialization.ts  # State serialization
├── components/
│   ├── error-boundary.ts
│   └── head-manager.ts
└── utils/
    └── render-utils.ts
```

### bundler.ts → stages/

```
src/build/transforms/esm/
├── bundler.ts            # Main entry
├── stages/
│   ├── parse.ts          # AST parsing
│   ├── transform.ts      # Code transforms
│   ├── resolve.ts        # Import resolution
│   └── emit.ts           # Output generation
└── visitors/
    ├── import-visitor.ts
    └── export-visitor.ts
```

## Decision Required

**D015**: Large file decomposition strategy:
- A) Decompose all at once (big bang)
- B) Decompose incrementally (one file per sprint)
- C) Only decompose when touching the file
- D) Leave as-is, add better navigation aids

## Migration Approach

### Incremental (Recommended)

1. **Phase 1**: Extract types to separate files
2. **Phase 2**: Extract utility functions
3. **Phase 3**: Split by logical concern
4. **Phase 4**: Update imports across codebase

### Preserve Exports

```typescript
// src/ai/tools/index.ts
// Re-export everything for backward compatibility
export * from "./file-tools";
export * from "./web-tools";
export * from "./ai-tools";
// etc.
```

## Blast Radius

| Risk | Impact |
|------|--------|
| Import changes | Many files may import from these |
| Bundle splitting | May affect code splitting |
| Test organization | Tests may need restructuring |

## Migration Checklist

- [ ] Decide on decomposition strategy (D015)
- [ ] Start with lowest-coupling file
- [ ] Extract types first (safest)
- [ ] Maintain backward-compatible exports
- [ ] Update tests to match new structure
