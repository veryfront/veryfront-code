## The Problem

### Mode Detection Uses Multiple Inconsistent Signals

```typescript
// src/utils/constants/cache.ts:31-35
function isProductionMode(): boolean {
  return getEnvString("PROXY_MODE") === "1" ||
    getEnvString("NODE_ENV") === "production" ||
    getEnvString("PRODUCTION_MODE") === "1";
}
```

This function determines whether to use production or preview cache TTLs, but:
1. It only checks environment variables, not request context
2. It doesn't consider per-request `x-environment` header
3. A single pod serves both preview and production requests

### Massive TTL Differences


### getDistributedCacheTTL Uses Static Check

```typescript
// src/utils/constants/cache.ts:109-127
export function getDistributedCacheTTL(
  cacheType: "ssr-module" | "transform" | "file" | "css",
  isProduction: boolean = isProductionMode(),  // <-- Static default!
): number {
  switch (cacheType) {
    case "ssr-module":
      return isProduction
        ? DISTRIBUTED_SSR_MODULE_TTL_PRODUCTION_SEC
        : DISTRIBUTED_SSR_MODULE_TTL_PREVIEW_SEC;
    // ... etc
  }
}
```

**Problem**: Callers that don't pass `isProduction` will get the static check, not the per-request environment.

## Why It Matters

### Scenario 1: Production Request Gets Preview TTL

If a production request is misclassified as preview:
- 6 hour cache becomes 10 minute cache
- Cache efficiency drops 36x
- API receives 36x more requests
- Latency increases due to cache misses

### Scenario 2: Preview Request Gets Production TTL

If a preview request is misclassified as production:
- 10 minute cache becomes 6 hour cache
- Users see stale content for hours after changes
- Draft changes don't appear
- Developers think feature is broken

### Multi-Tenant Risk

In a multi-tenant pod:
- Pod env vars are static
- Requests for different projects/environments share the pod
- Cache TTL is determined at pod level, not request level

## Concrete Fix

### Option A: Pass Environment to TTL Functions (Recommended)

```typescript
// src/utils/constants/cache.ts

/**
 * Get cache TTL based on request environment, not pod environment.
 *
 * @param cacheType - Type of cache
 * @param environment - Request environment ("preview" or "production")
 */
export function getDistributedCacheTTLForRequest(
  cacheType: "ssr-module" | "transform" | "file" | "css",
  environment: "preview" | "production",
): number {
  const isProduction = environment === "production";
  return getDistributedCacheTTL(cacheType, isProduction);
}

// Update all call sites to use the new function with explicit environment
```

### Option B: Use ModeResolver Context

```typescript
// src/utils/constants/cache.ts
import { getResolvedMode } from "#veryfront/core/mode/resolver";

export function getDistributedCacheTTLFromContext(
  cacheType: "ssr-module" | "transform" | "file" | "css",
): number {
  const mode = getResolvedMode();

  // Fall back to static check if outside request context
  if (!mode) {
    return getDistributedCacheTTL(cacheType, isProductionMode());
  }

  return getDistributedCacheTTL(cacheType, mode.environment === "production");
}
```


1. **Keep cache key isolation** (already correct)
2. **Update TTL selection to be per-request**
3. **Log when TTL differs from expectation**
