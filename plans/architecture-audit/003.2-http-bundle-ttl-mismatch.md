# 003.2 - HTTP Bundle TTL vs Transform TTL Mismatch

## Severity: HIGH - Intermittent Module Load Failures

## The Problem

HTTP bundles (esm.sh packages) have a **24-hour** TTL in distributed cache, while SSR transforms that reference them have only a **6-hour** TTL. This creates a window where:

1. Transform expires from Redis
2. Pod requests transform, gets cache MISS, re-transforms from source
3. New transform references HTTP bundles by their `file://` paths
4. HTTP bundles expired from Redis AND evicted from local filesystem
5. `ensureHttpBundlesExist` tries to recover from Redis but bundles are gone
6. Result: Unrecoverable "Module not found" for HTTP imports

### Location: `src/utils/constants/cache.ts:72-164`

```typescript
// SSR Module Transform TTL (6 hours production)
export const DISTRIBUTED_SSR_MODULE_TTL_PRODUCTION_SEC = getEnvNumber(
  "DISTRIBUTED_SSR_MODULE_TTL_SEC",
  6 * MINUTES_PER_HOUR * SECONDS_PER_MINUTE,  // 21600 seconds = 6 hours
);

// HTTP Bundle TTL (24 hours)
export const HTTP_MODULE_DISTRIBUTED_TTL_SEC = getEnvNumber(
  "HTTP_MODULE_DISTRIBUTED_TTL_SEC",
  HOURS_PER_DAY * MINUTES_PER_HOUR * SECONDS_PER_MINUTE, // 86400 seconds = 24 hours
);
```

**Wait, HTTP bundles have LONGER TTL than transforms - that should be safe, right?**

**Wrong.** The problem is the **direction of dependency**:
- Transforms DEPEND ON HTTP bundles
- Transforms should have SHORTER TTL than their dependencies
- But the real issue is the **refresh mechanism** and **local filesystem eviction**

## The Real Bug: TTL Refresh Race Condition

### Location: `src/transforms/esm/http-cache.ts:234-262`

```typescript
// Local filesystem cache hit with TTL refresh
if (await exists(cachePath)) {
  cachedPaths.set(cacheKey, cachePath);

  // Refresh distributed cache TTL so bundles outlive transforms that reference them.
  // Without this, bundles expire (24h) while SSR transforms (6h) are still valid.
  const distributed = await getDistributedCache();
  if (distributed) {
    const hash = simpleHash(normalizedUrl);
    const hashStr = String(hash);
    const now = Date.now();
    const lastRefresh = lastDistributedRefresh.get(hashStr);
    const needsRefresh = !lastRefresh || (now - lastRefresh > DISTRIBUTED_REFRESH_INTERVAL_MS);

    if (needsRefresh) {
      try {
        const code = await fs.readTextFile(cachePath);
        await Promise.all([
          distributed.set(distributedKey("url", hash), code, HTTP_MODULE_DISTRIBUTED_TTL_SEC),
          distributed.set(distributedKey("code", hash), code, HTTP_MODULE_DISTRIBUTED_TTL_SEC),
          distributed.set(distributedKey("hash", hash), normalizedUrl, HTTP_MODULE_DISTRIBUTED_TTL_SEC),
        ]);
        lastDistributedRefresh.set(hashStr, now);
      } catch (error) {
        // SILENT FAILURE - refresh fails, bundle will expire
        logger.debug("[HTTP-CACHE] Distributed cache refresh failed", { hash, error });
      }
    }
  }
  return cachePath;
}
```

**Bug:** The TTL refresh only happens on **local filesystem hit**. If the local file is evicted (memory pressure, pod restart, LRU), no refresh occurs, and the bundle expires from Redis.

## Failure Timeline

```
T+0h:   Transform A created, depends on HTTP bundle B
        Transform A stored in Redis (TTL: 6h, expires T+6h)
        HTTP bundle B stored in Redis (TTL: 24h, expires T+24h)

T+5h:   Pod 1 restarts, loses local HTTP bundle cache
        Transform A still valid in Redis

T+6h:   Transform A expires from Redis
        Pod 2 requests page, transform cache MISS

T+7h:   Pod 2 re-transforms from source
        Transform references HTTP bundle B by path
        HTTP bundle B still in Redis (expires T+24h) - OK this time

T+23h:  Pod 2 restarts, loses local HTTP bundle cache
        No local access to bundle B = no TTL refresh

T+24h:  HTTP bundle B expires from Redis
        Transform references stale bundle path
        ensureHttpBundlesExist() tries Redis: MISS
        ensureHttpBundlesExist() tries network: maybe works, maybe esm.sh changed

T+25h:  esm.sh returns different bundle content (version changed)
        Or network fails (rate limit, timeout)
        Result: "Module not found" or version mismatch
```

## Code Evidence

### 1. ensureHttpBundlesExist Recovery Path

```typescript
// http-cache.ts:557-620
export async function ensureHttpBundlesExist(
  bundlePaths: string[],
  cacheDir: string,
): Promise<string[]> {
  const failed: string[] = [];

  for (const bundlePath of bundlePaths) {
    // Check local filesystem first
    if (await exists(bundlePath)) continue;

    // Try to recover from distributed cache
    const hash = extractHashFromPath(bundlePath);
    const distributed = await getDistributedCache();
    if (distributed && hash) {
      const cachedCode = await distributed.get(distributedKey("code", hash));
      if (cachedCode) {
        // Recovery successful - write to local filesystem
        await writeFile(bundlePath, cachedCode);
        continue;
      }
    }

    // FALLBACK: Re-fetch from network
    // BUG: This may return DIFFERENT content than originally cached!
    try {
      const url = await distributed?.get(distributedKey("hash", hash));
      if (url) {
        const response = await fetch(url);
        const code = await response.text();
        await writeFile(bundlePath, code);
        continue;
      }
    } catch {
      // Network fetch failed
    }

    // Unrecoverable
    failed.push(bundlePath);
  }

  return failed;
}
```

**Bug:** The network re-fetch fallback can return **different content** than the original bundle. esm.sh may have updated the package, causing version mismatches with other bundles.

### 2. Memory Cache Doesn't Track Bundle Dependencies

```typescript
// loader.ts:450-491 - Memory cache hit
const cachedEntry = globalModuleCache.get(contentCacheKey);
if (cachedEntry) {
  // Verify HTTP bundles exist for in-memory cached transforms (once per path+content)
  const verifyKey = `${cachedEntry.tempPath}:${cachedEntry.contentHash}`;
  if (!verifiedHttpBundlePaths.get(verifyKey)) {
    // ...verification logic
  }
}
```

**Bug:** `verifiedHttpBundlePaths` is a per-pod memory Map. If bundle verification passes once, it's never re-checked. Bundles can be evicted from local filesystem AFTER verification, causing runtime failures.

## Cache HIT vs MISS Divergence

| Aspect | Cache HIT | Cache MISS |
|--------|-----------|------------|
| HTTP bundle freshness | Assumed valid | Fetched fresh |
| Bundle version consistency | May be stale | Guaranteed current |
| Recovery from missing bundles | Tries Redis, then network (risky) | N/A - creates bundles |
| TTL coordination | None | Bundles created with transform |

## The Fix

### Option 1: Align TTLs (Quick Fix)

```typescript
// Make HTTP bundle TTL >= SSR transform TTL + safety margin
// Currently: Bundles 24h, Transforms 6h
// Safe as-is? NO - because of refresh race condition

// Better: Bundles infinite, Transforms 6h
// Bundles only expire on version change (content hash in key)
export const HTTP_MODULE_DISTRIBUTED_TTL_SEC = getEnvNumber(
  "HTTP_MODULE_DISTRIBUTED_TTL_SEC",
  7 * HOURS_PER_DAY * MINUTES_PER_HOUR * SECONDS_PER_MINUTE, // 7 days
);
```

### Option 2: Content-Hash Keyed Bundles (Correct Fix)

```typescript
// Store bundles by content hash, not URL hash
// Bundles with same content share the same key
const contentHash = hashCode(bundleCode);
await distributed.set(`bundle:${contentHash}`, bundleCode, VERY_LONG_TTL);

// Transform stores content hash reference
// Transform: import "file://${cacheDir}/http-${contentHash}.mjs"

// On cache hit, lookup by content hash (immutable)
const bundleCode = await distributed.get(`bundle:${contentHash}`);
```

### Option 3: Dependency Graph Tracking (Complete Fix)

```typescript
interface TransformMetadata {
  code: string;
  httpBundleDeps: Array<{
    url: string;
    contentHash: string;
  }>;
  localImportDeps: string[];
}

// Store transform with dependency metadata
// On cache hit, validate ALL dependencies exist
// If any missing, invalidate entire transform
```

## Solution

See **[003.0 - Cache Consistency RFC](./003.0-cache-consistency-rfc.md)** for the complete solution.

The key insight is that cache entries should track their dependencies, and dependent entries should have shorter TTLs than their dependencies (or infinite TTLs with content-hash keys for immutable data).
