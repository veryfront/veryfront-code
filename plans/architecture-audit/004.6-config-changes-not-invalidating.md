# 004.6 - Config Changes Not Invalidating Transform Caches

## Severity: MEDIUM - Stale Behavior After Config Updates

## The Problem

Transform cache keys do not include configuration values that affect transform output. When configuration changes (e.g., React version, import maps, build settings), cached transforms continue to use the old configuration.

### Location: `/Users/mattboon/Sites/veryfront-renderer/src/cache/keys.ts:260-269`

```typescript
export function buildTransformCacheKey(
  filePath: string,
  contentHash: string,
  ssr: boolean = false,
  studioEmbed: boolean = false,
): string {
  const ssrKey = ssr ? "ssr" : "browser";
  const studioKey = studioEmbed ? ":studio" : "";
  return `v${TRANSFORM_CACHE_VERSION}:${filePath}:${contentHash}:${ssrKey}${studioKey}`;
  // NO config hash! No React version! No import map!
}
```

### Location: `/Users/mattboon/Sites/veryfront-renderer/src/transforms/pipeline/context.ts`

The transform context uses config values, but those values are not included in the cache key.

## Configuration Values That Affect Transforms

| Config Option | Location | Affects |
|---------------|----------|---------|
| `react.version` | `config.react.version` | React import URLs, jsx-runtime |
| `resolve.importMap` | `config.resolve.importMap` | All import specifier resolution |
| `client.cdn.provider` | `config.client.cdn.provider` | CDN URL generation |
| `client.cdn.versions` | `config.client.cdn.versions` | Package version resolution |
| `build.esbuild` | `config.build.esbuild` | esbuild transform options |
| `experimental.*` | Various | Feature flags affecting transforms |

## Failure Scenario

### Sequence of Events

| Time | Action | Config | Cache Key | Result |
|------|--------|--------|-----------|--------|
| T1 | Request `/page` | React 18.3.1 | `v18:page.tsx:abc:browser` | Miss - transforms with React 18 |
| T2 | Update config | React 19.0.0 | - | Config cache cleared |
| T3 | Request `/page` | React 19.0.0 | `v18:page.tsx:abc:browser` | **HIT** - stale React 18 transform! |

The file content didn't change, so the cache key is the same. The cached transform still uses React 18 imports.

## Impact Assessment

| Config Change | Expected | Actual |
|---------------|----------|--------|
| React version upgrade | All transforms re-run | Only new files get new React |
| Import map change | All imports re-resolve | Old resolutions cached |
| CDN provider change | All CDN URLs update | Mixed CDN URLs |
| esbuild options change | All transforms update | Old transform options used |

## Partial Mitigation: TRANSFORM_CACHE_VERSION

When transform logic changes significantly, `TRANSFORM_CACHE_VERSION` is bumped:

**Location:** `/Users/mattboon/Sites/veryfront-renderer/src/transforms/esm/package-registry.ts:49-71`

```typescript
/**
 * Transform cache version - bump when transform logic changes.
 * This invalidates all cached modules (local + Redis) to prevent stale transform issues.
 *
 * v18: Invalidate transforms with local dev machine paths in file:// URLs
 */
export const TRANSFORM_CACHE_VERSION = 18;
```

But this is a **nuclear option** - it invalidates ALL transforms, not just those affected by the config change.

## What Should Be in Cache Key

A comprehensive cache key should include:

```typescript
function buildTransformCacheKey(
  filePath: string,
  contentHash: string,
  configHash: string,      // NEW: Hash of transform-affecting config
  ssr: boolean = false,
  studioEmbed: boolean = false,
): string {
  const ssrKey = ssr ? "ssr" : "browser";
  const studioKey = studioEmbed ? ":studio" : "";
  return `v${TRANSFORM_CACHE_VERSION}:${filePath}:${contentHash}:${configHash}:${ssrKey}${studioKey}`;
}
```

Where `configHash` is computed from:

```typescript
function computeConfigHash(config: VeryfrontConfig, ssr: boolean): string {
  const transformConfig = {
    reactVersion: config.react?.version,
    importMap: config.resolve?.importMap,
    cdnProvider: config.client?.cdn?.provider,
    cdnVersions: config.client?.cdn?.versions,
    esbuildOptions: config.build?.esbuild,
    experimental: config.experimental,
  };
  return hashObject(transformConfig);
}
```

## Solution

See **[004.0 - Dependency Tracking RFC](./004.0-dependency-tracking-rfc.md)** for the complete solution.

The fix requires:
1. Identify all config values that affect transforms
2. Compute stable hash of those config values
3. Include config hash in all transform cache keys
4. Invalidate on config change via hash mismatch

This is complementary to dependency tracking - the cache key should include:
- File content hash (what we have)
- Dependency hash (see 004.1)
- Config hash (this issue)
