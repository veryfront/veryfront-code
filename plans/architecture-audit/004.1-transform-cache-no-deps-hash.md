# 004.1 - Transform Cache Keys Missing Dependency Hash

## Severity: CRITICAL - Stale Bundles in Production

## The Problem

Transform cache keys are built from **only the main file's content hash**, completely ignoring dependencies. When a dependency file changes but the main file doesn't, stale bundles are served.

### Location: `/Users/mattboon/Sites/veryfront-renderer/src/cache/keys.ts:260-269`

```typescript
export function buildTransformCacheKey(
  filePath: string,
  contentHash: string,       // <-- ONLY the main file's hash
  ssr: boolean = false,
  studioEmbed: boolean = false,
): string {
  const ssrKey = ssr ? "ssr" : "browser";
  const studioKey = studioEmbed ? ":studio" : "";
  return `v${TRANSFORM_CACHE_VERSION}:${filePath}:${contentHash}:${ssrKey}${studioKey}`;
  // NO depsHash component!
}
```

### Location: `/Users/mattboon/Sites/veryfront-renderer/src/transforms/pipeline/index.ts:69-85`

```typescript

  filePath,
```

## Failure Scenario

### Project Structure

```
app/
  components/
    Button.tsx          # Exports Button component
    helper.ts           # Exports formatPrice()
  pages/
    products.tsx        # Imports Button and helper
```

### Sequence of Events

| Time | Action | Cache Key | Result |
|------|--------|-----------|--------|
| T1 | Request `/products` | `v18:products.tsx:abc123:browser` | Miss - transforms, caches |
| T2 | Developer updates `helper.ts` (new formatPrice) | - | helper.ts hash changes to def456 |
| T3 | Request `/products` | `v18:products.tsx:abc123:browser` | **HIT** - stale bundle! |

The cache key `abc123` didn't change because `products.tsx` wasn't modified, even though its dependency `helper.ts` was.

## Impact Assessment

| Scenario | Expected Behavior | Actual Behavior |
|----------|-------------------|-----------------|
| Update shared component | All importers rebuild | Only modified files rebuild |
| Fix bug in utility | Fix deployed immediately | Fix invisible until cache expires |
| Update layout component | All child pages re-render | Child pages show old layout |
| Shared helper change | Consistent behavior | Inconsistent behavior across pages |

## Risk Assessment

| Factor | Rating |
|--------|--------|
| **Impact** | CRITICAL - Users see stale content |
| **Frequency** | HIGH - Every shared file change |
| **Detectability** | LOW - No errors, just wrong content |
| **Blast Radius** | HIGH - All files depending on changed file |
| **Fix Complexity** | MEDIUM - Requires import graph traversal |

## Current Workarounds

1. **TRANSFORM_CACHE_VERSION bump** - Nuclear option, invalidates ALL caches
2. **5-minute cache TTL** - Stale window limited but still exists
3. **Project cache clear** - Manual intervention required

## What's Missing

1. **Import graph construction** - Must parse and track all imports
2. **Dependency hash computation** - Hash all transitive dependencies
3. **Cache key inclusion** - Include depsHash in buildTransformCacheKey()
4. **Inverse dependency index** - For targeted invalidation when a file changes

## Solution

See **[004.0 - Dependency Tracking RFC](./004.0-dependency-tracking-rfc.md)** for the complete solution.

The fix requires:
1. Building dependency graph during transform
2. Computing stable hash of all transitive dependencies
3. Including dependency hash in cache key
4. Maintaining inverse dependency index for invalidation
