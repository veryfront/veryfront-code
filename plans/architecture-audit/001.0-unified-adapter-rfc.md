# 001.0: RFC - Unified Adapter Architecture

## TL;DR

**The Problem**: We have 4 different filesystem adapters that behave differently, causing "works locally, breaks in production" bugs that are nearly impossible to debug.

**The Solution**: Unify all adapters behind a single interface so business logic has ONE code path, not 4.

---

## The Complexity Problem

### Multiple Adapters = Multiple Code Paths = Multiple Bugs

The renderer currently has **4 filesystem adapters**:

| Adapter | When Used | Behavior |
|---------|-----------|----------|
| Local Filesystem | `deno task dev` | Direct Deno fs |
| Veryfront API (preview) | `*.preview.veryfront.com` | API calls, draft content |
| Veryfront API (production) | `*.veryfront.com` | API calls, released content |
| GitHub | GitHub-backed projects | GitHub API |

Instead of a unified interface, **business logic is littered with adapter-specific conditionals**:

```typescript
// This pattern appears 8+ times across the codebase
if (isVeryfrontAdapter(fs)) {
  // Do thing one way
} else {
  // Do thing completely differently
}
```

### Why This Causes Bugs

Each conditional **doubles** the number of code paths:

```
1 conditional  = 2 paths
2 conditionals = 4 paths
3 conditionals = 8 paths
8 conditionals = 256 possible paths
```

**We cannot test 256 combinations.** So bugs slip through.

### Real Example: The Layout Bug

The layout collector has this code:

```typescript
if (isVeryfrontAPI) {
  return await this.collectAPILayoutConfiguration(fs);  // Only checks config + components/
} else {
  return await this.collectFilesystemLayouts(pageFilePath); // Walks app/ directory tree
}
```

**Result**: App Router nested layouts (`app/dashboard/layout.tsx`) work locally but **completely disappear** in production. Users report "my sidebar is gone" and we spend hours debugging.

## The Unified Adapter Solution

### Principle: One Interface, One Code Path

All adapters implement the **exact same interface**. Business logic calls the interface. Adapters handle the differences internally.

```typescript
// BEFORE: Business logic knows about adapter types
if (isVeryfrontAdapter(fs)) {
  layouts = await collectAPILayouts(fs);
} else {
  layouts = await collectFilesystemLayouts(fs);
}

// AFTER: Business logic is adapter-agnostic
layouts = await collectLayouts(adapter);  // Adapter handles internally
```

### Phase 1: Define Unified Interface

```typescript
interface UnifiedFSAdapter {
  // Core ops - ALL adapters must behave identically
  readFile(path: string): Promise<string>;
  exists(path: string): Promise<boolean>;
  stat(path: string): Promise<FileInfo>;
  readDir(path: string): AsyncIterable<DirEntry>;

  // Directory walking - required for layout discovery
  walkDirectory(root: string, filter?: FilterFn): AsyncIterable<string>;

  // Optional metadata - returns undefined if not available
  getProjectMetadata?(): ProjectMetadata | undefined;
}
```

**Key insight**: API adapters CAN walk directories - they just need to implement it. The current code assumes they can't, which is wrong.

### Phase 2: Fix Layout Discovery (Critical Bug Fix)

Merge `collectAPILayoutConfiguration` and `collectFilesystemLayouts` into ONE function:

```typescript
private async collectNestedLayouts(pageInfo: EntityInfo): Promise<LayoutItem[]> {
  const useAppRouter = await detectAppRouter(this.projectDir, this.config, this.adapter);
  const rootDir = useAppRouter
    ? join(this.projectDir, "app")
    : join(this.projectDir, "pages");

  // SINGLE code path for ALL adapters
  return await discoverNestedLayouts(pageFilePath, rootDir, this.adapter);
}
```

### Phase 3: Remove Adapter Conditionals

For each `isVeryfrontAdapter()` check:

1. **If behavior should be identical** → delete the conditional, use unified interface
2. **If behavior must differ** → move the conditional INTO the adapter, not business logic

```typescript
// Move this INTO the adapter
class VeryfrontFSAdapter implements UnifiedFSAdapter {
  async loadConfig(path: string): Promise<Config> {
    // Adapter-specific: use esbuild for virtual FS
    return this.transpileAndEvaluate(path);
  }
}

class LocalFSAdapter implements UnifiedFSAdapter {
  async loadConfig(path: string): Promise<Config> {
    // Adapter-specific: use native import
    return import(path);
  }
}
```

### Phase 4: Cross-Adapter Integration Tests

The only way to guarantee identical behavior is to **run the same tests against all adapters**:

```typescript
describe.each(['local', 'veryfront-api', 'github'])('Layout Discovery (%s)', (adapterType) => {
  it('discovers nested app router layouts', async () => {
    const adapter = createAdapter(adapterType, {
      files: {
        'app/layout.tsx': rootLayout,
        'app/dashboard/layout.tsx': dashboardLayout,
        'app/dashboard/page.tsx': page,
      }
    });

    const layouts = await collectLayouts(adapter);

    // Same assertion for ALL adapters
    expect(layouts.map(l => l.path)).toEqual([
      'app/layout.tsx',
      'app/dashboard/layout.tsx',
    ]);
  });
});
```

## Why This Matters

Every hour spent debugging "works locally, breaks in production" is an hour not spent shipping features. The current architecture **guarantees** these bugs will keep happening.

Unifying the adapters is a one-time investment that:
- Eliminates an entire class of bugs
- Makes the codebase easier to understand
- Enables confident refactoring
- Reduces test complexity from O(2^n) to O(n)
