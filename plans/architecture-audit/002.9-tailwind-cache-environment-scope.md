# 002.9: Tailwind CSS Cache Missing Environment Scope

## Severity: HIGH

## Location

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/html/styles-builder/tailwind-compiler.ts`
**Line:** 133

```typescript
// Current implementation - NO environment in key
const cacheKey = `${projectSlug}:${stylesheetHash}`;
```

## The Problem

The project CSS cache key does not include environment (preview vs production). This means:

1. Preview and production share the same cache entry
2. CSS changes in preview can leak to production
3. Production rollbacks may serve stale preview CSS

### Failure Scenario

```
Timeline:
=========
t0: Production serving globals.css v1
    → cacheKey = "my-project:abc123"
    → CSS compiled with blue theme

t1: Developer changes globals.css to red theme (preview only)
    → Same project, different content
    → stylesheetHash changes to "def456"
    → cacheKey = "my-project:def456"
    → Preview shows red theme (correct)

t2: Developer reverts change, publishes original
    → stylesheetHash = "abc123" again
    → cacheKey = "my-project:abc123"
    → Cache HIT - returns old blue CSS
    → Looks correct... but wait

PROBLEM SCENARIO:

t0: Production has globals.css with @theme { --primary: blue }
    → cacheKey = "my-project:abc123"

t1: Preview adds new utility class usage (bg-gradient-to-r)
    → Same stylesheet hash (content unchanged)
    → But different candidates!
    → candidatesHash changes
    → Cache MISS, recompiles with preview candidates
    → cacheKey = "my-project:abc123" now has preview's candidatesHash

t2: Production request comes in
    → cacheKey = "my-project:abc123"
    → candidatesHash doesn't match (prod has fewer classes)
    → Recompiles... but this is wasted work

OR WORSE:

t1: Preview and Production have same stylesheet AND same candidates
    → Preview compiles first with preview-only file content
    → Production gets preview's compiled CSS
```

### Environment Bleed-Through

| Source | Destination | Risk |
|--------|-------------|------|
| Preview → Production | Preview CSS served to production users | HIGH |
| Production → Preview | Stale production CSS masks preview changes | MEDIUM |
| Branch A → Branch B | Different branches share cache | MEDIUM |

## Proposed Fix

### Option A: Add Environment to Cache Key (Recommended)

```typescript
export async function getProjectCSS(
  projectSlug: string,
  environment: "preview" | "production",  // Add parameter
  stylesheet: string | undefined,
  candidates: Set<string>,
  options?: GenerateOptions,
): Promise<{ css: string; hash: string; fromCache: boolean }> {
  const stylesheetHash = hashString(stylesheet ?? DEFAULT_STYLESHEET);
  const candidatesHash = hashCandidates(candidates);

  // Include environment in cache key
  const cacheKey = `${projectSlug}:${environment}:${stylesheetHash}`;

  // ... rest unchanged
}
```

### Option B: Include Release/Branch ID

For even finer isolation:

```typescript
interface CSSCacheContext {
  projectSlug: string;
  environment: "preview" | "production";
  releaseId?: string;  // For production versions
  branchId?: string;   // For preview branches
}

function buildCSSCacheKey(ctx: CSSCacheContext, stylesheetHash: string): string {
  const scope = ctx.releaseId ?? ctx.branchId ?? ctx.environment;
  return `css:${ctx.projectSlug}:${scope}:${stylesheetHash}`;
}
```

  // Production compiles first
  const prod1 = await getProjectCSS("test", "production", baseStylesheet, new Set(["text-primary"]));

  // Preview compiles with different stylesheet
  const prev = await getProjectCSS("test", "preview", previewStylesheet, new Set(["text-primary"]));

  const prod2 = await getProjectCSS("test", "production", baseStylesheet, new Set(["text-primary"]));

  assertNotEquals(prod1.css, prev.css);
});
