# 008.4 - Incomplete HMR Cache Invalidation

## Severity: HIGH

## Location

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/server/dev-server/request-handler.ts`
**Lines:** 148-161

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/server/bootstrap.ts`
**Lines:** 84-86

## The Problem

HMR invalidation calls `clearConfigCache()` which clears **all** cached configs:

```typescript
// src/server/dev-server/request-handler.ts:148-161
invalidateUniversalHandler(): void {
  this.universalHandler = undefined;

  // Also reset the API handler cache to pick up new/modified handlers
  resetApiHandler(this.projectDir).catch((error) => {
    logger.debug("[dev] resetApiHandler failed", error);
  });

  // Clear config cache so HMR picks up config changes
  clearConfigCache();  // CLEARS ALL PROJECTS

  // Clear layout discovery cache so HMR picks up layout changes
  clearLayoutDiscoveryCache();
}
```

The `clearConfigCache()` function:

```typescript
// src/config/loader.ts:401-404
export function clearConfigCache(): void {
  configCacheByProject.clear();  // Clears ENTIRE Map
  cacheRevision++;               // Bumps GLOBAL revision
}
```

## Why It's Bad

### Issue 1: Over-Broad Invalidation

When one project's config changes, ALL projects' configs are invalidated:

```
Developer edits codersociety.veryfront.config.ts
  -> HMR triggers invalidateUniversalHandler()
  -> clearConfigCache() called
  -> configCacheByProject.clear() removes ALL entries
  -> Next request for ANY project reloads config
```

### Issue 2: Cascading Invalidation

Other caches depend on config but aren't invalidated with it:

```typescript
// Related caches that may contain stale config references:

// Layout discovery (cleared separately)
clearLayoutDiscoveryCache();

// But these are NOT cleared:
// - Module transform cache (depends on config.experimental.rsc)
// - Tailwind compiler state (depends on config.tailwind)
// - SSR module cache (depends on config.react.version)
// - Import map cache (depends on config.resolve.importMap)
```

### Issue 3: Timing Dependency

The invalidation order matters but isn't enforced:

```typescript
invalidateUniversalHandler(): void {
  this.universalHandler = undefined;

  // Async without await - race condition possible
  resetApiHandler(this.projectDir).catch((error) => {
    logger.debug("[dev] resetApiHandler failed", error);
  });

  clearConfigCache();           // Sync
  clearLayoutDiscoveryCache();  // Sync
}
```

If a request arrives while `resetApiHandler` is still running, it might use:
- New config (cleared cache)
- Old API handlers (still being reset)

## Observed Symptoms

1. Config changes don't take effect immediately
2. Inconsistent state where some features use new config, others use old
3. "Works if I restart the server" - indicates cache invalidation is incomplete
4. HMR sometimes requires multiple saves to pick up config changes

## Multi-Tenancy Impact

In development with multiple projects running on same server:
- Config change in Project A invalidates Project B's cached config
- Project B experiences unexpected latency spike on next request
- If Project B's config load fails, it could affect development

None of these caches are invalidated when config changes.

## Solution

See **[008.0 - Userland Config RFC](./008.0-userland-config-rfc.md)** for the complete solution.

Key changes:
1. Add per-project config invalidation: `invalidateProjectConfig(projectId)`
2. Create dependency graph of caches that depend on config
3. Implement cascading invalidation that clears dependent caches
4. Await all invalidation operations before proceeding

```typescript
async function invalidateProjectCaches(projectId: string): Promise<void> {
  // Invalidate in dependency order
  await invalidateProjectConfig(projectId);
  await invalidateLayoutDiscoveryCache(projectId);
  await invalidateModuleTransformCache(projectId);
  await invalidateTailwindCache(projectId);
  await invalidateSSRModuleCache(projectId);
}
```
