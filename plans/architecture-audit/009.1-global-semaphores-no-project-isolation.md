# 009.1 - Global Semaphores Without Per-Project Isolation

## Severity: CRITICAL

## Location

**Files:**
- `/Users/mattboon/Sites/veryfront-renderer/src/rendering/renderer.ts` (lines 98-110)
- `/Users/mattboon/Sites/veryfront-renderer/src/modules/react-loader/ssr-module-loader/cache/memory.ts` (lines 43)
- `/Users/mattboon/Sites/veryfront-renderer/src/data/static-data-fetcher.ts` (lines 17-19)

## The Problem

All three critical semaphores are **global singletons** shared across all projects:

```typescript
// rendering/renderer.ts:98-110
const RENDER_MAX_CONCURRENT = parseInt(getEnv("RENDER_MAX_CONCURRENT") ?? "30", 10);
const RENDER_ACQUIRE_TIMEOUT_MS = 5000;
const renderSemaphore = new Semaphore(RENDER_MAX_CONCURRENT);

// ssr-module-loader/cache/memory.ts:43
export const transformSemaphore = new Semaphore(MAX_CONCURRENT_TRANSFORMS);

// data/static-data-fetcher.ts:17-19
const revalidationSemaphore = getSemaphore("revalidation", MAX_CONCURRENT_REVALIDATIONS, {
  acquireTimeoutMs: 5000,
});
```

## Why It's Bad for Multi-Tenancy

In a multi-tenant deployment, one pod serves **hundreds of projects**. When Project A has a slow/hanging operation:

1. **Render Semaphore (30 permits)**: Project A's 30 slow renders exhaust all permits
2. **Transform Semaphore (50 permits)**: Project A's complex module tree uses all transforms
3. **Revalidation Semaphore (32 permits)**: Project A's many stale pages trigger revalidations

**Result**: Projects B, C, D, ... Z cannot render, transform, or revalidate until Project A's operations complete or timeout.

## Failure Scenario

### Timeline: Project A Starves Other Projects

```
T+0s:    Project A receives 30 concurrent requests
         renderSemaphore: 30/30 permits acquired

T+0.1s:  Projects B, C, D each receive 5 requests
         All 15 requests enter waitQueue (no permits available)

T+5s:    RENDER_ACQUIRE_TIMEOUT_MS fires for waiting requests
         Projects B, C, D get 503 "Service is overloaded"

T+5-60s: Project A's renders continue (up to 60s each)
         More requests from B, C, D continue failing

T+60s:   Project A's first render completes
         One permit released, one B/C/D request succeeds
         29 B/C/D requests still waiting/failing
```

### Real-World Impact

| Metric | Without Isolation | With Per-Project Isolation |
|--------|-------------------|---------------------------|
| Max affected projects | All | Only the slow project |
| 503 error blast radius | 100% of traffic | ~3% (one project) |
| Recovery time | 60s (full pipeline timeout) | 5s (per-project limit) |

## Solution

See **[009.0 - Timeout Handling RFC](./009.0-timeout-handling-rfc.md)** Section 4 for the complete per-project semaphore solution.

### Summary

Replace global semaphores with `ProjectSemaphoreManager` that enforces:
- Per-project limit (e.g., 5 renders per project)
- Global limit (e.g., 30 renders total per pod)
- Both must be acquired, preventing any single project from monopolizing resources

```typescript
// Before (global)
await renderSemaphore.tryAcquire(RENDER_ACQUIRE_TIMEOUT_MS);

// After (per-project)
await renderSemaphores.acquire(ctx.projectId, async () => {
  // render logic
});
```
