# 007.3 - DEFAULT_CONFIG Shared Reference Risk

## Severity: HIGH

## Location

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/config/loader.ts`
**Lines:** 26-72, 111-162, 207

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/config/runtime-config.ts`
**Lines:** 53-81, 139

## The Problem

`DEFAULT_CONFIG` is a module-level mutable object that is spread into merged configs:

```typescript
// src/config/loader.ts:26-72
const DEFAULT_CONFIG: Partial<VeryfrontConfig> = {
  title: "Veryfront App",
  description: "Built with Veryfront",
  experimental: {
    esmLayouts: true,  // Nested mutable object
  },
  // ...
  resolve: {
    importMap: getDefaultImportMapForConfig(),  // FUNCTION CALL at module load time
  },
  client: {
    moduleResolution: "cdn",
    cdn: {
      provider: "esm.sh",
      versions: "auto",  // Nested nested object
    },
  },
};
```

The merge function uses shallow spreads:

```typescript
// src/config/loader.ts:111-162
function mergeConfigs(userConfig: Partial<VeryfrontConfig>): VeryfrontConfig {
  const merged = {
    ...DEFAULT_CONFIG,        // Reference to shared nested objects
    ...userConfig,
    client: {
      ...DEFAULT_CONFIG.client,    // Shallow copy of shared object
      ...userConfig.client,
      cdn: {
        ...DEFAULT_CONFIG.client?.cdn,  // Even deeper nesting
        ...userConfig.client?.cdn,
      },
    },
    // ...
  };
  // ...
}
```

The merged config is then cached:

```typescript
// src/config/loader.ts:207
configCacheByProject.set(cacheKey, { revision: cacheRevision, config: merged });
```

## Why It's Bad for Multi-Tenancy

### Issue 1: `getDefaultImportMapForConfig()` Called Once

```typescript
// Line 63-64
resolve: {
  importMap: getDefaultImportMapForConfig(),  // Executed ONCE at module load
},
```

This function returns an object based on `REACT_DEFAULT_VERSION`. If the version changes or needs to differ per-project, all projects share the same stale import map.

### Issue 2: Shallow Spread Doesn't Deep Clone

While the top-level spread creates a new object, nested objects may share references:

```typescript
const merged = {
  ...DEFAULT_CONFIG,  // Creates new top-level object
  // BUT if userConfig doesn't override 'experimental',
  // merged.experimental === DEFAULT_CONFIG.experimental (same reference!)
};
```

### Issue 3: Cached Config Mutation

If ANY code later mutates the cached config, that mutation affects all future accesses:

```typescript
// Hypothetical code path that modifies config:
const config = await getConfig(projectDir, adapter);
config.experimental.newFeature = true;  // MUTATES CACHED OBJECT!

// Next request to same project:
const config2 = await getConfig(projectDir, adapter);  // Returns SAME cached object
config2.experimental.newFeature  // TRUE - mutation leaked!
```

### Multi-Tenancy Scenario

```
Timeline:
=========
t0: Module loads, DEFAULT_CONFIG.resolve.importMap created with React 18.x

t1: Project Alpha request
    → mergeConfigs() runs
    → merged.resolve.importMap = { ...DEFAULT_CONFIG.resolve.importMap, ...userMap }
    → IF userMap is empty, this SHARES the reference with DEFAULT_CONFIG
    → Config cached in configCacheByProject

t2: Code somewhere adds a custom import for Alpha:
    cachedConfig.resolve.importMap.imports["@alpha/lib"] = "..."
    → This MUTATES the object shared with DEFAULT_CONFIG!

t3: Project Beta request
    → mergeConfigs() runs
    → merged.resolve.importMap = { ...DEFAULT_CONFIG.resolve.importMap, ...userMap }
    → DEFAULT_CONFIG.resolve.importMap now contains "@alpha/lib" from Project Alpha!
    → Project Beta gets Alpha's custom imports

t4: Project Gamma request
    → Same contamination - all projects share Alpha's imports
```

## Failure Scenario

```
Step 1: Project Alpha config: { resolve: { importMap: { imports: {} } } }
Step 2: mergeConfigs creates: { resolve: { importMap: { imports: { ...DEFAULT, ...{} } } } }
Step 3: Spread is shallow - imports object may share DEFAULT references
Step 4: Plugin/middleware adds Alpha-specific import: config.resolve.importMap.imports["@alpha/ui"] = "..."
Step 5: Mutation affects shared DEFAULT_CONFIG.resolve.importMap
Step 6: Project Beta loads - gets contaminated import map
Step 7: Beta tries to import "@alpha/ui" - finds it! Uses wrong URL!
Step 8: Beta renders with Alpha's UI library - cross-tenant code execution
```

## Solution

See **[007.0 - Config Normalization RFC](./007.0-config-normalization-rfc.md)** for the complete solution.

Key fixes:
1. **Deep freeze DEFAULT_CONFIG** to prevent mutation
2. **Deep clone when merging** to prevent shared references
3. **Freeze normalized configs** before caching
4. **Call import map function per-request**, not at module load

```typescript
// Fresh defaults per request
function createFreshDefaults(): VeryfrontConfig {
  return {
    title: "Veryfront App",
    resolve: {
      importMap: getDefaultImportMapForConfig(),  // Fresh call each time
    },
    // ...
  };
}

// Deep clone in merge
function mergeConfigs(userConfig: Partial<VeryfrontConfig>): VeryfrontConfig {
  const defaults = createFreshDefaults();  // Fresh copy
  return structuredClone({  // Deep clone result
    ...defaults,
    ...userConfig,
    // Deep merge nested objects...
  });
}

// Freeze before caching
const frozen = deepFreeze(normalized);
configCacheByProject.set(cacheKey, { revision: cacheRevision, config: frozen });
```
