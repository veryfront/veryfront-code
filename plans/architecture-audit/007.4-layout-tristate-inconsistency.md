## The Problem

Layout configuration accepts three states (`undefined`, `false`, `string`) but handling is scattered across multiple files:

```typescript
// src/config/types.ts:22-25
/** Path to the layout component (e.g., 'components/layout.tsx'), or false to disable */
layout?: string | false;
/** Path to the app wrapper component (e.g., 'components/app.tsx'), or false to disable */
app?: string | false;

// src/config/schema.ts:45-47
layout: z.union([z.string(), z.literal(false)]).optional(),
app: z.union([z.string(), z.literal(false)]).optional(),
```

The three states have different meanings:
- `undefined` = Auto-discover layout (default behavior)
- `false` = Explicitly disable layout
- `string` = Use specific layout path

Handling is inconsistent across the codebase:

```typescript
// src/rendering/layouts/layout-collector.ts:74-75

// src/rendering/layouts/layout-collector.ts:201-204
} else if (this.config?.layout === false) {
  layoutName = null;
} else if (typeof this.config?.layout === "string" && this.config.layout.length > 0) {
  layoutName = this.config.layout;

// src/rendering/layouts/layout-collector.ts:275-276
if (configLayout === false) {
```

## Why It's Bad for Multi-Tenancy

### Inconsistent Behavior Based on Source

The frontmatter handler accepts the string `"false"`, but the config handler doesn't:

```typescript
// Frontmatter: layout: "false" → treated as FALSE (disabled)
if (layoutValue === false || layoutValue === "false")

// Config: layout: "false" → treated as STRING path "false" → error!
typeof this.config?.layout === "string" && this.config.layout.length > 0
// This would try to find a layout file named "false"!
```

### Multi-Tenancy Scenario

```
Project Alpha config: { layout: false }      → Layout disabled (correct)
Project Beta frontmatter: layout: "false"    → Layout disabled (correct, but inconsistent)
Project Gamma config: { layout: "false" }    → Tries to load "false" as path (WRONG!)
```

### No Clear Default Handling

```typescript
// What happens when layout is undefined?

  // Not matched
} else if (typeof this.config?.layout === "string" && this.config.layout.length > 0) {
  // Not matched

```

## Failure Scenario

```
Step 1: Developer sets layout: "false" (string, typo - meant boolean false)
Step 2: Schema validation passes (it's a valid string)
Step 3: layout-collector.ts:201-204 runs:
        typeof this.config?.layout === "string" → TRUE
        this.config.layout.length > 0 → TRUE
        layoutName = "false"
Step 4: getLayoutEntity("false") called
Step 5: System tries to find "false.tsx", "false.jsx", etc.
Step 6: Layout not found → throws error:
        'Layout "false" not found. Specified in config for page...'
Step 7: User confused - they thought they disabled layout!
```

## Solution

See **[007.0 - Config Normalization RFC](./007.0-config-normalization-rfc.md)** for the complete solution.

Normalize to a consistent object format:

```typescript
interface NormalizedLayoutConfig {
  readonly enabled: boolean;
  readonly path: string | null;
}

function normalizeLayout(layout?: string | false): NormalizedLayoutConfig {
  // Handle string "false" as false
  if (layout === false || layout === "false") {
    return Object.freeze({ enabled: false, path: null });
  }
  if (typeof layout === "string" && layout.length > 0) {
    return Object.freeze({ enabled: true, path: layout });
  }
  // undefined = auto-discover
  return Object.freeze({ enabled: true, path: null });
}
```

Consumer code becomes simpler:

```typescript
// BEFORE: Multiple checks
if (this.config?.layout === false) {
  // disabled
} else if (typeof this.config?.layout === "string" && this.config.layout.length > 0) {
  // custom path
} else {
  // auto-discover
}

// AFTER: Single normalized structure
if (!this.config.layout.enabled) {
  // disabled
} else if (this.config.layout.path) {
  // custom path
} else {
  // auto-discover
}
```
