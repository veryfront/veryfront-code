# RFC 013.0: Cache Key Consistency and Multi-Tenant Isolation

## Status
PROPOSED

## Summary

The veryfront-renderer uses **18+ distinct cache systems** with inconsistent key formats, mixed project-scoping strategies, and varying TTLs. This creates risk of cross-tenant data leakage and makes debugging difficult. This RFC proposes unifying cache key patterns while preserving the intentional content-addressed caching that enables cross-project code sharing.

## Problem Statement

### Current State
The codebase has evolved organically, resulting in:

1. **Inconsistent key formats**: Some use `:` separators, some use `::`, some use `-`
2. **Mixed project-scoping**: Some caches include projectId, some rely on content-hash alone
3. **No unified version prefix**: Different caches use different version schemes
4. **Unclear documentation**: Cache key formats are not documented consistently

### Multi-Tenancy Context
Veryfront is a **multi-tenant SSR framework** where:
- One renderer pod serves multiple projects
- Cache isolation between projects is critical for security
- Some caches (transforms, HTTP bundles) are intentionally shared across projects via content-addressing

### The Design Principle
**Content-addressed caches are safe to share** because:
- If two projects have identical source code, the same transform output is correct
- Content hash ensures cache invalidation when code changes
- This significantly improves cache hit rates for common patterns

**Identity-based caches must be project-scoped** because:
- Render cache contains project-specific HTML
- Layout cache contains project-specific component trees
- Data fetching cache contains project-specific API responses

## Analysis

### Cache Categories

| Category | Scoping | Examples | Risk Level |
|----------|---------|----------|------------|
| **Content-Addressed** | Hash-based (safe to share) | Transform Cache, HTTP Module Cache, MDX Bundle Cache | Low |
| **Identity-Based** | Project-scoped (must isolate) | Render Cache, Data Fetching Cache, Layout Cache | High if unscoped |
| **Context-Scoped** | Request/session-scoped | Request Cache Batcher | None (request lifetime) |

### Key Format Inconsistencies

| Cache | Current Format | Issues |
|-------|----------------|--------|
| Transform Cache | `v{ver}:{filePath}:{contentHash}:{ssr}` | No projectId - intentional (content-addressed) |
| HTTP Module Cache | `{prefix}:{hash}` | URL-hash based - safe |
| MDX Bundle Cache | `mdx:{mode}:{contentHash}` | No projectId - intentional (content-addressed) |
| SSR Module Cache | `v{ver}:{projectId}:{contentSourceId}:...` | Project-scoped - correct |
| Agent Cache | `cache_{hash(input)}` | **No projectId - RISK** |
| Render Cache | `{projectId}:{env}:{releaseKey}:{ver}:...` | Project-scoped - correct |
| Data Fetching Cache | `veryfront:data:{projectId}:...` | Project-scoped - correct |

### Critical Findings

1. **Agent Cache Missing Project Context** (Medium Risk)
   - File: `src/agent/middleware/cache/cache.ts:204`
   - Key: `cache_{hash(input)}`
   - If same prompt used in different projects, could return wrong response

2. **Transform Cache Correctly Content-Addressed** (No Fix Needed)
   - File: `src/cache/keys.ts:260-269`
   - Key: `v{ver}:{filePath}:{contentHash}:{ssr}`
   - Content-hash makes cross-project sharing safe and beneficial

3. **Separator Inconsistency** (Low Risk, Maintenance Issue)
   - Data fetching uses `::` between pathname and params
   - Theme suffix uses `-` (`theme-light`)
   - Most others use `:`

## Proposed Solution

### 1. Unified Key Format Standard

```
{namespace}:{version}:{scope}:{qualifier}:{resource}
```

Where:
- `namespace`: Cache type identifier (e.g., `veryfront:transform`)
- `version`: Cache format version for invalidation
- `scope`: Project-scoping strategy
  - For content-addressed: `content` (no projectId)
  - For identity-based: `{projectId}:{environment}`
- `qualifier`: Content-specific identifier
- `resource`: Resource path or key

### 2. Type-Safe Key Builders

```typescript
// Content-addressed (shareable across projects)
function buildContentAddressedKey(
  namespace: string,
  contentHash: string,
  suffix?: string
): string {
  return `${namespace}:v${VERSION}:content:${contentHash}${suffix ? `:${suffix}` : ''}`;
}

// Identity-based (project-isolated)
function buildProjectScopedKey(
  namespace: string,
  projectId: string,
  environment: 'preview' | 'production',
  versionId: string,
  resource: string
): string {
  return `${namespace}:v${VERSION}:${projectId}:${environment}:${versionId}:${resource}`;
}
```

### 3. Agent Cache Fix

Add project context to agent cache key generation:

```typescript
// BEFORE (cache.ts:204)
const keyGenerator = config.keyGenerator ?? ((input: string) => `cache_${hashString(input)}`);

// AFTER
const keyGenerator = config.keyGenerator ?? ((input: string, context?: Record<string, unknown>) => {
  const projectId = context?.projectId ?? 'global';
  return `agent:${projectId}:${hashString(input)}`;
});
```

### 4. Documentation Standard

Each cache key builder should include JSDoc:

```typescript
/**
 * Build transform cache key (content-addressed, cross-project shareable)
 *
 * Key Format: v{version}:{filePath}:{contentHash}:{ssr|browser}[:studio]
 *
 * Content-addressed keys are safe to share across projects because:
 * - Same source code produces same transform output
 * - Content hash ensures invalidation on code changes
 *
 * @param filePath - Normalized file path (no project prefix needed)
 * @param contentHash - SHA256 hash of source content
 * @param ssr - Whether transform is for server-side rendering
 * @param studioEmbed - Whether transform includes studio embed markers
 */
export function buildTransformCacheKey(...): string
```

## Implementation Plan

### Phase 1: Documentation (No Code Changes)
- Add JSDoc to all key builders in `src/cache/keys.ts`
- Document cache categories in architecture docs
- Add cache key format table to CLAUDE.md

### Phase 2: Fix Agent Cache (Small Change)
- Update `src/agent/middleware/cache/cache.ts` to include project context
- Add test for cross-project isolation

### Phase 3: Standardize Separators (Optional)
- Migrate `::` to `:` where safe
- Migrate `theme-{color}` to `theme:{color}`
- Requires careful cache invalidation planning

### Phase 4: Unified Key Builders (Future)
- Create typed key builder factory
- Migrate existing builders to use factory
- Add compile-time enforcement of key format

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Cache invalidation during migration | Phase rollouts with cache warmup periods |
| Breaking existing Redis keys | Version bump in key prefix forces fresh cache |
| Performance regression | Content-addressed sharing is maintained |

## Decision Required

1. **Approve RFC** - Proceed with phased implementation
2. **Modify** - Adjust scope or approach
3. **Defer** - Document but don't implement
