# 016.4: Path Traversal in Adapters

## Severity: MEDIUM

## Location

**Files:** FS adapter implementations
- `src/platform/adapters/fs/local/`
- `src/platform/adapters/fs/veryfront/`

## The Problem

Path operations don't consistently validate against directory traversal attacks. Malicious paths like `../../etc/passwd` may escape project boundaries.

### Vulnerable Pattern

```typescript
async function readFile(projectRoot: string, relativePath: string): Promise<string> {
  // VULNERABLE - no validation
  const fullPath = join(projectRoot, relativePath);
  return await Deno.readTextFile(fullPath);
}

// Attack:
readFile("/app/projects/my-project", "../../etc/passwd")
// Resolves to: /app/etc/passwd
```

### Attack Vectors

```
Simple traversal:
../../../etc/passwd

Encoded traversal:
..%2F..%2F..%2Fetc/passwd

Unicode normalization:
..／..／etc/passwd (fullwidth slash)

Null byte (legacy):
../../../etc/passwd%00.txt
```

## Proposed Fix

### Centralized Path Validation

```typescript
import { resolve, relative } from "std/path/mod.ts";

function validatePath(root: string, requestedPath: string): string {
  // Resolve to absolute path
  const absoluteRoot = resolve(root);
  const absoluteRequested = resolve(root, requestedPath);

  // Check if resolved path is within root
  const relativized = relative(absoluteRoot, absoluteRequested);

  if (relativized.startsWith("..") || resolve(relativized) !== relativized) {
    throw new Error(`Path traversal detected: ${requestedPath}`);
  }

  return absoluteRequested;
}

// Safe usage
async function readFile(projectRoot: string, relativePath: string): Promise<string> {
  const safePath = validatePath(projectRoot, relativePath);
  return await Deno.readTextFile(safePath);
}
```

### Input Sanitization

```typescript
function sanitizePath(path: string): string {
  return path
    // Remove null bytes
    .replace(/\0/g, "")
    // Normalize slashes
    .replace(/\\/g, "/")
    // Remove double slashes
    .replace(/\/+/g, "/")
    // Decode URL encoding
    .replace(/%2F/gi, "/")
    // Remove leading traversal
    .replace(/^(\.\.\/)+/, "");
}
```

## Blast Radius

| Affected | Impact |
|----------|--------|
| Local adapter | Read any file on system |
| Config loading | Expose secrets |
| Static file serving | Serve unintended files |

## Decision Required

**D011**: Should path validation be:
- A) Centralized utility (all adapters call it)
- B) Per-adapter (each implements own validation)
- C) Middleware (validate before reaching adapter)

## Related

- OWASP: Path Traversal
- CWE-22: Improper Limitation of a Pathname to a Restricted Directory

## Status: ❌ FALSE POSITIVE

### Resolution

Investigation found the codebase already has comprehensive, defense-in-depth path traversal protection:

1. **SecureFs wrapper** (`src/security/secure-fs.ts`) — All user-facing file operations go through this layer
2. **Multi-layer validation** (`src/security/path-validation/`):
   - `rules.ts` — Null byte detection, path length limits, forbidden patterns, traversal depth limits
   - `normalization.ts` — Resolves `..` and `.` segments, normalizes separators, `isWithinDirectory()` boundary check
   - `canonical.ts` — Symlink resolution, directory whitelist validation, base directory enforcement
3. **Context-based presets** — Different security levels for static serving, dev tools, etc.

The `PathNormalizer` classes in FS adapters don't validate traversal, but this is safe because they only operate on **pre-validated paths** from `SecureFs`.

All user-facing entry points (static file handler, dev file path handler) use `createSecureFs()` with appropriate context presets.
