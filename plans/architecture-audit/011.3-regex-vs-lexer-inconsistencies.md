## The Problem

The codebase uses two fundamentally different parsing approaches that handle edge cases differently:

### 1. Dynamic Import Expression Handling

**es-module-lexer (lexer.ts:126-143):**
```typescript
// For dynamic imports with string literals, es-module-lexer's s/e include the quotes.
// We need to preserve the quote style when replacing.
const isDynamic = imp.d > -1;
if (isDynamic) {
  const quote = result[imp.s];
  if (quote === '"' || quote === "'" || quote === "`") {
    result = result.substring(0, imp.s) + quote + replacement + quote + result.substring(imp.e);
  } else {
    // Dynamic import with expression, not string literal - shouldn't happen if n is defined
    result = result.substring(0, imp.s) + replacement + result.substring(imp.e);
  }
}
```

**Regex (ssr-import-rewriter.ts:77-89):**
```typescript
function rewriteBareImports(code: string, version?: string): string {
  const v = version ?? getReactVersion();
  return code.replace(/from\s+["']([^"'./][^"']*)["']/g, (_match, specifier: string) => {
    // ...
  });
}
```

**The Difference:**

Lexer correctly handles:
```javascript
// Dynamic import with expression - lexer skips (imp.n undefined)
const mod = await import(`./modules/${name}.js`);

// Dynamic import with string - lexer processes
const mod = await import("./module.js");
```

Regex misses dynamic imports entirely because pattern only matches `from "..."`:
```javascript
// NOT MATCHED by regex pattern
const mod = await import("lodash");  // Bare specifier not rewritten!
```

### 2. Multi-line Import Handling

**MDX import-rewriter.ts (line-by-line regex):**
```typescript
// Line 89-96
return body
  .split(/\r?\n/)
  .map((line) => {
    const trimmed = line.trimStart();
    if (!trimmed.startsWith("import") && !trimmed.startsWith("export")) return line;
    return rewriteLine(line, basedir, config.target, config.baseUrl);
  })
  .join("\n");
```

**Fails on multi-line imports:**
```javascript
// This import spans two lines - MDX rewriter MISSES IT
import {
  Button,
  Card,
  Modal
} from "@/components";  // <-- Never rewritten!
```

**es-module-lexer handles correctly:**
```typescript
// Lexer parses full AST, not line-by-line
const [imports] = parse(code);
// imports[0].n === "@/components" regardless of line breaks
```

### 3. Quote Style Preservation

**Regex (import-map/transformer.ts:29-35):**
```typescript
transformedCode = transformedCode.replace(
  /((?:import|export)\s+(?:[\w,{}\s*]+\s+from\s+)?|export\s+(?:\*|\{[^}]+\})\s+from\s+)["']([^"']+)["']/g,
  (_match, prefix, specifier) => {
    if (!shouldResolve(specifier, options)) return `${prefix}"${specifier}"`;
    return `${prefix}"${resolve(specifier)}"`;  // Always double quotes!
  },
);
```

**Problem:** Input with single quotes becomes double quotes:
```javascript
// Input
import React from 'react';

// Output (quote style changed!)
import React from "https://esm.sh/react@19.0.0";
```

This can break code style consistency and cause lint errors in user projects.

### 4. Comment and String Literal False Positives

**Regex patterns don't understand JavaScript context:**

```javascript
// This comment mentions import "lodash" from somewhere
const readme = `
  To use: import something from "lodash"
`;
const pattern = /from "lodash"/;  // Also a string!
```

Regex-based rewriters may incorrectly match:
1. Imports mentioned in comments
2. Import statements inside template literals
3. Import patterns in regex literals

**es-module-lexer correctly ignores:**
- Comments (single-line and multi-line)
- String content (template literals, regular strings)
- Regex literals

### 5. Re-export Handling Gaps

**esm-rewriter.ts patterns (lines 15-24):**
```typescript
const patterns: Array<[RegExp, number, PathResolver]> = [
  // Named export from
  [/export\s*\{([^}]+)\}\s*from\s*(["'])(\/[^"']+)\2/g, 3, resolveAbsolute],
  // Star export
  [/export\s*\*\s*from\s*(["'])(\/[^"']+)\1/g, 2, resolveAbsolute],
  // Missing: export default from
];
```

**Missing pattern:**
```javascript
// NOT MATCHED
export { default } from "./module.js";
export { default as Name } from "./other.js";
```

These valid ES module re-export patterns are not handled.

## Why It's Bad for Multi-Tenancy

### Inconsistent Behavior Across Code Paths

Project A's module goes through transform pipeline (lexer-based):
```javascript
// Correctly handles multi-line imports
import {
  Button
} from "@/ui";
// -> import { Button } from "/_vf_modules/ui.js"
```

Project B's MDX content goes through MDX rewriter (regex-based):
```javascript
// FAILS to handle multi-line imports
import {
  Button
} from "@/ui";
// -> import { Button } from "@/ui"  // NOT REWRITTEN!
// Runtime Error: Cannot find module '@/ui'
```

Same framework, different behavior based on file type.

## Failure Scenario

```
Scenario: MDX with Multi-line Import
====================================

1. Developer creates docs/guide.mdx:
   ```mdx
   import {
     CodeBlock,
     Callout,
     Tabs
   } from "@/components/docs";

   # Getting Started
   <CodeBlock language="javascript">...</CodeBlock>
   ```

2. MDX compiler transforms content

3. MDX import-rewriter processes line by line:
   - Line "import {" -> not an import line (no from)
   - Line "  CodeBlock," -> not an import line
   - Line "  Callout," -> not an import line
   - Line "  Tabs" -> not an import line
   - Line "} from "@/components/docs";" -> starts with "}", not import/export
   -> ENTIRE IMPORT SKIPPED!

4. Browser loads page:
   -> Error: Cannot resolve "@/components/docs"
   -> Page fails to render
```

## Solution

See **[011.0 - Import Rewriting RFC](./011.0-import-rewriting-rfc.md)** for the complete unified solution.

### Key Principle: Single Parsing Approach

All import rewriting MUST use es-module-lexer:

```typescript
// ALWAYS use the lexer wrapper
import { replaceSpecifiers } from "./lexer.ts";

// Even for MDX compiled output
export function rewriteMdxImports(
  code: string,
  context: RewriteContext
): Promise<string> {
  return replaceSpecifiers(code, (specifier, isDynamic) => {
    // Unified resolution logic
    return resolveSpecifier(specifier, context);
  });
}
```

### Benefits of Lexer-Only Approach

1. **Correct multi-line handling** - AST-based, not line-based
2. **Dynamic import support** - Properly identifies `import()` calls
3. **Context awareness** - Ignores comments and strings
4. **Quote preservation** - Maintains original quote style
5. **Complete coverage** - All import/export variants

## Testing

```typescript
// tests/parser-consistency.test.ts
import { assertEquals } from "@std/assert";
import { replaceSpecifiers } from "../src/transforms/esm/lexer.ts";

Deno.test("handles multi-line imports", async () => {
  const code = `import {
    Button,
    Card
  } from "@/components";`;

  const result = await replaceSpecifiers(code, (spec) => {
    if (spec.startsWith("@/")) return "/_vf_modules/components.js";
    return null;
  });

  assertEquals(result.includes("/_vf_modules/components.js"), true);
});

Deno.test("handles dynamic imports with bare specifiers", async () => {
  const code = `const lodash = await import("lodash");`;

  const result = await replaceSpecifiers(code, (spec) => {
    if (spec === "lodash") return "https://esm.sh/lodash";
    return null;
  });

  assertEquals(result.includes("https://esm.sh/lodash"), true);
});

Deno.test("ignores imports in comments", async () => {
  const code = `
    // import something from "fake";
    import real from "lodash";
  `;

  let callCount = 0;
  await replaceSpecifiers(code, () => {
    callCount++;
    return null;
  });

  assertEquals(callCount, 1); // Only real import
});

Deno.test("ignores imports in strings", async () => {
  const code = `
    const example = 'import x from "y"';
    import real from "lodash";
  `;

  let callCount = 0;
  await replaceSpecifiers(code, () => {
    callCount++;
    return null;
  });

  assertEquals(callCount, 1); // Only real import
});

Deno.test("preserves quote style", async () => {
  const singleQuote = `import React from 'react';`;
  const doubleQuote = `import React from "react";`;
  const backtick = `const m = await import(\`react\`);`;

  const rewriter = (spec: string) => spec === "react" ? "https://esm.sh/react" : null;

  const r1 = await replaceSpecifiers(singleQuote, rewriter);
  const r2 = await replaceSpecifiers(doubleQuote, rewriter);
  const r3 = await replaceSpecifiers(backtick, rewriter);

  assertEquals(r1.includes("'https://esm.sh/react'"), true);
  assertEquals(r2.includes('"https://esm.sh/react"'), true);
  assertEquals(r3.includes("`https://esm.sh/react`"), true);
});
```
