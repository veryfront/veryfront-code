# 007.6 - Security Config CORS Default Mutation

## Severity: HIGH

## Location

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/security/http/config.ts`
**Lines:** 39-48

## The Problem

`SecurityConfigLoader` mutates the security config object in place:

```typescript
// src/security/http/config.ts:39-48
private applyConfig(cfg?: VeryfrontConfig): void {
  const security: SecurityConfig = cfg?.security ? { ...cfg.security } : {};

  if (security.headers) {
    security.headers = { ...security.headers };  // Shallow clone headers
  }

  security.cors ??= true;  // MUTATES the security object!

  this.securityConfig = security;  // Stores reference
  this.cspUserHeader = this.parseCspUserHeader(security.csp);
  this.isLoaded = true;
}
```

The `??=` operator mutates the `security` object. While a shallow copy is made from `cfg.security`, this happens AFTER the config has been loaded and potentially cached.

## Why It's Bad for Multi-Tenancy

### The Config Object May Be Cached

The `cfg` parameter comes from `getConfig()`, which returns cached configs:

```typescript
// loader.ts:350-358
const cached = configCacheByProject.get(effectiveCacheKey);
if (cached?.revision === cacheRevision) {
  serverLogger.debug("[CONFIG] Cache HIT - using cached config");
  return cached.config;  // Returns CACHED config
}
```

If the cached config's security object is later accessed and mutated, the mutation persists:

```typescript
// Step 1: Config loaded, cached without cors
cached.config = { security: { csp: {...} } }  // No cors field

// Step 2: SecurityConfigLoader.applyConfig runs
const security = { ...cfg.security };  // security = { csp: {...} }
security.cors ??= true;  // security = { csp: {...}, cors: true }

// Step 3: If cfg.security was shallow-cloned but nested objects share references...
// OR if configOverride was passed directly (not from cache)...
```

### The configOverride Path

```typescript
// src/security/http/config.ts:28-31
private async load(): Promise<void> {
  try {
    const cfg = this.configOverride ?? await getConfig(this.projectDir, this.adapter);
    this.applyConfig(cfg);
```

When `configOverride` is provided, it may be a shared object that gets mutated.

### SecurityConfigLoader is Instance-Based

```typescript
constructor(
  private projectDir: string,
  private adapter: RuntimeAdapter,
  private configOverride?: VeryfrontConfig,  // Stored as instance field
) {}
```

Multiple `SecurityConfigLoader` instances may share the same `configOverride` reference.

## Failure Scenario

```
Timeline (Multi-Tenancy):
=========================
t0: Both Project Alpha and Beta have NO cors config in their veryfront.config.ts

t1: Request for Project Alpha
    → SecurityConfigLoader created with configOverride from shared source
    → applyConfig({ security: { csp: {...} } })
    → security.cors ??= true  // Mutates to add cors: true
    → Original configOverride now has cors: true

t2: Request for Project Beta
    → SecurityConfigLoader created with SAME configOverride reference
    → configOverride.security.cors is now TRUE (from Alpha's mutation)
    → Beta now has cors: true even if it wanted cors: false

    → But configOverride was mutated, so cfg.security.cors is already true
    → ??= operator doesn't run (cors already truthy)
```

### The Real Risk: Cache Mutation

```
Timeline (Cache Mutation):
==========================
t0: Project Alpha config loaded, cached:
    configCacheByProject.set("alpha", {
      config: { security: { csp: {...} } }  // No cors
    })

t1: Request hits SecurityConfigLoader
    → const cfg = await getConfig(...)  // Returns cached config
    → const security = { ...cfg.security }  // Shallow clone
    → security.cors ??= true  // Safe - mutates the clone

BUT WHAT IF:
t2: Some code path does:
    const config = await getConfig("alpha", adapter);
    config.security.cors = false;  // MUTATES CACHED CONFIG!

t3: Next request gets cached config with cors: false
    → Security behavior changed for all Alpha requests!
```

The real issue is that configs aren't frozen - `SecurityConfigLoader` could mutate the source if the spread was removed or if `configOverride` is used directly.

## Solution

See **[007.0 - Config Normalization RFC](./007.0-config-normalization-rfc.md)** for the complete solution.

Specific fixes:
1. **Freeze cached configs** so mutation attempts throw errors
2. **Normalize CORS at config load time**, not at SecurityConfigLoader time
3. **Remove the ??= mutation** - normalized config always has cors defined

```typescript
// After normalization, CORS is ALWAYS defined:
interface NormalizedConfig {
  readonly security: {
    readonly cors: NormalizedCorsConfig;  // ALWAYS present, never undefined
    // ...
  };
}

// SecurityConfigLoader becomes simpler:
private applyConfig(cfg: NormalizedConfig): void {
  // No need to set defaults - normalization already did it
  this.securityConfig = cfg.security;  // Frozen, safe to store
  this.cspUserHeader = this.parseCspUserHeader(cfg.security.csp);
  this.isLoaded = true;
}
```

With frozen configs, any attempt to mutate throws:

```typescript
const config = await getConfig("alpha", adapter);
config.security.cors = false;  // TypeError: Cannot assign to read only property 'cors'
```
