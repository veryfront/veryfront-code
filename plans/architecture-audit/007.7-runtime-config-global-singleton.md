# 007.7 - Runtime Config Global Singleton

## Severity: CRITICAL

## Location

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/config/runtime-config.ts`
**Lines:** 151-171

## The Problem

`runtimeConfig` is a global singleton that stores a single config for the entire process:

```typescript
// src/config/runtime-config.ts:151-171
let runtimeConfig: RuntimeConfig | null = null;

export function initRuntimeConfig(fileConfig: VeryfrontConfig = {}): RuntimeConfig {
  if (runtimeConfig) return runtimeConfig;  // Returns existing singleton!
  runtimeConfig = createRuntimeConfig(fileConfig);
  return runtimeConfig;
}

export function getRuntimeConfig(): RuntimeConfig {
  return runtimeConfig ?? initRuntimeConfig();  // Always returns THE SAME config
}

export function updateRuntimeConfig(fileConfig: VeryfrontConfig): RuntimeConfig {
  runtimeConfig = createRuntimeConfig(fileConfig);  // REPLACES global singleton
  return runtimeConfig;
}
```

## Why It's Bad for Multi-Tenancy

In a multi-tenant SSR framework, ONE renderer pod serves MULTIPLE projects. Each project has its own config. A global singleton cannot represent multiple projects' configs simultaneously.

### Race Condition Scenario

```
Timeline:
=========
t0: Renderer pod starts
    → runtimeConfig = null

t1: Request for Project Alpha arrives
    → initRuntimeConfig({ title: "Alpha Dashboard" })
    → runtimeConfig = { title: "Alpha Dashboard", runtime: {...} }

t2: Request for Project Beta arrives (concurrent)
    → initRuntimeConfig({ title: "Beta Analytics" })
    → runtimeConfig already exists, returns Alpha's config!
    → Beta now uses Alpha's title!

t3: Alpha's request calls getRuntimeConfig()
    → Returns { title: "Alpha Dashboard" } (correct, coincidentally)

t4: Beta's request calls getRuntimeConfig()
    → Returns { title: "Alpha Dashboard" } (WRONG!)
    → Beta renders with "Alpha Dashboard" in <title> tag!
```

### Update Scenario

```
Timeline:
=========
t0: runtimeConfig = { title: "Alpha Dashboard" }

t1: Request for Project Gamma
    → Some code path calls updateRuntimeConfig({ title: "Gamma Portal" })
    → runtimeConfig = { title: "Gamma Portal" }  // REPLACES global!

t2: Alpha's ongoing request calls getRuntimeConfig()
    → Returns { title: "Gamma Portal" } (WRONG!)
    → Alpha's page renders with Gamma's title!

t3: Beta's request calls getRuntimeConfig()
    → Returns { title: "Gamma Portal" } (WRONG!)
    → Cross-tenant data leakage!
```

| Affected | Impact |

## Solution

See **[007.0 - Config Normalization RFC](./007.0-config-normalization-rfc.md)** for the complete solution.

### Architecture: Singleton Renderer + AsyncLocalStorage

**Key principle:** Keep a singleton renderer for performance, use AsyncLocalStorage for request-scoped data isolation.

```typescript
import { AsyncLocalStorage } from "node:async_hooks";

const configStorage = new AsyncLocalStorage<RuntimeConfig>();

export function runWithConfig<T>(
  config: RuntimeConfig,
  fn: () => T | Promise<T>
): Promise<T> {
  return configStorage.run(config, async () => await fn());
}

export function getRuntimeConfig(): RuntimeConfig {
  const config = configStorage.getStore();
  if (!config) {
    throw new Error("getRuntimeConfig called outside request context");
  }
  return config;
}

// Usage in request handler:
async function handleRequest(request: Request, adapter: RuntimeAdapter): Promise<Response> {
  const projectDir = getProjectDir(request);
  const config = await getConfig(projectDir, adapter);
  const runtimeConfig = createRuntimeConfig(config);

  return runWithConfig(runtimeConfig, async () => {
    // All code in this async context can call getRuntimeConfig()
    // and get the correct per-request config
    return renderPage(request);
  });
}
```

This pattern:
- **Singleton renderer** - one process handles all projects (performance)
- **AsyncLocalStorage** - each request gets isolated config context (correctness)
- **Zero code change at call sites** - `getRuntimeConfig()` works the same, just request-scoped
