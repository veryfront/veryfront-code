# 010.4: withErrorContext - Silent Failure Pattern Hides Production Issues

## Severity: HIGH

```typescript
export async function withErrorContext<T>(
  operation: () => Promise<T>,

  try {
```

## The Problem

`withErrorContext` is designed to "handle" errors by:
1. Logging them (at debug level by default - invisible in production)
2. Returning a fallback value
3. **Completely swallowing the error** - caller has no idea failure occurred

This pattern hides production issues because:
- **Debug logs are disabled in production** - errors never logged
- **Caller sees fallback, not error** - can't distinguish "no data" from "fetch failed"
- **No stack trace by default** - even if logged, can't find root cause
- **Error propagation stopped** - monitoring/alerting never sees it

## Why It's Bad for Multi-Tenancy

### Scenario 1: Data Fetch Failure Masked

```typescript
// In data fetching code
const data = await withErrorContext(
  () => fetchProjectData(projectId),
  { operation: "fetchData", slug },
  { fallback: null, logLevel: "debug" }  // Debug = invisible in prod
);

// Caller code
if (data === null) {
  // Is this:
  // A) No data exists (valid state)?
  // B) Fetch failed (error state)?
  // C) Permission denied (security issue)?
  // D) Network timeout (transient failure)?
  // IMPOSSIBLE TO TELL!
}
```

**Multi-tenant impact:**
- Project A's API returns 500 -> data = null -> page renders with empty state
- User thinks content is missing, reports "my content is gone"
- Admin can't find any errors in logs (debug level)
- Meanwhile, Project B works fine (different API endpoint)
- Intermittent reports, impossible to correlate

### Scenario 2: File System Errors Hidden


## Solution

See **[010.0 - Error Handling RFC](./010.0-error-handling-rfc.md)** for the complete solution.

**Summary of fixes:**

### 1. Replace with Explicit Patterns

```typescript
// OPTION A: When failure should propagate (most cases)
export async function withErrorLogging<T>(
  operation: () => Promise<T>,
  context: ErrorContext,
): Promise<T> {
  try {
    return await operation();
  } catch (error) {
    // Log at ERROR level (visible in production)
    logError(error, context, "error", true);
    throw error;  // ALWAYS re-throw!
  }
}

// OPTION B: When specific errors are expected
export async function withExpectedFailure<T>(
  operation: () => Promise<T>,
  fallback: T,
  expectedErrors: string[],  // e.g., ["ENOENT", "FILE_NOT_FOUND"]
  context: ErrorContext,
): Promise<T> {
  try {
    return await operation();
  } catch (error) {
    if (isExpectedError(error, expectedErrors)) {
      logError(error, context, "debug", false);  // OK to debug log expected errors
      return fallback;
    }
    // Unexpected error - log at ERROR and re-throw!
    logError(error, context, "error", true);
    throw error;
  }
}
```

```typescript
export async function withSilentFailback<T>(
```

### 3. Migration Example

**Before (dangerous):**
```typescript
const data = await withErrorContext(
  () => fetchData(id),
  { operation: "fetchData" },
  { fallback: null }
);
```

**After (safe):**
```typescript
// If fetch failure is unexpected
const data = await withErrorLogging(
  () => fetchData(id),
  { operation: "fetchData" }
);  // Throws on failure, caller must handle

// If specific failures are expected
const data = await withExpectedFailure(
  () => fetchData(id),
  null,  // fallback
  ["NOT_FOUND"],  // expected error codes
  { operation: "fetchData" }
);  // Returns null only for NOT_FOUND, throws for other errors
```

## Quick Fix

If full migration is delayed, at least change default log level:

```typescript
// In error-context.ts, change defaults
export interface ErrorHandlingOptions<T> {
  fallback: T;
  logLevel?: LogLevel;  // REMOVE DEFAULT
  includeStack?: boolean;  // REMOVE DEFAULT
}

// In withErrorContext:
export async function withErrorContext<T>(
  operation: () => Promise<T>,
  context: ErrorContext,
  options: ErrorHandlingOptions<T>,
): Promise<T> {
  try {
    return await operation();
  } catch (error) {
    // Change: default to "warn" not "debug", include stack by default
    logError(error, context, options.logLevel ?? "warn", options.includeStack ?? true);
    return options.fallback;
  }
}
```

```typescript
import { assertEquals, assertThrows } from "@std/assert";
import { withExpectedFailure, withErrorLogging } from "../src/errors/error-context.ts";

Deno.test("withErrorLogging always re-throws", async () => {
  await assertThrows(
    async () => {
      await withErrorLogging(
        () => Promise.reject(new Error("test error")),
        { operation: "test" }

    Error,
    "test error"
  );

Deno.test("withExpectedFailure returns fallback only for expected errors", async () => {
  // Expected error -> fallback
  const result1 = await withExpectedFailure(
    () => Promise.reject(new Error("NOT_FOUND")),
    "fallback",
    ["NOT_FOUND"],
    { operation: "test" }
  );
  assertEquals(result1, "fallback");

  // Unexpected error -> throws
  await assertThrows(
    async () => {
      await withExpectedFailure(
        () => Promise.reject(new Error("NETWORK_ERROR")),
        "fallback",
        ["NOT_FOUND"],
        { operation: "test" }
      );
    },
    Error,
    "NETWORK_ERROR"
  );
});
```
