# 001.6 - CSS Cache Key Divergence

## Severity: 🟠 HIGH - Cache Effectiveness Varies By Adapter

## The Problem

CSS cache keys include `projectUpdatedAt` - but this value is **ONLY available** for Veryfront API adapter projects.

```typescript
// src/rendering/orchestrator/pipeline.ts:648-655
let projectUpdatedAt: string | undefined;
const fs = this.config.adapter?.fs;
if (fs && isExtendedFSAdapter(fs) && fs.isVeryfrontAdapter()) {
  const wrappedAdapter = fs.getUnderlyingAdapter() as {
    getProjectData?: () => { updated_at?: string } | undefined;
  };
  projectUpdatedAt = wrappedAdapter.getProjectData?.()?.updated_at;  // 🟢 Only for Veryfront API
}
```

## The Cache Key Function

```typescript
// src/rendering/orchestrator/pipeline.ts:62-67
function getPageCssCacheKey(
  projectId: string | undefined,
  environment: string | undefined,
  slug: string,
  projectUpdatedAt: string | undefined,
): string {
  return `${projectId || "default"}:${environment || "preview"}:${slug}:${projectUpdatedAt || "draft"}`;
}
```

## Cache Keys by Adapter

### Veryfront API Adapter (Production)

```typescript
// projectId = "proj-12345"
// environment = "production"
// slug = "/dashboard"
// projectUpdatedAt = "2024-01-15T10:30:00Z"

cacheKey = "proj-12345:production:/dashboard:2024-01-15T10:30:00Z"
```

**Good**: Cache invalidates when project is updated.

### Local Filesystem Adapter (Dev)

```typescript
// projectId = undefined (or projectDir)
// environment = undefined
// slug = "/dashboard"
// projectUpdatedAt = undefined  // 🔴 NEVER SET

cacheKey = "default:preview:/dashboard:draft"
```

**Bad**: Every local project with `/dashboard` shares the SAME cache key!

### GitHub Adapter

```typescript
// projectId = undefined
// environment = undefined
// slug = "/dashboard"
// projectUpdatedAt = undefined  // 🔴 NEVER SET (not Veryfront adapter)

cacheKey = "default:preview:/dashboard:draft"
```

**Same problem** as local filesystem.

## Why This Breaks Things

### Problem 1: Local Dev Cache Collisions

Running multiple local projects in multi-project mode:

```bash
# Project A: /Users/dev/project-a
# Project B: /Users/dev/project-b
```

Both `/dashboard` pages produce the same cache key:
```
default:preview:/dashboard:draft
```

**Project B may serve Project A's CSS!**

### Problem 2: Stale Cache in Production

When `projectUpdatedAt` is available, cache invalidates correctly:

```
Before deploy: proj-12345:production:/home:2024-01-14T...
After deploy:  proj-12345:production:/home:2024-01-15T...  // 🟢 Different key
```

But for non-Veryfront adapters:

```
Before change: default:preview:/home:draft
After change:  default:preview:/home:draft  // 🔴 SAME KEY - stale cache
```

### Problem 3: Cross-Project CSS Pollution in Multi-Tenant Renderer

The CSS cache is a **global Map**:

```typescript
// From cache inspection - pageCssCache is module-level
const pageCssCache = new Map<string, string>();
```

In multi-project mode, projects can poison each other's CSS cache.

## The Cache Key Components

| Component | Veryfront API | Local FS | GitHub |
|-----------|---------------|----------|--------|
| projectId | `proj-xxxxx` | `"default"` | `"default"` |
| environment | `production/preview` | `"preview"` | `"preview"` |
| slug | ✅ Route path | ✅ Route path | ✅ Route path |
| projectUpdatedAt | ✅ Timestamp | ❌ `"draft"` | ❌ `"draft"` |

**Three adapters, three different cache behaviors.**

## The Unsafe Type Cast (Related to 001.2)

```typescript
const wrappedAdapter = fs.getUnderlyingAdapter() as {
  getProjectData?: () => { updated_at?: string } | undefined;
};
```

1. No interface defines `getProjectData`
2. Cast to ad-hoc type
3. Method may not exist
4. Silent fallback to `undefined`

## Multi-Tenancy Impact

### Scenario: Shared Renderer Pod

1. User A requests `https://projectA.veryfront.com/dashboard`
   - Cache key: `proj-A:production:/dashboard:2024-01-15T10:30:00Z`
   - CSS cached

2. User B runs local dev `http://localhost:3000/dashboard`
   - Cache key: `default:preview:/dashboard:draft`
   - Different CSS cached

3. User C runs local dev `http://localhost:3000/dashboard` (different project)
   - Cache key: `default:preview:/dashboard:draft` (SAME!)
   - **Gets User B's CSS**

### Memory Impact

Over time, the renderer accumulates:
- Properly-keyed production caches (many keys, small duplication)
- Poorly-keyed dev caches (few keys, wrong CSS served)

## The Fix

### Option A: Include Project Directory in Cache Key

```typescript
function getPageCssCacheKey(
  projectId: string | undefined,
  projectDir: string,              // 🟢 Always available
  environment: string | undefined,
  slug: string,
  projectUpdatedAt: string | undefined,
): string {
  const project = projectId || hashString(projectDir);  // 🟢 Hash the dir
  return `${project}:${environment || "preview"}:${slug}:${projectUpdatedAt || "draft"}`;
}
```

### Option B: Always Require Project Identifier

```typescript
function getPageCssCacheKey(
  projectId: string,  // 🟢 Required - no fallback to "default"
  environment: string,
  slug: string,
  projectUpdatedAt: string | undefined,
): string {
  if (!projectId) throw new Error("projectId required for cache key");
  return `${projectId}:${environment}:${slug}:${projectUpdatedAt || Date.now()}`;
}
```

### Option C: File-Based Cache Invalidation

```typescript
let projectUpdatedAt: string | undefined;

// For all adapters, check file modification times
const configPath = join(projectDir, "veryfront.config.ts");
if (await adapter.fs.exists(configPath)) {
  const stat = await adapter.fs.stat(configPath);
  projectUpdatedAt = stat.mtime?.toISOString();  // 🟢 Works for all adapters
}
```

## Solution

See **[001.0 - Unified Adapter RFC](./001.0-unified-adapter-rfc.md)** for the complete solution.

The fix:
1. Add optional `getProjectMetadata?()` to the adapter interface
2. Create a single `buildCacheKey(adapter, slug, projectDir)` function that uses metadata when available:

```typescript
function buildCacheKey(adapter, slug, projectDir) {
  const meta = adapter.getProjectMetadata?.();
  return `${meta?.id ?? hashString(projectDir)}:${slug}:${meta?.updatedAt ?? 'draft'}`;
}
```

One function, works with any adapter.
