# 019.5: Naming Inconsistencies

## Severity: LOW

## Overview

15 categories of naming inconsistencies identified across the codebase.

## Categories

### 1. Handler vs Middleware vs Interceptor

```typescript
// All mean similar things but named differently:
requestHandler()      // src/server/
routeHandler()        // src/routing/
errorHandler()        // src/core/errors/
requestMiddleware()   // src/middleware/
authMiddleware()      // src/middleware/
responseInterceptor() // src/platform/
```

**Proposed Standard**: Use `middleware` for request/response processing, `handler` for terminal request handling.

### 2. ctx vs context vs requestContext

```typescript
// Short form
function process(ctx: Context) {}

// Full form
function process(context: Context) {}

// Prefixed
function process(requestContext: RequestContext) {}
function process(renderContext: RenderContext) {}
function process(buildContext: BuildContext) {}
```

**Proposed Standard**: Use `ctx` for parameter names, full name for types.

### 3. Project Identifiers

```typescript
projectId: string;     // Internal ID
projectSlug: string;   // URL-friendly name
slug: string;          // Ambiguous
id: string;            // Even more ambiguous
```

**Proposed Standard**: Always use `projectId` or `projectSlug`, never bare `id` or `slug`.

### 4. Cache-Related Naming

```typescript
cache: Cache;
cacheStore: CacheStore;
cacheManager: CacheManager;
cacheClient: CacheClient;
cached: CachedValue;
```

**Proposed Standard**: Use `cache` for the abstract interface, `cacheStore` for implementation.

### 5. Transform-Related Naming

```typescript
transform()           // Verb form
transformer()         // Agent noun
Transformer           // Class
TransformResult       // Output
TransformInput        // Input
transformOptions      // Config
```

**Proposed Standard**: Consistent `transform` prefix for all related types.

### 6. Boolean Naming

```typescript
isEnabled: boolean;    // Good: is-prefix
enabled: boolean;      // Okay: adjective
shouldEnable: boolean; // Good: should-prefix
enable: boolean;       // Bad: verb form
```

**Proposed Standard**: Use `is`, `has`, `should`, or `can` prefix for booleans.

### 7. Context Types (12+ Variations)

```typescript
type Context = ...
type RequestContext = ...
type RenderContext = ...
type BuildContext = ...
type SSRContext = ...
type FSContext = ...
type RouteContext = ...
type PluginContext = ...
type ToolContext = ...
type AgentContext = ...
type ModuleContext = ...
type TransformContext = ...
```

**Proposed Standard**: Document each context's purpose, consider consolidation.

### 8. Async Naming

```typescript
getUser()       // Returns Promise
getUserAsync()  // Redundant
fetchUser()     // Implies async
loadUser()      // Implies async
```

**Proposed Standard**: Don't suffix with `Async`, use `fetch`/`load` for I/O operations.

## Decision Required

**D014**: Naming convention standard
- Create style guide document
- Define patterns for each category
- ESLint rules where enforceable

## Migration Approach

### Don't Mass Rename

Mass renaming causes:
- Large diffs
- Merge conflicts
- Import breakage
- Git blame pollution

### Incremental Adoption

1. Document standards
2. Apply to new code
3. Fix when touching existing code
4. Consider alias exports for gradual migration

### Alias Example

```typescript
// During migration period
export const requestHandler = routeHandler;  // Deprecated
export { routeHandler as routeMiddleware };  // New name
```
