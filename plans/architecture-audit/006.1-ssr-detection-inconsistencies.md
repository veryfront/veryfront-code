# 006.1 - SSR Detection Inconsistencies

## Severity: HIGH

## Summary

The codebase uses **5 different patterns** to detect SSR vs browser environment. These patterns can disagree, especially when SSR globals stub the `window` object, leading to hydration mismatches and incorrect behavior.

## The Problem

### Pattern 1: Global Flag + typeof window (Head.tsx)

```typescript
// src/react/components/Head.tsx:4-7
function isServerEnvironment(): boolean {
  const ssrFlag = (globalThis as Record<string, unknown>).__VERYFRONT_SSR__;
  return ssrFlag === true || typeof window === "undefined";
}
```

**Logic**: Returns true if `__VERYFRONT_SSR__` flag is set OR window is undefined.

### Pattern 2: typeof window Only (LivePageContextProvider.tsx)

```typescript
// src/react/components/live/LivePageContextProvider.tsx:21-25
const context: PageContext = pageContext ?? {
  slug: "",
  path: typeof window !== "undefined" ? globalThis.location.pathname : "/",
  params: {},
  query: typeof window !== "undefined"
    ? Object.fromEntries(new URLSearchParams(globalThis.location.search))
    : {},
  frontmatter: {},
};
```

**Logic**: Checks only `typeof window !== "undefined"`.

### Pattern 3: Module-Level Constant (markdown.tsx)

```typescript
// src/react/components/ai/markdown.tsx:32
const isBrowser = typeof window !== "undefined" && typeof document !== "undefined";
```

**Logic**: Evaluated ONCE at module load time, not per-request.

### Pattern 4: Function-Level Check (mdx-module-loader.ts)

```typescript
// src/rendering/ssr/mdx-module-loader.ts:63
const isBrowser = typeof window !== "undefined" && typeof document !== "undefined";
if (isBrowser) return await loadViaBlobURL(compiledCode, cacheKey, key);
```

**Logic**: Correct placement (per-call) but still uses `typeof window`.

### Pattern 5: Global Flag Setter (ssr-globals/index.ts)

```typescript
// src/rendering/ssr-globals/index.ts:56-69
const windowStub = createWindowStub();
setGlobal("window", windowStub);
setGlobal("document", windowStub.document);
// ...
setGlobal("__VERYFRONT_SSR__", true);
```

**Logic**: Sets `window` to a stub AND sets `__VERYFRONT_SSR__` flag.

## Why It's Bad

### Disagreement Scenario

When `setupSSRGlobals()` runs:
1. `window` becomes defined (stub object)
2. `__VERYFRONT_SSR__` becomes `true`

Now the patterns disagree:

| Pattern | Location | Result After SSR Globals Setup |
|---------|----------|-------------------------------|
| `ssrFlag === true \|\| typeof window === "undefined"` | Head.tsx | **SSR** (flag is true) |
| `typeof window !== "undefined"` | LivePageContextProvider.tsx | **BROWSER** (window exists!) |
| `typeof window !== "undefined" && typeof document !== "undefined"` | markdown.tsx | **BROWSER** (both exist!) |

### Concrete Bug: Hydration Mismatch

```
SSR Phase:
1. setupSSRGlobals() creates window stub
2. Head.tsx sees __VERYFRONT_SSR__=true, behaves as SSR
3. LivePageContextProvider sees window exists, uses globalThis.location.pathname
4. But stub location.pathname is "" or "/" - not the actual URL!

Result: SSR renders with path="/", browser hydrates with path="/dashboard"
       → Hydration mismatch error
```

### Multi-Tenancy Risk

```
T1: Project A SSR starts → setupSSRGlobals() → window=stub, __VERYFRONT_SSR__=true
T2: Project B SSR starts → isSSRGlobalsActive() returns true, skips setup
T3: Project A finishes → (no cleanup of globals in current code)
T4: Project B continues with Project A's stale state
```

## Failure Scenario

**Timeline for hydration mismatch:**

```
1. Server receives request for /dashboard/settings
2. setupSSRGlobals() runs:
   - globalThis.window = { location: { pathname: "" }, ... }
   - globalThis.__VERYFRONT_SSR__ = true

3. React SSR starts
4. <Head> renders: isServerEnvironment() → true (correct)
5. <LivePageContextProvider> renders:
   - typeof window !== "undefined" → true (WRONG - sees stub!)
   - Uses globalThis.location.pathname → "" (WRONG!)
   - context.path = ""

6. Page renders with path=""
7. HTML sent to client

8. Browser hydration starts
9. <LivePageContextProvider> renders:
   - typeof window !== "undefined" → true
   - Uses window.location.pathname → "/dashboard/settings" (CORRECT)
   - context.path = "/dashboard/settings"

10. React detects mismatch: SSR path="" vs client path="/dashboard/settings"
11. HYDRATION ERROR or silent re-render flash
```

## Solution

See **[006.0 - Environment Detection RFC](./006.0-environment-detection-rfc.md)** for the complete solution.

**Quick fix approach:**

```typescript
// Unified check that works correctly with SSR globals
function isServerEnvironment(): boolean {
  // Check explicit SSR flag first (most reliable)
  const ssrFlag = (globalThis as Record<string, unknown>).__VERYFRONT_SSR__;
  if (ssrFlag === true) return true;

  // Check if we're in a real browser (not a stub)
  // Real browsers have window.addEventListener as a native function
  if (typeof window !== "undefined") {
    const hasRealDOM = typeof window.addEventListener === "function" &&
      window.addEventListener.toString().includes("[native code]");
    return !hasRealDOM;
  }

  return true; // No window at all = definitely SSR
}
```

**Better fix: Use React context** (see RFC for full implementation)
