# 001.1 - The Layout Bug (CRITICAL)

## Severity: 🔴 CRITICAL - Production Breaking

## The Bug

**App Router nested layouts work locally but are COMPLETELY IGNORED in production.**

When using the Veryfront API adapter (production/preview), nested layouts like `app/dashboard/layout.tsx` are never discovered. The code simply doesn't look for them.

## The Smoking Gun

**File**: `src/rendering/layouts/layout-collector.ts`

```typescript
// Lines 296-308
private async collectNestedLayouts(pageInfo: EntityInfo): Promise<LayoutItem[]> {
  const pageFilePath = pageInfo.entity.path;
  const useAppRouter = await detectAppRouter(this.projectDir, this.config, this.adapter);

  const fs = this.adapter?.fs;
  const isVeryfrontAPI = !!fs && isExtendedFSAdapter(fs) && fs.isVeryfrontAdapter();

  if (isVeryfrontAPI && fs && isExtendedFSAdapter(fs)) {
    // 🔴 THIS PATH NEVER WALKS DIRECTORIES
    return await this.collectAPILayoutConfiguration(fs.getUnderlyingAdapter());
  }

  // ✅ This path walks app/ or pages/ directories
  return await this.collectFilesystemLayouts(pageFilePath, useAppRouter);
}
```

## What `collectAPILayoutConfiguration` Actually Does

**Lines 310-358** - It ONLY checks two places:

```typescript
private async collectAPILayoutConfiguration(wrappedAdapter: unknown): Promise<LayoutItem[]> {
  const configLayout = this.config?.layout;

  // Check 1: config.layout
  if (configLayout && isValidLayoutPath(configLayout)) {
    // ... handles explicit config.layout
    return [await this.createLayoutItemWithBundle(layoutPath)];
  }

  // Check 2: components/layout.*
  if (!configLayout) {
    const layoutPath = await discoverComponentsLayoutPath(this.projectDir, checker);
    if (layoutPath) {
      return [createLayoutItem(layoutPath)];
    }
  }

  return [];  // 🔴 THAT'S IT. NO DIRECTORY WALKING.
}
```

**What's missing:**
- NO call to `discoverNestedLayouts()`
- NO walking of `app/` directory
- NO discovery of `app/dashboard/layout.tsx`, `app/settings/layout.tsx`, etc.

## What `collectFilesystemLayouts` Does (Correctly)

**Lines 360-379**:

```typescript
private async collectFilesystemLayouts(
  pageFilePath: string,
  useAppRouter: boolean,
): Promise<LayoutItem[]> {
  const rootDir = useAppRouter
    ? join(this.projectDir, "app")      // ✅ Uses app/ for App Router
    : join(this.projectDir, "pages");

  // ✅ Actually walks the directory tree
  const nestedLayouts = await discoverNestedLayouts(
    pageFilePath,
    rootDir,
    this.projectDir,
    this.adapter,
  );
  // ...
}
```

## Real-World Impact

### User's Project Structure
```
app/
  layout.tsx              ← Root layout (Nav, Footer)
  dashboard/
    layout.tsx            ← Dashboard layout (Sidebar) 🔴 IGNORED IN PROD
    page.tsx              ← Dashboard page
    settings/
      layout.tsx          ← Settings layout 🔴 IGNORED IN PROD
      page.tsx            ← Settings page
```

### Expected Behavior
```
/dashboard/settings renders with:
  1. app/layout.tsx (root)
  2. app/dashboard/layout.tsx (dashboard)
  3. app/dashboard/settings/layout.tsx (settings)
  4. app/dashboard/settings/page.tsx (content)
```

### Actual Behavior in Production
```
/dashboard/settings renders with:
  1. components/layout.tsx OR config.layout (if exists)
  2. app/dashboard/settings/page.tsx (content)

🔴 Dashboard sidebar MISSING
🔴 Settings layout MISSING
🔴 Only root layout applied (if found in components/)
```

## Why This Happened

The `collectAPILayoutConfiguration` method was written with a **fundamentally different mental model**:

1. Someone assumed API adapter projects only use simple layouts
2. They only checked `config.layout` and `components/layout.*`
3. They never considered App Router's nested layout feature
4. The filesystem adapter was written by someone who understood App Router
5. **Two completely different implementations with no shared code**

## The Fix Is Obvious

`collectAPILayoutConfiguration` should call the same `discoverNestedLayouts` function:

```typescript
private async collectNestedLayouts(pageInfo: EntityInfo): Promise<LayoutItem[]> {
  const useAppRouter = await detectAppRouter(this.projectDir, this.config, this.adapter);
  const rootDir = useAppRouter
    ? join(this.projectDir, "app")
    : join(this.projectDir, "pages");

  // 🟢 SINGLE CODE PATH FOR ALL ADAPTERS
  return await discoverNestedLayouts(
    pageInfo.entity.path,
    rootDir,
    this.projectDir,
    this.adapter,
  );
}
```

But this requires `discoverNestedLayouts` to work with the API adapter's filesystem methods, which it should already - it uses `adapter.fs.stat()` which is implemented by all adapters.

## Multi-Tenancy Impact

This bug affects **every project** using:
- App Router with nested layouts
- Deployed via Veryfront (API adapter)

The renderer serves incorrect HTML to users. Layouts are missing. Sidebars don't appear. Navigation breaks.

## Questions

1. **Why was `collectAPILayoutConfiguration` written separately?**
   - Was there a perceived performance reason?
   - Did the API adapter not support directory walking at the time?

2. **Why wasn't this caught in testing?**
   - Tests likely only run against filesystem adapter
   - No cross-adapter integration tests exist

3. **How many production projects are affected?**
   - Any project with `app/*/layout.tsx` files

## Immediate Action Required

1. Delete `collectAPILayoutConfiguration` entirely
2. Use `collectFilesystemLayouts` for ALL adapters
3. Add cross-adapter integration test that verifies identical layout discovery
4. Audit all other places where `isVeryfrontAdapter()` creates divergent paths

## Solution

See **[001.0 - Unified Adapter RFC](./001.0-unified-adapter-rfc.md)** for the complete solution.

The fix:
1. Add `walkDirectory()` to the adapter interface
2. Create a single `discoverLayouts(adapter, pageFilePath, config)` function
3. Delete `collectAPILayoutConfiguration` entirely
