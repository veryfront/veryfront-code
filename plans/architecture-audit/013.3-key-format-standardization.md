# 013.3 Cache Key Format Standardization

## Summary

Cache key formats vary across the codebase with different separators, ordering conventions, and version prefixes. This document catalogs inconsistencies and proposes a standardization path.

## Value Table

| Issue | Severity | Effort | Priority |
|-------|----------|--------|----------|
| Inconsistent separators | Low | Medium | P3 |
| Missing version prefix in some caches | Low | Medium | P3 |
| Documentation gaps | Low | Low | P3 |

## Current Key Formats

### Separator Inconsistencies

| Cache | Key Format | Separator Style |
|-------|------------|-----------------|
| Transform Cache | `v{ver}:{filePath}:{contentHash}:{ssr}` | Colons `:` |
| Data Fetching | `veryfront:data:{projectId}:{pathname}::{params}` | **Double colon `::`** |
| Theme suffix | `page:{slug}:theme-{color}` | **Hyphen `-`** |
| File Cache | `file:{sourceType}:{projectSlug}:{qualifier}:{path}` | Colons `:` |
| Render Cache | `{projectId}:{env}:{releaseKey}:{ver}:page:{slug}` | Colons `:` |

### Version Prefix Inconsistencies

| Cache | Version Format | Location |
|-------|----------------|----------|
| Transform Cache | `v{TRANSFORM_CACHE_VERSION}:...` | Prefix |
| Render Cache | `{projectId}:{env}:{releaseKey}:{VERSION}:...` | 4th position |
| SSR Module | `v{version}:{projectId}:...` | Prefix |
| File Cache | No version | **Missing** |
| Data Fetching | No version | **Missing** |

### Namespace Prefix Inconsistencies

| Cache | Namespace | Style |
|-------|-----------|-------|
| Redis SSR Module | `veryfront:ssr-module:` | Trailing colon |
| Redis File Cache | `veryfront:file-cache:` | Trailing colon |
| Redis Transform | `veryfront:transform:` | Trailing colon |
| Data Fetching | `veryfront:data:` | Trailing colon |
| Local caches | Various (`component`, `layout`, `proxy`) | No namespace |

### Hyphen in Theme Key

**File**: `/Users/mattboon/Sites/veryfront-renderer/src/rendering/shared/context-aware-cache.ts:208-210`

```typescript
private getCacheKey(
  slug: string,
  ctx: RenderContext,
  colorScheme?: "light" | "dark",
): string {
  // Use hyphen instead of equals sign (API cache key validation only allows: a-z A-Z 0-9 _ : . * - /)
  const themeKey = colorScheme ? `:theme-${colorScheme}` : "";
  return createCacheKey(ctx, `page:${slug}${themeKey}`);
}
```

**Explanation**: Comment indicates `theme-{color}` is used because API cache key validation disallows `=`. This is a **valid constraint**, not a bug.

### Missing Version in Data Fetching Keys

**File**: `/Users/mattboon/Sites/veryfront-renderer/src/cache/cache-key-builder.ts:96-101`

```typescript
export function getProjectScopedKey(prefix: string, resourceKey: string): string | null {
  const ctx = tryGetCacheKeyContext();
  if (!ctx || ctx.mode === "preview") return null;

  return `${prefix}:${ctx.projectId}:${ctx.mode}:${ctx.versionId}:${resourceKey}`;
  // versionId is release/branch ID, not framework VERSION
}
```

**Note**: `versionId` is the release ID, not the framework version. Data fetching cache has no framework version prefix.

## Proposed Standard Format

### Template

```
{namespace}:v{framework_version}:{scope}:{resource}
```

### Scope Types

| Scope Type | Format | Use Case |
|------------|--------|----------|
| Content-addressed | `content:{contentHash}` | Transform, HTTP modules |
| Project-scoped | `{projectId}:{env}:{versionId}` | Render, data fetching |
| Global | `global` | Shared utilities |

### Examples

```typescript
// Content-addressed (shareable)
"veryfront:transform:v1:content:abc123:ssr"
"veryfront:http-module:v1:content:987654"

// Project-scoped (isolated)
"veryfront:render:v1:project-abc:production:release-xyz:page:index"
"veryfront:data:v1:project-abc:production:release-xyz:api-users"

// Global (framework utilities)
"veryfront:config:v1:global:esm-version"
```

## Migration Considerations

### Breaking Changes

Changing key formats causes cache misses. Mitigation strategies:

1. **Version bump**: Include format version in key (already done for most)
2. **Dual-read period**: Read from old and new key formats during transition
3. **Warm-up**: Pre-populate new keys before switching

### Safe Changes (No Migration)

- Adding JSDoc documentation
- Creating type-safe key builder wrappers
- Adding validation for key format

### Risky Changes (Need Migration)

- Changing separator from `::` to `:`
- Adding version prefix to file/data caches
- Changing namespace prefixes

## Recommendations

### Phase 1: Documentation Only (P3)
- Add JSDoc to all key builders explaining format and rationale
- Document the `::` and `-` exceptions and why they exist
- No code changes

### Phase 2: Type-Safe Wrappers (P3)
- Create typed key builder factory with validation
- Wrap existing builders for backward compatibility
- Add compile-time format checking

### Phase 3: Optional Format Migration (P4)
- Only if needed for operational reasons
- Requires careful coordination with cache invalidation
- Consider cost/benefit carefully

## Code Changes for Phase 1

Add JSDoc to `src/cache/keys.ts`:

```typescript
/**
 * Build data fetching cache key prefix.
 *
 * Key Format: veryfront:data:{projectId}:{mode}:{versionId}:{pathname}::{params}
 *
 * Note: Uses `::` between pathname and params for historical compatibility.
 * This differs from standard `:` separator but is preserved for backward
 * compatibility with existing cached entries.
 *
 * @param prefix - Cache namespace prefix
 * @param resourceKey - Resource identifier (pathname::params)
 * @returns Scoped cache key or null if context unavailable/preview mode
 */
export function getProjectScopedKey(prefix: string, resourceKey: string): string | null
```
