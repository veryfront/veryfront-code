# 005.1 - Global Router Detection Cache

## Severity: HIGH

## Location

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/rendering/router-detection.ts`
**Lines:** 24-27, 33-35, 56-64

## The Problem

```typescript
// Line 24-27 - GLOBAL MODULE-LEVEL CACHE
const routerDetectionCache = new LRUCache<string, boolean>({
  maxEntries: 200,
  ttlMs: 60_000,
});

// Line 33-35 - GLOBAL clear function
export function clearRouterDetectionCache(): void {
  routerDetectionCache.clear();  // Clears ALL projects!
}

// Line 56-64 - Cache key is just projectDir, no tenant isolation
const cached = routerDetectionCache.get(projectDir);
if (cached !== undefined) return cached;
// ...
routerDetectionCache.set(projectDir, result);
```

## Why It's Bad for Multi-Tenancy

### Problem 1: Cache Key Collision

In multi-tenant deployments, `projectDir` may be a relative path like `""` (empty string for API-based projects) or a generic path. Different projects with the same relative path structure will share cache entries.

```typescript
// Project A (Veryfront API adapter): projectDir = ""
// Project B (Veryfront API adapter): projectDir = ""
// SAME CACHE KEY! Project A's router detection cached for Project B
```

### Problem 2: Global Clear Affects All Tenants

When one project's filesystem changes trigger `clearRouterDetectionCache()`, it wipes the cache for ALL projects:

```
Timeline:
=========
t0: Project A requests /page
    → detectAppRouter() called
    → Scans filesystem, returns true (App Router)
    → routerDetectionCache.set("", true)

t1: Project B requests /page
    → detectAppRouter() called
    → Cache hit! Returns true
    → BUT Project B uses Pages Router!
    → WRONG ROUTING BEHAVIOR

t2: Project A's file watcher detects change
    → clearRouterDetectionCache()
    → ALL projects' cache cleared (including unrelated Project C, D, E...)
    → Performance degradation across all tenants
```

### Problem 3: Stale Cache After Project Config Change

If a project changes from Pages Router to App Router (via config update), the cache may return stale results:

```typescript
// Project config changed: router: "pages" → router: "app"
// Cache still has: { projectDir: false }  // Pages Router
// User sees Pages Router behavior until 60s TTL expires or cache cleared
```

## Failure Scenario

**Scenario: Two projects with identical relative paths**

```
t0: Request for project-alpha arrives
    → projectDir = "" (API adapter)
    → detectAppRouter() scans, finds app/ directory
    → Cache: { "": true }

t1: Request for project-beta arrives (different project!)
    → projectDir = "" (same key!)
    → Cache hit! Returns true
    → project-beta actually uses pages/ directory
    → App Router resolution attempted on Pages Router project
    → Routes not found or wrong layout applied!
```

## Blast Radius

| Affected | Impact |
|----------|--------|
| All API-adapter projects | Potential router misdetection |
| Projects with identical paths | Cache key collision |
| All projects on pod | Performance hit when one project clears cache |
| Config changes | Up to 60s stale results |

## Solution

See **[005.0 - Router Unification RFC](./005.0-router-unification-rfc.md)** for the complete solution.

The fix involves:

1. **Project-scoped cache keys**: Include project ID/slug in cache key
2. **Per-project clear**: Only invalidate the affected project's cache
3. **Move to RouteRegistry**: Router detection becomes part of registry build, not separate cache

### Immediate Mitigation

```typescript
// Option 1: Include project identifier in cache key
function getCacheKey(projectDir: string, projectId?: string): string {
  return projectId ? `${projectId}:${projectDir}` : projectDir;
}

// Option 2: Use the existing project-specific clear (already exists but underused)
export function clearRouterDetectionCacheForProject(projectDir: string): void {
  routerDetectionCache.delete(projectDir);
}
// ^ But this still has the projectDir collision problem
```

### Proper Fix (from RFC)

```typescript
// Router detection becomes part of RouteRegistry, scoped per-project
interface RouteRegistry {
  projectId: string;  // Unique identifier
  projectDir: string;
  primaryRouter: "app" | "pages";
  routes: UnifiedRoute[];
  builtAt: number;
}

// Cache keyed by projectId, not projectDir
const registryCache = new LRUCache<string, RouteRegistry>({
  maxEntries: 200,
  ttlMs: 60_000,
});

function getRouteRegistry(
  projectId: string,
  projectDir: string,
  adapter: RuntimeAdapter,
): Promise<RouteRegistry> {
  const cached = registryCache.get(projectId);
  if (cached) return cached;

  const registry = await discoverAllRoutes(projectDir, adapter, config);
  registry.projectId = projectId;
  registryCache.set(projectId, registry);
  return registry;
}
```

```typescript
// tests/router-detection-isolation.test.ts
Deno.test("router detection isolates by project ID", async () => {
  // Two projects with same projectDir but different IDs
  const projectA = { id: "project-a", dir: "" };
  const projectB = { id: "project-b", dir: "" };

  // Set up different router types
  // Project A: app/page.tsx (App Router)
  // Project B: pages/index.tsx (Pages Router)

  const resultA = await detectAppRouter(projectA.dir, configA, adapterA, projectA.id);
  const resultB = await detectAppRouter(projectB.dir, configB, adapterB, projectB.id);

  // Should detect correctly despite same projectDir
  assertEquals(resultA, true);   // App Router
  assertEquals(resultB, false);  // Pages Router
});
```
