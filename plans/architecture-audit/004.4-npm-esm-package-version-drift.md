# 004.4 - npm/esm.sh Package Version Drift

## Severity: HIGH - Inconsistent Package Versions Across Pods

## The Problem

HTTP module caching uses URL hash as cache key, but esm.sh URLs can resolve to **different package versions** over time. When esm.sh updates a package, cached transforms on existing pods use the old version while new transforms get the new version.

## Failure Scenarios

### Scenario 1: Semver Range Resolution

```typescript
// User code imports a package without exact version
import { something } from "some-package";

// Resolves to esm.sh:
// https://esm.sh/some-package?target=es2022

// At T1: esm.sh serves some-package@1.2.3
// At T2: esm.sh serves some-package@1.2.4 (patch update)
```

Pod A has cached transform with 1.2.3, Pod B gets request and transforms with 1.2.4.

### Scenario 2: Transitive Dependency Updates

```typescript
// React is pinned to explicit version
const REACT_VERSION = "19.1.1";

// But React's internal dependencies may update:
// https://esm.sh/react@19.1.1?target=es2022&deps=csstype@3.2.3
//
// esm.sh's internal caching/bundling may produce different output
// when their bundler or infrastructure changes
```

### Scenario 3: esm.sh CDN Version Changes

```typescript
// esm.sh has version prefixes that can change:
```

## Package Version Sources

| Source | Version Stability | Cache Key Includes |
|--------|-------------------|-------------------|
| React | Explicit in `REACT_VERSION` | URL hash (good) |
| React subpaths | Derived from REACT_VERSION | URL hash (good) |
| Tailwind | Explicit in `TAILWIND_VERSION` | URL hash (good) |
| User packages | Bare specifier -> esm.sh | URL hash (no version!) |
| Transitive deps | esm.sh internal | Not tracked |

## Impact Assessment

| Scenario | Impact |
|----------|--------|
| esm.sh updates package | Different behavior on different pods |
| Pod restart | New pod gets different package version |
| Cache expiry | Same page renders differently after TTL |
| New deployment | Old cached transforms use old versions |

## Risk Assessment

| Factor | Rating |
|--------|--------|
| **Impact** | HIGH - Inconsistent behavior |
| **Frequency** | MEDIUM - Depends on esm.sh updates |
| **Detectability** | VERY LOW - Subtle behavioral differences |
| **Blast Radius** | HIGH - All users of affected package |
| **Fix Complexity** | HIGH - Need lock file or content hashing |

## Current Mitigations

1. **Explicit React version** - `REACT_VERSION = "19.1.1"` pins React
2. **Explicit Tailwind version** - `TAILWIND_VERSION = "4.1.8"` pins Tailwind
3. **24-hour cache TTL** - Limits version drift window

## What's Missing

1. **Package lock file** - Pin all dependencies to exact versions
2. **Content hash verification** - Verify fetched content matches expected hash
3. **Version in cache key** - Include resolved version, not just URL
4. **Transitive dependency tracking** - Track and hash all dependencies

## Solution

See **[004.0 - Dependency Tracking RFC](./004.0-dependency-tracking-rfc.md)** for the complete solution.

The fix requires:
1. Resolve bare specifiers to explicit versions before caching
2. Include package version in cache key, not just URL hash
3. Verify content hash on cache read
4. Consider package lock file for production builds
