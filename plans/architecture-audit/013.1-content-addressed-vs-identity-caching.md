# 013.1 Content-Addressed vs Identity-Based Caching

## Summary

The veryfront-renderer intentionally uses two caching strategies:
1. **Content-addressed**: Keyed by content hash, safe to share across projects
2. **Identity-based**: Keyed by project/environment, must be isolated

This document clarifies which caches use which strategy and why.

## Content-Addressed Caches (Safe to Share)

These caches use content hash as the primary key. Two projects with identical source code will share cached results - this is **intentional and correct**.

### Transform Cache
- **File**: `/Users/mattboon/Sites/veryfront-renderer/src/transforms/esm/transform-cache.ts`
- **Key Format**: `v{TRANSFORM_CACHE_VERSION}:{filePath}:{contentHash}:{ssr|browser}[:studio]`
- **Builder**: `src/cache/keys.ts:260-269`

```typescript
// src/cache/keys.ts:260-269
export function buildTransformCacheKey(
  filePath: string,
  contentHash: string,
  ssr: boolean = false,
  studioEmbed: boolean = false,
): string {
  const ssrKey = ssr ? "ssr" : "browser";
  const studioKey = studioEmbed ? ":studio" : "";
  return `v${TRANSFORM_CACHE_VERSION}:${filePath}:${contentHash}:${ssrKey}${studioKey}`;
}
```

**Why no projectId?**
- Transform output depends only on source code + transform options
- If project A and B have identical `Button.tsx`, same transform result is correct
- Content hash ensures cache miss when code changes
- Significantly improves cache hit rates for shared patterns (e.g., identical component libraries)

### HTTP Module Cache (esm.sh Bundles)
- **File**: `/Users/mattboon/Sites/veryfront-renderer/src/transforms/esm/http-cache.ts`
- **Key Format**: `{prefix}:{hash}` where hash is URL hash

```typescript
// src/transforms/esm/http-cache.ts:41
const distributedKey = (prefix: string, hash: string | number) => `${prefix}:${hash}`;
```

**Why no projectId?**
- HTTP bundles from esm.sh are identical for all projects
- Same npm package version produces same bundle
- URL-based hash ensures same package = same cache entry

### MDX Bundle Cache
- **File**: `/Users/mattboon/Sites/veryfront-renderer/src/transforms/mdx/mdx-cache-adapter.ts`
- **Key Format**: `mdx:{mode}:{contentHash}`

```typescript
// src/transforms/mdx/mdx-cache-adapter.ts:37-39
private getCacheKey(contentHash: string): string {
  return `mdx:${this.mode}:${contentHash}`;
}
```

**Why no projectId?**
- MDX compilation depends only on source content + mode
- Same MDX content produces same compiled output
- Content hash ensures correctness

### Component Cache
- **File**: `/Users/mattboon/Sites/veryfront-renderer/src/cache/keys.ts`
- **Key Format**: `component:{projectId}:{filePath}:{contentHash}`

```typescript
// src/cache/keys.ts:281-287
export function buildComponentCacheKey(
  projectId: string,
  filePath: string,
  contentHash: string,
): string {
  return `${CacheKeyPrefix.COMPONENT}:${projectId}:${filePath}:${contentHash}`;
}
```

**Note**: This includes projectId but also contentHash - hybrid approach where projectId provides namespace isolation while contentHash ensures correctness.

## Identity-Based Caches (Must Be Isolated)

These caches contain project-specific data that MUST NOT be shared.

### Render Cache
- **File**: `/Users/mattboon/Sites/veryfront-renderer/src/rendering/shared/context-aware-cache.ts`
- **Key Format**: `{projectId}:{environment}:{releaseKey}:{VERSION}:page:{slug}[:theme-{color}]`
- **Builder**: `src/cache/keys.ts:87-93`

```typescript
// src/cache/keys.ts:87-93
export function buildRenderCachePrefix(
  projectId: string,
  environment: "preview" | "production",
  releaseKey: string,
): string {
  return `${projectId}:${environment}:${releaseKey}:${VERSION}`;
}
```

**Why projectId required?**
- Render output includes project-specific HTML, layouts, and data
- Project A's homepage is different from Project B's homepage
- Environment and release key add additional isolation dimensions

### Data Fetching Cache
- **File**: `/Users/mattboon/Sites/veryfront-renderer/src/data/data-fetching-cache.ts`
- **Key Format**: `veryfront:data:{projectId}:{pathname}::{params}`

```typescript
// src/data/data-fetching-cache.ts:54-58
createCacheKey(context: DataContext): string | null {
  const params = JSON.stringify(context.params);
  const resourceKey = `${context.url.pathname}::${params}`;
  return getProjectScopedKey("veryfront:data", resourceKey);
}
```

**Why projectId required?**
- Data fetching returns project-specific API responses
- Same API path in different projects returns different data
- Cache uses `getProjectScopedKey` which includes projectId

### SSR Module Cache
- **File**: `/Users/mattboon/Sites/veryfront-renderer/src/modules/react-loader/ssr-module-loader/loader.ts`
- **Key Format**: `v{ver}:{projectId}:{contentSourceId}:{reactVersion}:{filePath}`

```typescript
// src/modules/react-loader/ssr-module-loader/loader.ts:259-272
private getCacheKey(filePath: string): string {
  if (!this.options.contentSourceId) {
    throw new Error(
      `Missing contentSourceId for SSR module cache (project: ${this.options.projectId}, file: ${filePath})`,
    );
  }
  const reactVersion = this.options.reactVersion ?? "default";
  return buildSSRModuleCacheKey(
    TRANSFORM_CACHE_VERSION,
    this.options.projectId,
    `${this.options.contentSourceId}:${reactVersion}:${filePath}`,
  );
}
```

**Why projectId + contentSourceId required?**
- SSR modules reference project-specific imports
- contentSourceId distinguishes branches/releases
- Different projects may have different React versions

### Layout Component Cache
- **File**: `/Users/mattboon/Sites/veryfront-renderer/src/cache/keys.ts`
- **Key Format**: `layout:{projectId}:{contentSourceId}:{componentPath}:{hash}`

```typescript
// src/cache/keys.ts:289-296
export function buildLayoutComponentCacheKey(
  projectId: string,
  componentPath: string,
  hash: string,
  contentSourceId: string,
): string {
  return `${CacheKeyPrefix.LAYOUT}:${projectId}:${contentSourceId}:${componentPath}:${hash}`;
}
```

**Why projectId + contentSourceId required?**
- Layouts are project-specific UI structures
- Different branches may have different layout code

## Verification

The cache registry in `src/cache/registry.ts` provides methods to verify project isolation:

```typescript
// src/cache/registry.ts:384-390
export function isKeyForProject(key: string, projectId: string): boolean {
  const parts = key.split(":");
  if (parts.length < 2) return false;
  if (parts[1] === projectId) return true;
  if (parts[2] === projectId) return true;
  return parts.includes(projectId);
}
```

This function enables:
- `getKeysForProject(projectId)` - Find all cached entries for a project
- `deleteKeysForProject(projectId)` - Invalidate all project caches
- `deleteKeysForProjectEnvironment(projectId, env)` - Selective invalidation

## Recommendation

**No code changes needed** - the current architecture correctly distinguishes between content-addressed and identity-based caches. The key insight is:

1. **Transform/HTTP/MDX caches** - Content-addressed, sharing is safe
2. **Render/Data/Layout caches** - Identity-based, isolation is correct

The main gap is **documentation** - each cache key builder should have JSDoc explaining its scoping strategy.
