# 008.1 - Global Config Cache Pollution

## Severity: HIGH

## Location

**File:** `/Users/mattboon/Sites/veryfront-renderer/src/config/loader.ts`
**Lines:** 74-75, 350-358, 401-404

## The Problem

The config cache uses a **module-level global Map** shared across all projects:

```typescript
// src/config/loader.ts:74-75
const configCacheByProject = new Map<string, { revision: number; config: VeryfrontConfig }>();
let cacheRevision = 0;
```

Cache lookup uses both the key AND a global revision:

```typescript
// src/config/loader.ts:350-358
const cached = configCacheByProject.get(effectiveCacheKey);
if (cached?.revision === cacheRevision) {
  serverLogger.debug("[CONFIG] Cache HIT - using cached config", {
    cacheKey: effectiveCacheKey,
    isVirtualFS,
  });
  return cached.config;
}
```

Cache clearing invalidates **all projects** and bumps global revision:

```typescript
// src/config/loader.ts:401-404
export function clearConfigCache(): void {
  configCacheByProject.clear();
  cacheRevision++;
}
```

## Why It's Bad

In multi-tenant production:
1. One renderer pod serves requests for Projects A, B, C, D, etc.
2. When Project A triggers HMR or reloads config, `clearConfigCache()` is called
3. This clears **every project's** cached config
4. This bumps `cacheRevision` for **all projects**
5. Next requests for Projects B, C, D all experience cache misses
6. All projects reload their configs unnecessarily

## Failure Scenario

### Timeline

| Time | Event | Impact |
|------|-------|--------|
| T0 | Project A cached at revision 5 | - |
| T0 | Project B cached at revision 5 | - |
| T0 | Project C cached at revision 5 | - |
| T1 | Project A user saves veryfront.config.ts | - |
| T1 | HMR triggers `clearConfigCache()` | **All projects cleared** |
| T1 | `cacheRevision` becomes 6 | - |
| T2 | Project B request arrives | Cache miss - reloads config |
| T2 | Project C request arrives | Cache miss - reloads config |
| T3 | Config reload involves transpilation + import | **200-500ms latency per project** |

### Observed Symptoms

1. **Latency spikes** across unrelated projects when one project makes config changes
2. **Thundering herd** - multiple projects reload simultaneously after one cache clear
3. **Cascading failures** if config loading fails during high load
4. **Unpredictable performance** - config load time depends on other projects' activity

## Multi-Tenancy Impact

**CRITICAL for multi-tenant deployment:**

- One noisy neighbor (frequent config changes) can degrade all projects
- No isolation between tenant configurations
- Cache effectiveness is drastically reduced in production

## Production Evidence

The code paths that trigger `clearConfigCache()`:

```typescript
// src/server/dev-server/request-handler.ts:157
clearConfigCache();

// src/server/bootstrap.ts:85
clearConfigCache();

// src/testing/isolation.ts:499-500
const { clearConfigCache } = await import("../config/loader.ts");
clearConfigCache();
```

In dev server, this is called on **every file change** that triggers HMR invalidation.

## Solution

See **[008.0 - Userland Config RFC](./008.0-userland-config-rfc.md)** for the complete solution.

Key changes:
1. Replace global `cacheRevision` with per-project timestamps
2. Add `invalidateProjectConfig(projectId)` for targeted invalidation
3. Keep `clearAllConfigCaches()` only for server shutdown/restart
