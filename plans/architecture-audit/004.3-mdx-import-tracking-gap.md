# 004.3 - MDX Import Tracking Gap

## Severity: HIGH - Stale MDX Bundles in Production

## The Problem

MDX compilation caches bundles by **content hash of the MDX file only**, without tracking imported components. When a React component imported by an MDX file changes, the MDX bundle is not invalidated.

## Failure Scenario

### Project Structure

```
content/
  blog/
    post.mdx              # Imports <Button> and <Alert>
components/
  Button.tsx              # Button component
  Alert.tsx               # Alert component
```

### MDX File Content

```mdx
import { Button } from '@/components/Button'
import { Alert } from '@/components/Alert'

# My Blog Post

<Alert type="warning">
  This is important!
</Alert>

Click the button below:

<Button variant="primary">Click me</Button>
```

### Sequence of Events

| Time | Action | MDX Hash | Component Hashes | Cache Result |
|------|--------|----------|------------------|--------------|
| T1 | Request `/blog/post` | `mdx:abc123` | Button: `x1`, Alert: `y1` | Miss - compiles |
| T2 | Developer updates Button.tsx | `mdx:abc123` | Button: `x2`, Alert: `y1` | - |
| T3 | Request `/blog/post` | `mdx:abc123` | - | **HIT** - stale bundle! |

The MDX file's hash didn't change, so the cached bundle with the **old Button** is returned.

## What MDX Compilation Should Track

An MDX file has multiple dependency types:

| Dependency Type | Example | Currently Tracked? |
|-----------------|---------|-------------------|
| MDX content | The markdown/JSX in the file | YES |
| Explicit imports | `import { Button } from '@/Button'` | NO |
| Implicit imports | Components passed via MDX provider | NO |
| Layout | `export const layout = 'article'` | NO |
| Frontmatter schema | If schema validation is used | NO |

## Import Graph Not Built

The MDX compiler doesn't build an import graph:

  moduleCode: string,

  vfModules: Array<{ original: string; path: string }>;

} {
  // This FINDS imports but doesn't HASH them for cache keys
  const vfModules: Array<{ original: string; path: string }> = [];

  // ...

The imports are found and resolved, but their **content is not hashed** into the cache key.

## Difference from 004.1

| Issue | 004.1 Transform Cache | 004.3 MDX Cache |
|-------|----------------------|-----------------|
| **Scope** | All transforms | MDX compilation |
| **Cache Layer** | Transform cache (Redis) | Bundle manifest store |
| **Imports** | Local file imports | MDX component imports |
| **Parser** | esbuild AST | MDX compiler |

Both have the same root cause (missing dependency hashes) but different cache mechanisms.

## Solution

See **[004.0 - Dependency Tracking RFC](./004.0-dependency-tracking-rfc.md)** for the complete solution.

The fix requires:
1. Parse MDX imports during compilation
2. Resolve import paths to actual files
3. Hash imported component contents
4. Include dependency hash in MDX cache key
5. Invalidate MDX cache when any dependency changes
