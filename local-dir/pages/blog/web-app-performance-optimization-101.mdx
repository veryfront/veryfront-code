When we talk about web performance, we’re interested in how quickly a website loads and how smoothly it operates for the end user. Slow loading times or frequent crashes may cause frustration and lead users to abandon a website.

However, web performance is not only about achieving positive user experience. It’s also important for your overall business and revenue. By optimizing an application’s web performance, developers can increase customer satisfaction—driving more traffic and sales to their websites. 

In this article, we will learn about the key metrics for web performance and how to measure them. We’ll also review various techniques that can be used to optimize the performance of your web application.


### Measuring Performance

As a prominent player in the internet realm, Google invested considerable effort in research and experimentation to pinpoint the most crucial metrics for attaining an optimal user experience, emphasizing loading, interactivity, and visual stability. These metrics are collectively referred to as  [“Core Web Vitals.”](https://web.dev/vitals/) Moreover, these metrics have been incorporated into Google's search ranking algorithms, implying that websites with superior performance based on these metrics are more likely to rank higher in search results. This approach incentivizes web developers to prioritize website performance optimization.

Let’s take a closer look at these Core Web Vitals.


#### Largest Contentful Paint (LCP)

This indicates how quickly the largest **visible** element of the page and elements associated with it get rendered in the browser viewport. For example, this element could be background or introduction images, videos, large text, infographics, or interactive components such as carousels.


#### First Input Delay (FID)

This tells you how quickly the browser responds to a user’s first interaction with the app. An interaction in this case might be a click, tap, adding text in the field, or submitting a form. In turn, the browser’s response could be highlighting an input, validating the form, starting navigation, etc.


#### Cumulative Layout Shift (CLS)

This metric measures the visual stability of the layout by calculating the total number of unexpected layout shifts and their distance from the initial position. A layout shift indicates a situation when a visible element changes its position due to some other content loading more slowly or being generated dynamically. 


### Tools for Evaluating Performance

Fortunately, developers don’t need to reinvent the wheel and implement metric mechanisms themselves. Google took care of that as well, providing at least two tools that allow you to evaluate the performance of a website and receive total score marks.

The first such tool is [Lighthouse](https://developer.chrome.com/docs/lighthouse/overview/), which you can access under Chrome DevTools. Developers only need to open the browser’s development tools, navigate to the Lighthouse tab, run the analysis, and a report will be generated in just half a minute.

The second tool is [PageSpeed Insights](https://pagespeed.web.dev/) (PSI). This is basically a website where you enter the URL of the web page you want to measure. The report provided by PSI looks very similar to that of Lighthouse; however, PSI combines lab data (from a synthetic analysis, as with Lighthouse) with **real-world data** collected from actual users who previously visited the page.


### Performance Budgets

[Performance budgeting](https://web.dev/performance-budgets-101) is a development approach that involves setting targets for specific metrics related to website performance and using those targets to guide decision-making throughout the development process. 

Let’s say we’re developing a web application, and, being responsible engineers, we want to continuously maintain its performance at a “good enough” level. To determine what “good enough” is, we need to test the application from the user’s perspective, experiment with it, and get a critical threshold we can refer to in the future. Then, as our application evolves, we continuously monitor the actual results. Until they don’t pass this threshold, we consider the performance of our application adequately “good.” 

If adding a new feature to the application makes it too slow (passing a threshold), we need to reconsider the feature and make future decisions to prevent this.


#### Selecting Metrics for Performance Budgets

It’s important to choose and prioritize metrics that properly reflect user experience and are relevant to the context of the website or app. For example, a blog website should probably prioritize metrics related to content-loading speed, while e-commerce sites may prioritize interactivity metrics to make sure the checkout process is smooth. 

Finding and focusing on items that have the greatest impact on a website is also crucial. For example, if we develop a photo-gallery website and 80% of the content consists of images, it might make sense to start optimizing serving of assets before looking into other optimizations such as JavaScript bundle sizes.


### Optimizing React Application Performance

Now that you understand the aims of performance measurement and budgets, it’s time to get familiar with common performance optimization techniques. Most of the points below are described in the context of React applications, but they can be applied in other web applications as well.


#### Avoid Redundant Re-Renders

Frequent and unnecessary rerendering of components can significantly impact the performance of an application, making it less responsive for the end user. Memoization is a powerful and effective technique for avoiding such scenarios. By using the `useMemo()` hook or wrapping components with `React.memo()`, developers can cache the result of rendering these components depending on the props. 

This means that if a component is rendered with the same props multiple times, the cached result can simply be used instead of having to re-render it:


```jsx
import React from 'react';
const MyComp = React.memo((props) => {
  // logic here
});

export default MyComponent;
```



#### Adopt SSR Streaming

In case you have to improve the performance of [server-side rendered (SSR)](/blog/introduction-to-server-side-rendering) applications, implementing [streaming SSR](https://javascriptpatterns.vercel.app/patterns/rendering-patterns/streaming-ssr) can do the trick. In SSR implementations, the server must usually prepare the full content (HTML) of the page and send it to the browser, which will render it and hydrate the state of the application. 

However, with SSR streaming introduced in React 18, components of the page can be prepared and sent (streamed) to the client independently; they can also be hydrated separately, meaning the user will be able to interact with a component as soon as possible even if some parts of the page are not yet ready.

It should be noted that this technique is rather advanced and requires more detailed investigation to implement in an SSR application. 


#### Bundle Properly and Use Tree-Shaking

Bundling is a process of combining the required JavaScript from multiple files into a single file; this improves performance and decreases the number of requests required to load the code into a user’s browser.

[Tree-shaking](https://www.google.com/url?q=https://webpack.js.org/guides/tree-shaking/&sa=D&source=docs&ust=1681122256201713&usg=AOvVaw0kf4FCSadTjRr7bPHu8UEw) is a technique used by bundlers, such as Webpack, Rollup, and esbuild, to eliminate unused code from a project's bundle. The bundler builds a graph of all exported and imported modules, determines what pieces of the code are actually used, and includes only those into a final bundle.

It is also worth noting that it’s tricky but possible to exclude non-critical third-party scripts from the main bundle and execute them in a separate working thread. You can achieve this with the [Partytown](https://www.google.com/url?q=https://partytown.builder.io/react&sa=D&source=docs&ust=1681122256199995&usg=AOvVaw1H5qHCAvUKoSMTKTdawEU1) library.


#### Implement Code Splitting

This technique allows developers to break down JavaScript bundles into smaller chunks and load them on demand. 

For example, we developed an application consisting of 10 different pages, including sign-in and sign-up screens. But in fact, when an authenticated user opens this web app, they will never visit the sign-up page and will only sometimes need to sign in. So it would be good to move the code related to these pages into separate bundles and load them only when the user actually navigates to one of these pages. 

Fortunately, using dynamic imports and “React.lazy()”, we can easily achieve such behavior:


```jsx
import React, { lazy } from 'react';
const SignUpPageComponent = lazy(() => import('SignUpPageComponent'));

function App() {
  return(
     <SignUpPageComponent />
  )
}
```

With such an approach, you can separately bundle the code of one particular component so that the application requests it only when that component is required.


#### Optimize Assets

It sounds trivial, but developers should optimize all kinds of assets downloaded by their application—images, fonts, stylesheets, and other files. Here are some basic but very effective recommendations to consider:



* Use images of exactly the same dimension as needed in accordance with the layout of the page whenever possible. Don’t let CSS manage scaling; instead, generate properly scaled images upfront.
* Enable gzip compression for served assets. This usually can be done in the hosting settings (admin panel) or cloud provider’s console depending on how your app is hosted.
* Minify all text-based files whenever possible. This includes scripts, SVG images, stylesheets, JSON files, etc.
* Minimize the number of custom fonts used in your web application.


#### Use CDN

Hosting all static files on a content delivery network (CDN) instead of a single server, AWS S3 bucket, or other storage will have an extremely positive impact on the application load speed. 

The reason for this is that a CDN (for example, AWS CloudFront or CloudFlare) consists of hundreds of nodes distributed all over the world. Each node caches an application’s files, meaning that the end-user's browser will download files from the closest location. Moreover, a CDN provides more reliability since even if one node goes down, there are still dozens of others operating just fine.


### No Limit to Perfection

The performance of a web application is essential to its success. Users today expect fast, reliable, and seamless interactions with a web app. 

By implementing the techniques described in this article, developers can create fast and convenient applications for modern users. However, there is always something to improve, making the pursuit of performance a continuous process. 

Starting to think about performance and setting performance budgets today is a crucial investment to ensure the successful development of your project.
