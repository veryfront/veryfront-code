The serverless model of deploying applications is quickly becoming a norm given how easy it is to deploy, scale, and monitor such applications. With less overhead for managing the underlying infrastructure and a pay-as-you-use model, developers can now focus on developing their applications as opposed to worrying about scaling and billing.

There are a myriad of ways developers can choose to deploy their applications in a serverless way, but the easiest to start with is functions as a service (FaaS). FaaS lets you write functions in different programming languages and configure them to be triggered on various events. 

In this post, we’ll cover the Google Cloud Platform’s FaaS offering called Cloud Functions, as well as the framework that powers it. You’ll also see how to quickly and easily prototype REST APIs with functions written in Node.js that can be triggered via HTTP requests.
So let's get started. 

### What is the Functions Framework?

[Functions Framework](https://cloud.google.com/functions/docs/functions-framework) is the open-source framework that powers GCP Cloud Functions. It defines an interface to develop modular services, allowing developers to write functions in supported languages and then deploy in any environment such as Cloud Functions, Cloud Run, Knative, or a local developer machine for testing purposes. Functions Framework can be used in two ways, either as a CLI for local testing or as a library in a supported language when deploying in production.

Here, we’ll be walking you through a hands-on demo of writing and deploying your first Cloud Function, which uses Functions Framework under the hood, in Node.js.

### Deploying the First Function

Before getting started, you’ll need a Google Cloud Platform account. You can set up a trial account [here](https://console.cloud.google.com/), which will give you $300 in trial credits. Once the account is set up, from the left navigation bar, navigate to **Serverless > Cloud Functions**. You may get a prompt to enable the API. Click on **Enable API**, and you’re good to go. You’ll see a screen similar to this: 

![](https://cdn.veryfront.com/5b74466b-ad05-4594-82f6-8ed08ad1c37c/blog/google-functions/welcome.png)
Figure 1: Cloud Functions welcome screen.

Next, click **Create Function** to open up the function creation wizard. The entire process is divided into two steps: **Configuration** and **Code**.

In the Configuration step, you can configure various aspects of your functions such as the function’s name, the region in which to deploy the function, execution environment (gen 1 or gen 2), triggers, authentication, scaling parameters (min and max instances), allocated memory, timeout, environment variables, ingress setting, and many more. 

In the Code step, you can choose the language runtime, configure your package dependencies, and write code with an inline editor before finally deploying the function. 

You can also write the function code in the local editor and then use the gcloud CLI to configure various attributes and deploy your function, but in this post, we’ll use the Cloud Console. 

So let’s configure some attributes for your function: 

* **Name**: This is the name of the function; we’re using “hello-world.”
* **Region**: This is the GCP region for deploying the function; this demo uses europe-west3.
* **Environment**: The execution environment can be 1st gen or 2nd gen (which has some improvements over 1st gen); here, we’re using a 1st gen environment.
* **Trigger type**: We’re opting for HTTP in order to trigger the function against HTTP requests. (Note the HTTP URL that you’ll use to trigger your function.)
* **Authentication**: For this demo, unauthenticated requests will be allowed, but in production systems, always choose **Authentication Required**.

Figure 2: Cloud Function basic configuration

Next, you need to configure the runtime settings: 


* **Memory**: Amount of RAM provisioned for the function, left to the default 256 MB for this demo
* **Timeout**: Time after which function execution will be terminated; default timeout is 60 sec. 
* **Service Account**: Identity of the Cloud Function; in this demo,  `App Engine Default` will work but note that in production each function should have a separate [Service Account](https://cloud.google.com/functions/docs/securing/function-identity#individual) to manage fine-grained permissions 
* **Autoscaling**: Number of minimum and maximum instances GCP spins when traffic to the function increases/decreases; here, will be 0 and 10, respectively so that the function scales down to 0 when there is no traffic and can go up to 10 instances when there is more
* **Environment variables**: Available to the function during runtime; for this demo, `OUTPUT_LANG` lets us choose the language in which to send the default response 

We’ll leave the rest of the settings to default:

![](https://cdn.veryfront.com/5b74466b-ad05-4594-82f6-8ed08ad1c37c/blog/google-functions/runtime-configuration.png)
</div>
Figure 3: Cloud Function runtime configuration

Once done, hit **Next**. This will take you to the Code step of the function. In this window, you’ll configure the code and dependencies for your function: 



* **Runtime**: Cloud Functions support Node.js, Java, Python, PHP, Go, .Net, and Ruby as programming languages for your runtime. We’ll choose Node 18, the latest version supported at the time of this writing. 
* **Source code**: Here, you configure how you’ll provide the code. There are four options. You can write the code inline, upload a ZIP file from local, upload a ZIP file from Cloud Storage Bucket, or get the code from a [Cloud Source Repository](https://cloud.google.com/source-repositories/docs).  We’ll choose the inline editor. 
* **Entrypoint**: This is the name of the function, which will be executed on the trigger. We’ll use `greeting` as the function name that we’ll later define inline. 

Now it’s time to configure the Node.js code inline in the `index.js` file. Since you don’t have any extra dependencies, you won’t be making any changes to the `package.json` file: 


```javascript
/**
 * Responds to any HTTP request.
 *
 * @param {!express:Request} req HTTP request context.
 * @param {!express:Response} res HTTP response context.
 */

let vocabulary = {
  'english' : 'Hello World!',
  'german' : 'Hallo Welt!',
  'danish' : 'Hej Verden!',
  'french' : 'Bonjour monde!'
}

let output_lang = process.env.OUTPUT_LANG

let default_response = vocabulary[output_lang.toLowerCase()]

exports.greeting = (req, res) => {
  let message = req.query.message || req.body.message || default_response;
  res.status(200).send(message);
};
```

This code is a simple greeting service. It echoes back the message you send in the request or sends a default message in the chosen language if no message is found in the request.  The code looks for messages in both the query parameter (`req.query.message`) and request body (`req.body.message`), so you can use either GET or POST to trigger your function:

Figure 4: Cloud Function inline code editor

Once everything is configured, click **Deploy**. It will take some time:

### Test Your Function

Once your function is deployed, you can click on the function name (hello-world), and it will open up the function dashboard. This dashboard has all the details you configured in the previous step including the configuration and code. 

Cloud Functions are integrated with StackDriver for logging and Cloud Monitoring for metrics. You can see the following metrics in the Cloud Function dashboard: 

* **Invocations per second**: Number of times the function is invoked 
* **Execution time**: P99, P95 execution times of your function
* **Memory utilization**: P99, P95 memory utilization of your functions during execution
* **Active instances**: Number of active instances of your functions
* **Errors**: Any recent errors during execution


![](https://cdn.veryfront.com/5b74466b-ad05-4594-82f6-8ed08ad1c37c/blog/google-functions/metrics.png)
Figure 6: Cloud Function built-in metrics

Now, go ahead and test your function. Go to the **Testing **tab and click on “Test the Function**”:**

This will send a POST request to your function with an empty body. Since there is no message in the body, the function will return the default message. You can see the “Hello World!” output, as we have configured OUTPUT_LANG as English. You can also see the logs pertaining to this execution of the function. By default, there are execution starting and ending messages. If there are any logging statements in your function, they’ll also appear here.

Now let’s update the `OUTPUT_LANG` environment variable to some other value and deploy the updated version.  Click on the **Edit** button on the function dashboard. Under Runtime settings, update the value for `OUTPUT_LANG` variable and set it to German:

Hit **Next** and then hit **Deploy**. This will trigger the deployment of the new version of the function. A thing to note here is that each configuration of our function is immutable. Every time you make any change to the configuration or code, a new version of the function is deployed. Currently, with the 1st gen environment, there is no way to roll back to previous versions. But in a 2nd gen environment, you can split the traffic between multiple versions and can even move 100% traffic to any older version as well. You can read more about the differences in both environments [here](https://cloud.google.com/functions/docs/concepts/version-comparison). 

Once the new version is deployed, head over to the **Testing** tab and send the empty JSON in the request:

You can also see the historical logs for all of the executions in the **Logs** tab. This also shows you any error during deployment or execution of the function:

And that’s how you can easily configure REST endpoints without managing any infrastructure and with minimal lines of code. You can also add more libraries to your `package.json` and connect to other GCP services including Google Cloud Storage, Cloud SQL, Cloud Bigtable, etc. 


### Conclusion

In this post, we learned how we can use Cloud Functions to easily prototype and deploy REST endpoints without the hassle of managing underlying infrastructure. We also learned about the underlying Functions Framework and the different execution environments the Function Framework supports like Knative, Cloud Run, etc. 

Then, we did a hands-on demo of deploying a Greetings REST endpoint using Node.js and saw how simple it is to set up, monitor, and upgrade the config and code of the Cloud Function. We also covered the logging and monitoring that comes bundled with Cloud Functions.
