We often hear about rendering in the scope of 3D design, movies, animations, and computer games. However, there is also rendering on the web, which can entail more than one approach. In this article, we will discuss various types of rendering, focusing on the server-side option, its pros and cons, and implementation details.

### Rendering on the Web

So what is rendering from a web development perspective? In short, it is a process of converting your web application logic and code into readable content. When we say “readable”, we usually think of users, that is, real humans, and mean to indicate the visualization including all pictures, texts, and interactive elements of a web page. However, it is sometimes also necessary to make sites readable for assistive technologies like screen readers as well as for bots (for example, search engines); in these cases, since their comprehension skills are limited, this usually requires some changes in website implementation to improve [SEO](https://moz.com/learn/seo/what-is-seo).

The main actor in the rendering process is the web browser; it makes requests to a server, parses fetched HTML and CSS, builds the corresponding [DOM](https://developer.mozilla.org/en-US/docs/Web/API/Document_Object_Model/Introduction) elements and applies related styles. It also fetches and executes JavaScript code, making pages interactive and enabling application logic.

Of course, all these things happen regardless of the rendering approach we choose for our app. So then the question is how different approaches may exist at all. The answer lies in how and what tasks we distribute between the server and browser to perform. Let’s read a more detailed explanation of this in the section below.

### Client-Side Rendering (CSR)

Client-side rendering means that when a user initially opens a particular website, the browser, as usual, first downloads HTML which has almost no content except URLs to download the JavaScript files containing the website logic. The browser then downloads and executes the JavaScript; this is responsible for rendering the page content as well as sending additional requests to the server to fetch all the data (e.g., list of products, user profile, cart, etc.) required to display everything correctly.

Obviously, since the browser has a lot of work to do, it may need some noticeable time to prepare the page. However, with CSR, page navigation is also handled in the browser, so all further page transitions will be very fast. This is because the JavaScript logic of the web application will fetch only new data and redraw only the parts on the screen that have changed, meaning there is no need to download everything again for every page.

This is a very common and widespread approach and is the foundation stone of applications written in React and other similar libraries or frameworks.

### Server-Side Rendering (SSR)

On the other hand, SSR is an approach where the server is responsible for executing all the corresponding scripts to compose the full HTML code of the page in response to a request from the client; this means that the browser can represent the content right after this request.

Of course, unlike in CSR when a user navigates from one page to another, the above process runs again and again, assembling each page from scratch. It is worth noting that such an approach is not something new or revolutionary. It is a very old and classic way of how web applications work and has been used for dozens of years. For example, literally all the websites created with PHP or WordPress (there are millions of them) have SSR implementations.

But why is server-side rendering still used and even getting more and more popular? Let’s dive deep into this question by reviewing the pros and cons of SSR.

#### SSR Benefits

First of all, SSR is ultimately good for SEO. The reason for this is that bots and crawlers are not good at executing JavaScript. So when they visit a CSR website, they actually see an empty page, while an SSR site will give them the full content in a single response. Even if a particular bot, for example, Google, can run JavaScript, it will need more time and will index less content on the CSR site, resulting in fewer people visiting the website. This is a major issue for public sites and those depending on traffic, for example, e-commerce shops, blogs, or news sites.

From a user-experience perspective, the main benefit of SSR is that a visitor will see the content quickly—this is especially noticeable when the internet connection is poor. By the way, Google collects different metrics related to speed and performance that can impact a site’s ranking in its search results.

#### SSR Disadvantages

SSR solutions can potentially have bigger infrastructure costs since, unlike client-side rendering, a lot of work is being done on the server and cannot be delegated to the client. In reality, it might be a barely noticeable cost difference, but it all depends on the scale of the project.

As mentioned before, server-side rendering is good when the user first loads a website or wants to visit just one page. However, when it comes to navigation between pages, everything is reloaded each time a user clicks a different link, slowing down browsing and making UX far worse compared to client-side navigation.

As you can see, SSR shows bad results where CSR is good and vice versa. But what if we could combine these two somehow, picking the best parts of each to achieve great performance as well as optimized UX and SEO?

### A Modern Approach

Fortunately, with instruments like Next.js, it is possible to achieve great results nowadays in most of the aspects covered in the previous section. Next.js allows engineers to use literally the same technology on both the backend and frontend, meaning not only the programming language but libraries, for example, React. In short, it can execute React code, generate the HTML code of the resulting page, and send it to the client (browser).

#### Combining SSR and CSR

The above intro about Next.js may sound just like another SSR implementation, but there’s more. After the browser receives the HTML with the full content of the page, it also downloads JavaScript. This is basically the code of a regular React application, which then “revives” all the interactive parts of the web page and enables the CSR features. It also means that further page transitions will be done with client-side navigation.

Such a rendering approach is called “universal rendering” and the “reviving” process is named \*\*hydration \*\*and works in a similar way to how a regular React app is rendered, but with one main difference: It doesn’t create any HTML elements. Instead, it uses the existing DOM tree (remember, it was initially prepared on the server and sent to the browser), attaching all the event listeners and other logic to each corresponding element.

#### The Complexity

Despite it appearing as if Next.js and React will handle all the tricky parts of the hydration process, this is not always the case. It is often challenging for developers to implement everything correctly and avoid hydration errors, which require good debugging skills to resolve. It can happen, for example, that there’s a mismatch between the server-generated HTML and HTML at the hydration step caused by some third-party library.

Also, sometimes developers need to implement different hydration mechanisms like partial hydration, progressive hydration, unordered hydration, and other things outside the scope of this article. This might not be a problem for very experienced engineers, but for everyone else, there is an alternative way to build modern applications: [Veryfront](https://veryfront.com/). This is a development platform that allows you to create efficient SSR applications, reducing development complexity and requiring only basic knowledge of React, JSX, or MDX to get started.

### Worth a Mention—SSG

Although slightly outside the scope of this article, it is important to know that there is one more mechanism: static site generation (SSG). This is completely different from SSR or CSR because it basically doesn’t need a server to operate. The trick is to pre-render every single page of the website, generate all the HTML files, and deploy them to web hosting or, for example, an AWS S3 bucket.

This is ultimately cheap from an infrastructure perspective, but of course, such an approach is only suitable for websites with content that never or almost never changes, such as blog posts. However, such an approach is definitely not a good option for very large sites as the build process time and complexity directly depends on the amount of content. There are also some variations on how to implement SSG and make it more dynamic, but we are not going to cover those here.

### Conclusion

Although CSR applications are usually easier to develop and, in general, provide better user experience, there are still reasons for SSR sites to exist. At the same time, with a modern technical stack, it is possible to achieve hybrid rendering, despite the fact it may require specific knowledge to make it efficient. It is important to properly select the correct rendering mechanism for a project, as it may affect not only your site’s SEO but also UX, development, and infrastructure cost.
